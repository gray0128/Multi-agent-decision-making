# TypeScript 目标架构审查整合与后续处理指导报告

> 审查日期：2026-07-22  
> 对照基准：[`docs/TypeScript目标架构.md`](../TypeScript目标架构.md)  
> 材料范围：[`docs/审查报告/claude/`](./claude/)、[`docs/审查报告/grok/`](./grok/)、[`docs/审查报告/agy/`](./agy/)

## 1. 执行结论

项目整体已经形成较完整的 TypeScript 单包实现：配置与适配器边界、核心审议阶段、逻辑调用归档、观察服务、本地认证和工作目录安全等主链路基本落地。当前不宜按“整体重做”处理，而应采取“先封堵恢复正确性，再补齐契约与回归测试，最后处理体验和维护性”的渐进式策略。

本次整合的最高优先级结论是：**检查点 `pause` 的持久化语义存在恢复粘滞风险，应作为发布阻断项处理，直到专项回归测试证明 `pause -> resume -> continue/end -> completed` 可以闭环。**

除此之外，建议当前迭代一并处理两类剩余契约问题，并保持两项已经落地的门禁：

1. `--format json` 失败、暂停、取消路径的机器可读输出与退出码一致性；
2. 暂停/恢复、诊断脱敏和锁回收等可恢复性/安全边界。

项目审议只读硬门禁和 `LOCKED` 退出码在当前实现中已经存在，后续重点是保持行为并补齐端到端证据，而不是重复实现。

观察服务的 401 响应、SSE 生命周期、浏览器自动打开等问题重要性较低，可在上述主链路稳定后处理。

## 2. 审查材料如何合并

三组报告均以目标架构和当前源码/测试为依据，但严重度标注不完全一致：

| 主题 | Claude | Grok | agy | 整合判断 |
| --- | --- | --- | --- | --- |
| 检查点 `pause` 后恢复 | P1，认为功能仍可完成但即时性不足 | 高危/阻断，认为会再次读到 `pause` | P0，认为会陷入恢复死循环 | **按 P0 发布门禁处理**；源码路径已证实存在粘滞逻辑，需用 e2e 确认实际 CLI 闭环 |
| CLI JSON/退出码 | P1 | 低风险或未列为主要缺口 | 基本符合 | **保留为 P1 契约项**；成功 JSON 已覆盖，错误/暂停/取消 JSON 仍需明确契约 |
| 项目只读能力硬门禁 | P1 | 基本符合 | 符合 | **当前源码与单测已证实存在**；保留为发布回归门禁，不再列为未实现缺陷 |
| `LOCKED` 退出码 | P1 | 基本符合 | 基本符合 | **当前源码与 CLI e2e 已证实存在**；后续只处理 stale lock 的极端回收风险 |
| 脱敏、锁回收、SSE | P1/P2 | 多数未列为高风险 | 基本符合 | **按防御性修复处理**，区分已有覆盖与未覆盖边界 |

因此，本报告不采用任一单组报告的原始数量或星级，而采用以下判断规则：

- 多组报告指出且源码路径一致：直接进入修复清单；
- 只有一组报告指出但能由源码逻辑推导：进入验证优先清单；
- 只有一组报告指出且影响为极端场景/体验：进入后续维护清单；
- 报告间结论冲突：先补最小可复现测试，再确定最终严重度。

## 3. 已确认的保持项

以下设计在各组报告中总体一致，应作为修复时的不可回退约束：

- 配置注册表、CLI/preset 双层白名单和适配器安全参数硬编码；
- 结构化审议的阶段屏障、自由讨论的覆盖/收敛流程和共享限流；
- `freeze -> attempt -> invoke -> commit` 的逻辑调用边界、幂等提交和恢复时不重新组局；
- 归档目录的职责分离、原子写入、只追加事件/记录和用户权限限制；
- 观察服务只读档案、文件信箱通信、`127.0.0.1` 绑定和 Bearer Token 认证；
- 页面 Markdown 净化、CSP、URL fragment 取 token 后写入 `sessionStorage` 并清理地址栏；
- 项目模式的 runtime canary 只读验证设计，以及 `unsupported` 适配器的启动阻断。
- 锁竞争到 `MadError("LOCKED")` / 退出码 5 的现有实现。

后续修复不得为了简化流程而取消这些边界，尤其不得把原始模型名、可执行路径或未验证参数重新暴露给组局结果。

## 4. 后续处理清单

### A. 发布阻断：先修复并证明暂停恢复闭环

**问题**：`src/core/structured.ts` 的 `pauseAt` 和 `src/core/discussion.ts` 的 `atBoundary` 会读取已持久化的检查点决策；当历史动作是 `pause` 时，恢复会再次抛出 `PAUSED`。当前源码证据见上述两个函数中的 `remembered` 分支及 `decision.action === "pause"` 判断。

**风险**：暂停档案可能无法越过原检查点，导致用户不能通过 `mad resume` 完成审议。该风险同时影响结构化审议、自由讨论和检查点期间 SIGINT 兜底路径。

**处理建议**：

1. 先增加回归测试，覆盖页面/CLI `pause`、检查点等待期间 SIGINT、恢复后 `continue`、恢复后 `guide`、取消和最终完成；
2. 明确数据语义：`pause` 是运行状态/中断事件，不应被当成“已接受的推进决策”；
3. 采用其一：不把 `pause` 写入可重放的推进决策，或 resume 时消费/清除该 pause 决策并重新进入检查点；
4. 验证恢复后不会重复写入错误的 `ended` 事件，且 transcript、events、state 的状态转换一致。

**验收条件**：至少有一条真实 CLI e2e 能证明 `paused -> resume -> continue -> completed`；结构化和自由讨论各有一条恢复测试；失败时退出码仍为 `PAUSED`，恢复成功时为 0。

### B. 当前迭代必修：机器接口和中断契约

| 编号 | 处理项 | 主要证据/位置 | 建议动作 | 验收条件 |
| --- | --- | --- | --- | --- |
| B1 | JSON 失败路径 | `src/cli/output.ts`、`src/cli/index.ts` | 统一定义错误 JSON 的 `status/code/message/archive` 字段；明确 stdout/stderr 单一契约 | `--format json` 在配置错误、预检失败、暂停、取消、执行失败下均可稳定解析，且退出码区分场景 |
| B2 | `LOCKED` 退出码回归 | `src/core/errors.ts`、`src/archive/store.ts`、`tests-ts/cli-e2e.test.ts:116-136` | 保持现有锁竞争实现，补充或保留跨进程竞争场景；不要重复改造已存在的映射 | 第二次启动得到退出码 5，既有活动审议不受影响 |
| B3 | 项目只读门禁回归 | `src/core/planning.ts:184-218`、`tests-ts/planning.test.ts:84-128` | 保持 organizer/participant 的 `unsupported` 与 canary 失败阻断；补一次真实 CLI `--workspace` 场景 | 不支持/失败 canary 无法进入项目审议；普通文本审议不被误阻断 |
| B4 | 敏感诊断脱敏 | `src/archive/redact.ts`、`tests-ts/archive.test.ts:145-167` | 保持 diagnostics 边界的 token、环境变量和嵌套字段脱敏；补 `Cookie`、`Set-Cookie`、`Proxy-Authorization` 等 header 名称测试 | `diagnostics.jsonl` 中不出现测试 token、cookie、授权头原值；不要把该保证扩大到未经脱敏的 transcript |
| B5 | 中断语义 | `src/core/structured.ts`、`src/core/discussion.ts`、checkpoint mailbox | 传播 AbortSignal，在检查点等待期间立即转为可恢复的 `PAUSED`；清理未消费 response 文件 | SIGINT 不会长时间等待或产生残留响应；恢复状态可预测 |

### C. 第二批：恢复、并发和观察服务的防御性修复

这些问题不应先于 A/B 阻塞交付，但应纳入后续迭代：

- `ActiveDeliberationLock` 已写入 `ownerId` 和 `acquiredAt`，并以 ownerId 保护 release；后续可进一步增强 stale lock 的进程真实性校验，并补并发 resume 的 TOCTOU 测试；
- 自由讨论失败/暂停后的 pending invocation 恢复测试，优先使用冻结 prompt，避免重建上下文造成不一致；
- SSE 连接在客户端关闭、异常和最大空闲时间三种路径都清理 timer 并结束 response；
- 观察 API 未授权响应统一为 JSON 或空 body，补 `WWW-Authenticate`，避免泄漏可探测文案；
- 检查点信箱消费侧拒绝非法/过期 response，`submit` 也校验 action 集合；
- `state.json` 中 pending/completed invocation 做深层 schema 校验，损坏档案转换为可识别的 `EXECUTION` 错误；
- `schema_version` 使用统一常量，`recordCheckpointDecision` 避免重复写入覆盖原始时间戳。

### D. 第三批：体验、校准和维护性

- CLI 最终输出统一走 ANSI 清洗；
- 交互式 JSON 修改失败后回到确认循环，并明确提示需要再次确认；
- guided 无 TTY 时为观察服务离线提供有限重试和超时；
- 校准 token 估算和 context 摘要阈值，移除硬编码的摘要轮次；
- 强化报告结构校验、Semaphore 时序测试、共享 CLI/preset 去重测试和退出码 e2e；
- 如产品确实要求“自动可达”，再增加跨平台浏览器打开；否则保留清晰、可复制的 URL 输出并在文档中明确行为。

## 5. 推荐实施顺序

```text
1. 建立 pause/resume 最小回归
   ↓
2. 修复 pause 的持久化/重放语义，并完成结构化+自由讨论恢复验证
   ↓
3. 收敛 JSON 错误输出，并回归验证 LOCKED/项目只读现有门禁
   ↓
4. 补脱敏、中断清理、锁回收和自由讨论恢复防御
   ↓
5. 处理 SSE/401、体验、token/context 校准和维护性问题
```

每一步都应先补测试再改实现；对于只涉及体验的项目，不应以重构核心状态机为代价。

## 6. 建议的发布门禁

在 A/B 完成前，不建议宣称“完全符合目标架构”。可以使用“主链路基本符合，暂停恢复与若干机器接口契约待修复”的表述。

建议发布前至少检查：

- `npm test` 或项目既有 TypeScript 测试全部通过；
- pause/resume、SIGINT、cancel、bad config、preflight failure、locked contention 六类 e2e 均有明确状态和退出码断言；其中 checkpoint pause/resume 目前是必须新增的关键场景；
- 项目模式对不支持只读能力的适配器确实阻断，且现有 `planning.test.ts` 与 CLI 工作目录测试继续通过；
- `diagnostics.jsonl` 中不存在明文授权头、cookie 或测试 token；不要将该检查误写成 transcript 全量脱敏保证；
- 观察服务在客户端断连和长时间无事件时不会持续泄漏 timer/连接；
- 修复前后均保留逻辑调用幂等、唯一提交、档案原子写和观察服务只读约束。
