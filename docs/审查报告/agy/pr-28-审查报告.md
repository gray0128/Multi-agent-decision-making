## 4. P1 缺陷深度分析与修复建议

### 4.1 【F-01】[P1] `pendingDeliberationId` 漏检 `"planning"` 状态导致并发进程派生
- **定位**：[`src/server/launch-coordinator.ts#L204`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/server/launch-coordinator.ts#L204)
- **原理**：
  当请求 A 成功派生独立进程并建立档案后，`resolved.status` 变为 `"planning"`。但若子进程尚处于 Node 依赖加载阶段，未完成 `ActiveDeliberationLock.acquire` 写入 `active.lock`：
  此时并发请求 B 到达，`activeDeliberationId()` 返回 `null`；而 `pendingDeliberationId()` 仅匹配 `"reserved"` 与 `"spawned"`，对 `"planning"` 返回 `null`。请求 B 误判无活动进程，成功派生第二个进程！
- **修复**：
  ```typescript
  if (resolved.status === "reserved" || resolved.status === "spawned" || resolved.status === "planning") {
    return resolved.deliberationId;
  }
  ```

### 4.2 【F-02】[P1] `launchExclusive` 5 秒轮询阻塞全局 Promise 队列
- **定位**：[`src/server/launch-coordinator.ts#L94`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/server/launch-coordinator.ts#L94), [`L141-L149`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/server/launch-coordinator.ts#L141-L149)
- **原理**：
  `launchExclusive` 在派生进程后进入 50 次 * 100ms 轮询等待 `"planning"` 状态。由于 `this.queue` 在 `launchExclusive` 结束前不会 resolve，所有并发的 HTTP 请求（如读取启动记录、其它 launch POST）都会被硬性挂起在队列中。
- **修复**：
  `this.queue` 仅需保护记录创建与 `spawn` 调用的原子性。在写完 `"spawned"` 状态记录后立即退出队列锁，将对 `"planning"` 的轮询移至队列之外。

### 4.3 【F-07】[P1] `--web-plan` 作为公开 CLI Option 被暴露
- **定位**：[`src/cli/index.ts#L437`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/cli/index.ts#L437), [`L450`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/cli/index.ts#L450)
- **原理**：
  代码逻辑允许在指定 `--web-plan` 时跳过 `--auto-confirm-plan` 的强制检查。由于 `--web-plan` 被注册在 CLI 参数解析器中，终端用户可以在命令行直接运行 `mad deliberate "xxx" --auto --web-plan` 绕过显式授权安全约束。
- **修复**：
  改用内部环境变量 `process.env.MAD_INTERNAL_WEB_PLAN` 传递，并在 CLI 入口判断若为手动 CLI 调用则拒绝外部环境变量注入。

### 4.4 【F-08】[P1] Web 控制台草稿恢复预设丢失
- **定位**：[`src/web/index.ts#L27`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/web/index.ts#L27)
- **原理**：
  表单恢复保存的 `organizerCli` 时未触发 `change` 事件刷新 `organizerPreset` 下拉框的 `<option>` 选项。随后设置 `organizerPreset.value` 时因目标选项不存在而静默重置为默认值。
- **修复**：
  在回填 `organizerCli` 后，手动根据 CLI 重新生成 `organizerPreset` 的 DOM 选项，再回填预设值。

### 4.5 【F-09】[P1] SSE 事件流异步竞态导致跨档案 DOM 污染
- **定位**：[`src/web/index.ts#L29`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/web/index.ts#L29), [`L41`](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/web/index.ts#L41)
- **原理**：
  `openArchive(id)` 的 `eventAbort.abort()` 在网络请求之前，若用户快速连续点击档案 A 与 B，A 的 Fetch 在 B 的 `openArchive` 中被中断后若已成功到达，A 的 `stream(A)` 依然会在 A 响应返回后被触发，导致旧事件追加至新档案视图中。
- **修复**：
  在 `stream(id, ...)` 启动事件监听前，增加校验 `if (selected !== id) return;`。
