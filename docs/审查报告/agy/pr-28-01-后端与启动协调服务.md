## 2. 深入审查分析

### 2.1 启动协调逻辑与幂等性
- **启动隔离**: 通过 `spawn(process.execPath, args, { detached: true, stdio: "ignore" })` 实现控制台与审议子进程解耦，符合 ADR-0018 要求。
- **并发隐患 (F-01)**: 当请求 A 完成派生并创建档案后，`resolved.status` 变为 `"planning"`。但若子进程仍处于 Node.js 启动早期未写入 `active.lock`，并发请求 B 检查 `pendingDeliberationId` 时因为未匹配 `"planning"` 会判定无活动审议，导致重复派生进程。
- **队列阻塞 (F-02)**: `LaunchCoordinator` 的独占队列 `this.queue` 在 `launchExclusive` 的 50 次 * 100ms 轮询期间被一直持有，导致所有并发调用（如 GET `/api/launches/:id`）均被阻塞。

### 2.2 安全与脱敏
- **Bearer Token**: 使用 `crypto.timingSafeEqual` 对齐校验 Bearer Token，并支持通过 URL Fragment（`#token=...`）传递不泄露至 Referer / HTTP 日志。
- **Host 绑定**: 严苛限制为 `127.0.0.1` 监听。
- **字段脱敏**: `GET /api/launch-options` 过滤掉了 `executable`、`adapter` 等后端敏感字段。
