## 3. 核心偏差与缺陷汇总

### 3.1 阻断级（P0）

| 编号 | 偏差描述 | 架构条款 | 证据路径与代码点 |
|------|----------|----------|------------------|
| **H1** | 检查点动作 `pause` 决策在 `checkpointDecisions` 状态落盘后，`mad resume` 时重放决策会再次读取到 `pause`，导致直接抛出 `PAUSED` 错误并退出，使得审议处于不可恢复的死循环。 | §10 暂停为可恢复状态；§14 恢复与失败 | [src/core/structured.ts#L220](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/core/structured.ts#L220)<br>[src/core/discussion.ts#L244](file:///Users/libo/Documents/github/Multi-agent-decision-making/src/core/discussion.ts#L244) |

**详细分析与证据**：
- 在 `StructuredController` 的 `pauseAt` 逻辑中：
  ```typescript
  const remembered = (await this.archive.readState()).checkpointDecisions[key];
  ...
  if (remembered) {
    decision = { action: remembered.action as CheckpointDecision["action"], ... };
  }
  ...
  if (decision.action === "pause") throw new MadError("PAUSED", "审议已暂停");
  ```
- 如果用户通过命令行或页面下达了 `pause`（暂停）指令，该决策会被永久记录在 `checkpointDecisions` 中。
- 当用户运行 `mad resume <id>` 进行恢复时，控制器会再次按顺序回溯运行。当运行到此前被暂停的检查点时，`pauseAt` 检测到 `remembered` 已经存在且 `action === "pause"`，直接抛出 `PAUSED` 错误，终止执行。这使用户永远无法越过该检查点以继续（`/continue` 或 `/guide`）进行审议。
- `DiscussionController` 中的 `atBoundary` 存在完全相同的逻辑：
  ```typescript
  if (decision.action === "pause") throw new MadError("PAUSED", "审议已暂停");
  ```

**修复建议**：
- 在 `mad resume` 时，应当忽略或清除 `checkpointDecisions` 中属于被暂停状态（`action === "pause"`）的历史检查点决策，从而在恢复时能重新等待用户输入新的交互指令（如继续或指导）。

## 4. 分域符合性摘要

### 4.1 CLI / 适配器 / 配置
- 单 npm 包、ES Module 分发，`src/` 结构与规划一致。
- 配置文件 `config/clis.toml` 加载、静态类型拦截与推理选项白名单绑定（如 `reasoning_effort`）。
- 运行时 `planning.ts` 先校验组局器，后去重并并行预检所有的 `cli/preset` 组合，任一失败即报错。
- `guided` 策略在无 TTY 且无观察服务时立刻退出。退出码与 stdout/stderr 契约契合。

### 4.2 核心审议流程
- 组局阶段支持交互式确认（`/regroup`、`/cancel` 等）及自动接受（`--auto-confirm-plan`）。
- 结构化审议的 7 个独立阶段按顺序并屏障同步，参与者之间满足隔离性（在未完成本阶段输出前不可见同级输出）。
- 自由讨论使用 `DiscussionController`，完成窗口评估并进行收敛，主持决策不计入发言，共享最终报告流水线。
- limits 的三层控制，`context_budget` 自动触发报告 Agent 进行统一滚动摘要，恢复时预算保持一致。

### 4.3 归档与恢复
- 归档文件（`manifest.json`, `state.json`, `events.jsonl`, `transcript.jsonl`, `diagnostics.jsonl`, `report.md`）职责分离。
- `state.json` 与 `report.md` 采用写临时文件后原子重命名替换的策略防损。
- 逻辑调用生命周期（`freeze -> attempt -> invoke -> commit`）闭环，去重幂等。
- 支持并发隔离锁（`active.lock`）及死锁回收（检测 stale 进程）。
- 瞬时异常（限流/超时）与 Schema 校验异常只自动重试 1 次（共 2 次尝试），失败则优雅进入 `paused`。

### 4.4 观察页、服务与安全
- 观察服务独立运行，与审议进程通过 `runtime/checkpoints` 信箱文件单向响应通信。
- SSE 流式 API 仅分发已提交的结构化 events，不暴露原始 console 诊断流与 tokens。
- 静态 HTML 结合纯原生 TS，Markdown 展示在服务端经过 `marked` 与 `sanitizeHtml` 净化处理。
- 监听 `127.0.0.1` 配合动态 Base64 密钥，fragment 获取后在 sessionStorage 保存并立即擦除。
- 项目模式强制 Canary 写入拦截性验证。
