## 2. 总体结论

**综合判定：基本符合目标架构，核心链路可审计且可运行；存在 1 项阻断级可恢复性缺陷。**

| 维度 | 判定 |
|------|------|
| 单包 TS CLI 结构与分发 | 符合 |
| CLI 注册表 / 预设白名单 / 两层验证 | 符合 |
| 组局 → 结构化 / 自由讨论 → 报告流水线 | 高度符合 |
| 透明档案与逻辑调用恢复边界 | 高度符合（pause 粘滞除外） |
| 观察服务、信箱、Bearer、本机绑定 | 高度符合 |
| 项目只读工作目录与适配器安全硬编码 | 符合 |
| 暂停后可恢复（产品语义） | **不符合（P0）** |

实现与架构在「审议进程唯一写档案、cli+preset 可信边界、一次性非交互 CLI 子进程、观察页只响应检查点、127.0.0.1 + Bearer」等硬约束上证据充分。主要短板集中在 **guided 检查点 `pause` 决策被当作可重放终局动作**，导致 `mad resume` 无法越过该屏障。

## 3. 分域符合摘要

### 3.1 CLI / 适配器 / 配置（§3 / §4 / §6 / §16）

- 单 `package.json`，`src/{cli,core,adapters,archive,server,web}` 布局与架构一致。
- `clis.toml`：`defaults.generator` + `[[clis]]` + presets；方案仅 `cli+preset`；拒绝 `extra_args` 与裸模型字段。
- 静态加载校验 + 运行时唯一组合预检；`init` / `validate` / `check` 行为与条文一致。
- guided 默认；`--auto` 与 `--auto-confirm-plan` 分旗标；无 TTY 且无观察服务启动即失败。
- 成功 stdout 仅报告或完整 JSON；失败空 stdout；分退出码（USAGE/CONFIG/PREFLIGHT/LOCKED/PAUSED/CANCELLED/EXECUTION）。
- 七 CLI 适配器路径齐全；安全参数硬编码，配置不可覆盖只读/审批/工具等。

### 3.2 核心审议（§5 / §7 / §8 / §9）

- 固定组局：默认/覆盖组局器、安全注册表视图、参与者/报告/主持校验、预检、交互四动作、共享来源警告。
- 结构化：独立陈述 → 质疑 → 修订/争议 → 可选收敛 → 草稿/并行审阅/终稿；同阶段屏障；guided 四级检查点。
- 自由讨论：`DiscussionController` 权威；覆盖周期 → 开放讨论（禁连续同一人）→ 窗口边界；主持不计发言；复用 `OutcomePipeline`。
- 三层 limits、全局+CLI 限流、统一滚动摘要、预算落档且 resume 不改。

### 3.3 档案与恢复（§13 / §14）

- 档案六件套职责清晰：`manifest` / 原子 `state` / 只追加 events·transcript·diagnostics / 完成时 `report.md`。
- 逻辑调用：`freeze → attempt(attemptId) → invoke → commit` 幂等；权威 transcript 按 `logicalCallId` 去重。
- 并行阶段只重跑未完成；瞬时/schema 各重试一次后暂停；全局活动锁 + `registrySnapshot`。
- 观察服务只读档案，信箱写 runtime 响应文件。

### 3.4 观察 / 检查点 / 认证 / 工作目录（§10–12 / §15）

- 五类检查点动作齐全；`end` 仅 free；首份有效响应 `link` 独占；Ctrl-C → 杀进程组 + 暂停。
- 静态原生页面 + 服务端 Markdown 净化；SSE 仅状态事件；正文在 `invocation.committed` 后整体刷新。
- 每次启动新 Bearer；fragment → sessionStorage；API 仅 Bearer；仅 `127.0.0.1`；token 不落盘/query/Cookie。
- 项目须 `--workspace`；`direct-read-only` 入档；canary + 适配器只读；页面无发起/恢复/删除/改注册表 API。

## 4. 偏差汇总（按严重度）

### 4.1 高（P0，阻断可恢复性）

| ID | 偏差 | 架构条款 | 证据要点 | 来源分册 |
|----|------|----------|----------|----------|
| **H1** | 检查点动作 `pause` 写入 `checkpointDecisions` 后，`resume` 在 `pauseAt` / `atBoundary` 再次读到 `action===pause` 立即抛 `PAUSED`，无法真正恢复 | §10 暂停为可恢复；§14 恢复边界 | `structured.ts` / `discussion.ts` 记忆决策后对 pause 直接抛错；CLI 协调层与 mailbox abort 均会 `recordCheckpointDecision(..., pause)`；SIGINT 在检查点等待期同路径 | 03 |

**影响**：用户在 guided 检查点选「暂停」，或检查点等待期间 Ctrl-C → `mad resume` 卡在同一屏障。对比：逻辑调用中途中断（无 pause 决策）可正常 resume。

**修复方向（不改码，仅建议）**：

1. 不把 `pause` 记入可重放的「已接受推进决策」；或 resume 时忽略/清除 `action===pause` 后重新展示检查点。
2. 补 e2e：`guided pause → resume → continue → completed`。

### 4.2 中（P1）

| ID | 偏差 | 架构条款 | 证据要点 | 来源分册 |
|----|------|----------|----------|----------|
| **M1** | 非交互 `--auto-confirm-plan` 路径始终走 `InvocationRunner`；`allowRegeneration: false` 对该路径无效；schema 失败仍最多重试 2 次，可能接受**第二次**生成结果 | §5 只能自动接受第一次通过校验的方案 | `cli/index.ts` 注入 runner；`planning.ts` 仅无 runner 时 `maximumAttempts=1`；`execution.ts` parse 失败 `RetryableMadError` | 02 |
| **M2** | 自由讨论恢复依赖内存重放与冻结 prompt 一致；缺专用 pause/失败 resume 回归 | §14 自由讨论逐回合与恢复 | discussion 测试无 resume 重放用例；pending 重建 prompt 可能「冻结输入不一致」 | 03 |

### 4.3 低（P2–P3）

| ID | 偏差 | 架构条款 | 来源分册 |
|----|------|----------|----------|
| L1 | Codex 最终文本未统一走 ANSI 清洗（generic 有） | §16 非 TTY 无 ANSI | 01 |
| L2 | guided 无 TTY 时，观察服务中途离线，mailbox 可能长时间轮询 | §6 不无限等待（启动门禁已满足） | 01 |
| L3 | 页面 `APP_JS` 未编译期共享领域类型 | §3 共享 schema | 01 |
| L4 | `mad init` 除路径外还跑 version/help 探针 | §4 措辞级 | 01 |
| L5 | 交互式 JSON 修改校验失败直接退出确认循环 | §5 修改后再次确认 | 02 |
| L6 | 项目组局 prompt 未显式写入 workspace 路径 | §5 项目可读工作目录 | 02 |
| L7 | `pending`/`completed` invocations 读入无深层 schema 校验 | §14 可恢复状态可预期性 | 03 |
| L8 | 运行期非取消错误几乎一律 `paused`，鲜用 `failed` | 运维语义 | 03 |
| L9 | 非法独占 `response.json` 可导致检查点永久等待 | §10 无效响应拒绝 | 04 |
| L10 | `mailbox.submit` 自身不校验 action 集合 | 防御纵深 | 04 |

说明性项（不计入功能偏差）：`report.md` 仅完成时创建；`guide` 与 continue+guidance 并存等——见各分册。

## 5. 优先级建议

| 优先级 | 动作 | 对应偏差 |
|--------|------|----------|
| **P0** | 修正检查点 pause 持久化/重放语义；补 guided pause→resume e2e | H1 |
| **P1** | auto 组局单次生成：organizer parse 失败不可重试，或 runner `maxAttempts:1` 贯通 `allowRegeneration` | M1 |
| **P1** | 自由讨论中途失败/暂停 resume 测试；pending 优先使用冻结 prompt | M2 |
| **P2** | Codex 公开文本统一 `cleanPublicText`；确认循环修改失败回环；组局 prompt 写明 workspace | L1, L5, L6 |
| **P2** | 信箱消费侧丢弃无效独占响应；`submit` 对照当前 request 校验 | L9, L10 |
| **P3** | guided 无 TTY 时 observer 续检/超时；state 内 invocation 结构校验；SSE 安全头对齐 | L2, L7, 04 信息项 |

**应保持、勿回退的设计**：

- 配置与方案双白名单（防 extra_args / 裸模型逃逸）
- 逻辑调用 freeze/commit 幂等与 attemptId 诊断链
- 观察服务只读档案 + 信箱分离
- Bearer 仅内存/fragment/sessionStorage；固定 127.0.0.1
- 适配器安全参数硬编码 + 项目 canary
