## 3. Findings（测试质量）

### F1 · 核心路径有「真集成」，非纯 mock —— 优点

`launches a detached planning process from the authenticated console HTTP boundary` 实际：

1. `spawn` 真实 `mad serve`
2. 从 stderr 解析 `#token=`
3. `POST /api/launches`
4. 轮询独立审议档案上的规划检查点
5. 信箱 confirm 后等到 `completed`
6. 断言 `runtime/launches/<requestId>.json` 含 deliberationId

这直接支撑 ADR-0018「控制台协调启动、审议进程独立」与 #21 最小闭环，**价值高于仅 mock 的 HTTP 单测**。

### F2 · 幂等与失败路径断言到位 —— 优点

`idempotently launches an independent planning deliberation`：

- 未认证 401
- 首次 201 + `planning`
- 同 requestId 重复 200、**spawned 数组长度仍为 1**
- 未知字段 400
- 活动锁 409 + `activeDeliberation`
- 持久化 launches 记录
- **重启服务后** 注入「must not spawn」仍返回原 deliberationId

`persists and redacts a spawn failure...` 断言响应与磁盘记录均不含密钥、含 `[REDACTED]`、status failed —— 符合 #26 脱敏要求。

### F3 · 版本绑定规划检查点有负面路径 —— 优点

web-plan e2e 覆盖：

- `candidateVersion: 99` → **409**
- 无效 replace → validationError（权威在审议进程）
- 有效 replace → version 递增
- regroup → generation 递增
- 最终 confirm → completed + `planConfirmation: "interactive"`

cancel e2e 覆盖：档案 `cancelled`、active.lock 释放、**observer 重启后检查点仍可读**。

### F4 · `launchDeliberation` mock 掩盖真实 spawn 契约 —— 中风险

observer 幂等/失败用例通过依赖注入**完全替换** `defaultProcessLauncher`：

- 不验证 CLI 参数：`--web-plan`、`--auto`、`--workspace`、`--organizer`、limits 标志
- 假启动器**在服务侧回调内** `ArchiveStore.create`，模糊了「仅审议进程写正式档案」边界
- e2e 补偿了默认路径，但**未**断言服务进程未写 `manifest/state/events`（仅断言完成与 launches 记录）

**档案所有权**更多靠架构与 e2e 间接证明，**缺少显式负面断言**（例如：仅有 launches 记录、无 deliberation 目录时不得 201 伪成功——超时失败路径有实现但无测）。

### F5 · 高级启动字段 HTTP 接缝未测 —— 中风险（#24/#25）

实现中 `POST /api/launches` 支持 `workspace` / `organizer` / `limits` 严格校验，但测试仅：

- 最小 payload（topic/mode ± interaction）
- 未知字段 `surprise`

**未覆盖**：非法 limits、不可用 organizer、非法 workspace、空 topic、错误 mode/interaction。
UI 默认总会带 organizer/limits；回归时易漏服务端校验回归。

### F6 · #22/#23 参与者编辑边界大量落在前端 —— 中低风险

`/api/deliberations/:id/agent-id`、人数 ≥2/≤max、删除报告/主持前改选、脏方案禁用确认等主要在 `APP_JS`，**无 HTTP/前端自动化**。
引擎层有 planning 解析/预检单测，但**不能**证明控制台编辑协议端到端正确。

### F7 · #27 可访问性/草稿/响应式几乎零测 —— 低（与 PR 声明一致）

PR 写明「按仓库 UI 约定未做浏览器手工检查」。
自动化仅有：

- `new Function(APP_JS)`
- `toContain("launch-deliberation")` / `wizard-steps`

**sessionStorage 草稿、beforeunload、aria-live/role=alert、防重复提交** 等 AC 无证明。属声明内缺口，但 #27 仍标 Closes 时验收完整性偏弱。

### F8 · #26「页面/服务停止不终止独立进程」未直接测 —— 中风险

实现：`spawn(..., { detached: true, stdio: "ignore" })` + `unref()`。
测试：console e2e 在完成后才 `server.kill`；cancel 用例重启的是 **observer in-process**，审议子进程仍由测试持有。

**未验证**：kill `mad serve` 后 detached 子进程与规划档案仍存活并可被新服务发现。

### F9 · 并发与 pending 占位锁 —— 低到中

`LaunchCoordinator` 有进程内 queue + `pendingDeliberationId`（reserved/spawned），测试只有「已持有 active.lock 后的顺序第二请求」。
**无** 双 requestId 并行 POST 的竞态测试。
