### Warning

#### W1. `topic` 作为裸 argv 交给 `parseArgs`，以 `-` 开头的议题会触发选项解析而非位置参数

- **位置**：`src/server/launch-coordinator.ts` 约 L52–L64（`defaultProcessLauncher` 组装 argv）；消费端 `src/cli/index.ts` 约 L418–L442（`deliberate` → `parseArgs({ strict: true, allowPositionals: true })`）
- **可达失败路径**：
  1. 已认证客户端 `POST /api/launches`，`topic` 为 `--auto`、`--workspace=/etc`、`-mode` 等单 token；
  2. 协调器 `spawn(process.execPath, [..., "deliberate", request.topic, "--mode", ...])`，**未**插入 `--` 分隔；
  3. 子进程 `parseArgs` 将 `topic` 解析为选项：实测 `--auto` 置位且 **positionals 为空** → `USAGE: deliberate 需要且只接受一个非空问题参数`；`--id`/`--workspace` 等触发 ambiguous option 直接失败。
- **影响**：
  - 当前为**单 argv 元素**，无法同时注入 flag **并**保留合法议题字符串，故更像 **fail-closed 的错误解析 / 拒绝服务**（合法「以 `-` 开头的议题」无法启动），而非已证实的权限提升。
  - 脆弱点在于：未来若增加更多 boolean/string CLI 选项或改变参数顺序，同一拼接方式可能变为真实参数注入面；防御不在「无 shell」一层就足够。
- **最小修复**：
  1. 在 launcher 中固定为：`..., "deliberate", "--", request.topic, "--mode", ...`（或把议题放入环境变量 / 旁路文件，避免进选项解析）；
  2. API 层可选：拒绝 `topic.trimStart().startsWith("-")` 并返回 400（与 `--` 方案二选一或叠加）；
  3. 增加单测：`topic="--auto"` / `topic="-foo"` 不得改变 mode/id/workspace，且应稳定成功或稳定 400。

#### W2. 启动 API 对 `workspace` 仅做类型/长度校验，未在派生前做绝对路径 + 存在性 + 目录性校验

- **位置**：`src/server/observer.ts` 约 L285–L290（接受任意非空字符串 ≤4096）；`src/server/launch-coordinator.ts` 约 L66（`--workspace`, path）；真实校验在 `src/cli/index.ts` 约 L469–L472（`realpath` + `isDirectory`）
- **可达失败路径**：
  1. Bearer 有效时提交 `workspace: "relative/../path"`、不存在路径、或非目录路径；
  2. 控制服务仍 `spawn` 独立进程；子进程继承 **serve 的 cwd**（launcher 未设 `cwd`）；
  3. 子进程在 `realpath`/stat 失败后退出；协调记录可能经历 `reserved`→`spawned`→超时/`failed`。
- **影响**：
  - 威胁模型下持有 Bearer 的主体本就可指定工作区，**不等于任意文件写**（项目模式依赖适配器只读 canary / 权限参数，见既有只读链路）。
  - 问题在于：**控制面在派生昂贵/有副作用的独立进程前未收敛路径**，相对路径语义绑到 serve cwd，易造成意外目录授权与失败协调记录堆积；与「显式路径校验」的安全叙事不完全一致。
- **最小修复**：
  在 `POST /api/launches` 解析 workspace 后、调用 `coordinator.launch` 前：
  1. `path.isAbsolute` 否则 400；
  2. `await realpath(workspace)`，`stat` 必须为目录，否则 400；
  3. 将规范化后的绝对路径写入 `WebLaunchRequest`（冻结符号链接目标）；
  4. 单测覆盖相对路径 / 不存在路径 / 文件路径均不得 spawn。

#### W3. 观察服务顶层 500 响应仍直接回传 `error.message`，新启动链路扩大未脱敏异常出口

- **位置**：`src/server/observer.ts` 约 L433–L434
  ```ts
  } catch (error) {
    return send(response, 500, error instanceof Error ? error.message : String(error));
  }
  ```
  对比：启动失败记入协调记录时使用 `redactAdapterDiagnostic`（`launch-coordinator.ts` 约 L150–L157）；`/api/launch-options` 预检失败 reason 已脱敏（`observer.ts` 约 L189、L198）。
- **可达失败路径**：新代码路径中未单独 catch 的异常（例如注册表加载/文件系统/未预期 TypeError）冒泡到顶层 handler；HTTP 正文为原始 `message`。
- **影响**：在 Bearer 已泄露或本机恶意页面调用 API 的场景下，可能多泄漏路径、环境片段或适配器原文（若某条路径未先 redact）。启动**已知**失败路径（spawn throw）已脱敏，故为**残余出口**而非已证实的密钥直出。
- **最小修复**：顶层 catch 统一
  `send(response, 500, redactAdapterDiagnostic(...))`
  （或对客户端固定文案，明细只写本地日志）。为 launch 相关未捕获异常补一条「不得出现 token=…」回归测试。

### Note

#### N1. 幂等 `requestId` 不绑定请求体摘要

- **位置**：`src/server/observer.ts` 约 L243–L246；`src/server/launch-coordinator.ts` 约 L126–L127
- **行为**：相同 `requestId` 直接返回已有 `LaunchRecord`，即使 `topic`/`workspace`/limits 不同。
- **影响**：属幂等设计，非越权；持 token 者可「占坑」requestId。客户端应使用不可预测 requestId。
- **可选加固**：持久化请求指纹，冲突时 409 `REQUEST_ID_REUSED_WITH_DIFFERENT_PAYLOAD`。

#### N2. 方案校验失败信息写入信箱/档案时未显式 `redactAdapterDiagnostic`

- **位置**：`src/cli/index.ts` `confirmPlan` 中 `validationError = error.message`（regroup/replace catch）
- **影响**：`validationError` 进入 checkpoint `data` 与事件，经 `GET /api/deliberations/:id` 回显。`parseDeliberationPlan` 多为结构错误；预检 `detail` 在适配器侧通常已脱敏，但仍缺统一出口。
- **可选加固**：写入前 `redactAdapterDiagnostic(validationError)`。

#### N3. 「安全公开视图」仅严格作用于 `/api/launch-options`；档案详情仍含 `registrySnapshot.executable`

- **位置**：公开视图 `observer.ts` 约 L178–L201（不含 executable）；档案 `detail()` 返回完整 `manifest`（`snapshotRegistry` 含 `executable`，`cli/index.ts` 约 L142–L160）
- **影响**：预存能力，非本 PR 回归；与 PR 文案「页面只读取…安全公开视图」需限定为**发起选项 API**，避免误解为所有 API。
- **可选**：详情 API 对 snapshot 做字段裁剪（若产品允许损失审计字段）。

#### N4. 同 `MAD_HOME` 多 `serve` 实例时活动锁/启动队列仅进程内串行

- **位置**：`LaunchCoordinator.queue` 进程内；`active.lock` 跨进程但依赖 PID 探活
- **影响**：双服务实例理论上可能竞态双开（既有多实例假设外）。文档已要求单活动审议。
- **残余风险**：运维约束，非本 PR 独有。

## Residual Risk

1. **威胁模型依赖 loopback + Bearer 不外泄**：反向代理/隧道/CSRF 对自定义 `Authorization` 通常无效，但 XSS 窃取 `sessionStorage` token 后可调用全部新写 API（发起审议、响应检查点）。CSP 有助于降低，不在本 PR 扩大 Markdown 攻击面的前提下记为残余。
2. **工作区只读**依赖适配器权限参数与 canary，而非 OS 强制沙箱；Bearer 持有者选择的 workspace 可读范围等于本机用户可读范围。
3. **W1/W2/W3** 未修前，异常议题/路径与未捕获 500 仍是可控但粗糙的失败与信息暴露面。
4. **多 `mad serve` 同 MAD_HOME**、requestId 可预测时的占坑（N1/N4）属运维与客户端约定残余。
