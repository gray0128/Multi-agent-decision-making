## Verdict

**CHANGES REQUESTED（需修改后再合）**

架构方向符合 ADR-0018 / ADR-0014：控制服务只写启动协调记录与检查点响应文件，正式档案由独立 `deliberate` 进程拥有；`requestId → deliberationId` 持久映射与进程内启动串行队列整体正确；方案确认的 `candidateVersion` 绑定与一次性信箱也基本到位。

但 PR 将方案确认迁入邮箱后，**打断了“暂停审议”恢复边界**：`plan_confirmation` 不接受 `pause`，而邮箱在 abort 时固定写入 `pause`，会导致规划阶段 SIGINT / 中断挂死，并可能污染后续 resume。另有启动记录 **10s 假失败粘性落盘** 破坏“唯一映射 + 可查询真相”的幂等语义。这两项应在合并前修复。

## Findings

### F1 · [高] 方案确认检查点不接受 `pause`，abort 会独占信箱并挂死

- **位置**
  - `src/cli/index.ts:226-229`（`plan_confirmation` actions 仅 `confirm|replace|regroup|cancel`）
  - `src/server/mailbox.ts:66-68`（abort 时 `submit(checkpointId, "pause")`）
  - `src/server/mailbox.ts:80-81`（仅当 `pending.actions.includes(response.action)` 才接受）
- **失败路径**
  1. guided / `--web-plan` 进入 `confirmPlan` → `mailbox.wait`
  2. 用户 Ctrl-C / `interrupt.abort()` → 邮箱原子写入 `action: "pause"`
  3. `pause` 不在 actions → 轮询永不消费；`publishExclusiveJson` 又使后续合法响应无法写入
  4. 若进程被强杀后残留 `.response.json`，resume 因 `existingCheckpointId` 路径不清理响应文件，会再次挂死
- **影响**
  - 回归：旧 `confirmPlan` 在 signal abort 时抛 `PAUSED`；现无法暂停规划中审议
  - 违反 CONTEXT“暂停审议 / 检查点响应可暂停”与 ADR-0014 恢复边界
  - 终端与网页发起路径均受影响（guided 已统一走邮箱）
- **最小修复**
  1. `plan_confirmation.actions` 增加 `"pause"`
  2. `confirmPlan` 收到 `pause` 时 `throw new MadError("PAUSED", ...)`
  3. 可选加固：abort 提交的动作若不在 actions，改为不写信箱或写后立即让 wait 以中断错误退出

### F2 · [高] 启动记录 10s 无档案即粘性标 `failed`，永不自愈

- **位置**
  - `src/server/launch-coordinator.ts:161-177`（`withArchiveStatus`）
  - `src/server/launch-coordinator.ts:112-116`（`read` 将解析结果写回磁盘）
  - 尤其 `162`：`if (record.status === "failed") return record;`
- **失败路径**
  1. 协调器 spawn 成功，子进程尚未 `archive.create`（冷启动、慢磁盘、registry 加载等）
  2. 客户端轮询 `GET /api/launches/:requestId` 或冲突扫描触发 `read`/`pendingDeliberationId`
  3. `createdAt` 超过 10s 且读档案失败 → 记录被原子写为 `failed`
  4. 子进程随后建立档案并正常规划；协调记录仍永远 `failed`（短路不回读档案）
  5. 同 `requestId` 幂等重试只返回失败映射，无法反映真实审议；用户换新 `requestId` 又可能撞上活动锁
- **影响**
  - 破坏 PR 宣称的“启动请求 ID 持久映射到唯一审议 ID、可安全重试”
  - 控制台可能展示失败，而活动审议实际在跑
- **最小修复**
  1. `failed` 若错误码/文案为“未能建立规划档案”，每次 `read` 仍尝试探测档案；探测成功则升级为 `planning`/`finished`
  2. 将“无档案超时”与 spawn 硬失败分离：前者用 `status: "spawned"` + 可过期的 `lastError`，或独立 `status: "lost"`，且不永久短路
  3. 超时阈值与 spawn 后等待（当前 5s）对齐为命名常量，并拉长到更保守值（例如 30–60s）仅作告警

### F3 · [中] `canLaunch` / 活动冲突前置检查忽略 in-flight 启动记录

- **位置**
  - `src/server/observer.ts:141-152`（`activeDeliberation` 只看 `active.lock`）
  - `src/server/observer.ts:206-220`（`canLaunch: active === null`）
  - `src/server/observer.ts:253-256`（POST 前置冲突同样只看锁）
- **失败路径**
  1. 首次启动已 `reserved`/`spawned`，子进程尚未 `lock.acquire`
  2. `/api/launch-options` 仍返回 `canLaunch: true`
  3. UI 允许第二次点击；不同 `requestId` 进入 `LaunchCoordinator` 后虽会 `ActiveLaunchConflict`，但体验与状态不一致
- **影响**
  - 竞态窗口内误导“可发起”；依赖协调器兜底而非公开状态诚实
  - 与 CONTEXT“同一数据根最多一个活动审议”的产品语义部分脱节
- **最小修复**
  - 抽取与 `pendingDeliberationId` 相同的 pending 扫描，供 `activeDeliberation` / `canLaunch` / POST 409 共用：`active.lock` **或** 未终态 launch 记录均视为不可发起

### F4 · [中] 启动 API 不预检 workspace，失败被推迟到子进程并占用协调槽位

- **位置**
  - `src/server/observer.ts:285-290`（仅类型/长度校验）
  - `src/server/launch-coordinator.ts:66`（原样 `--workspace` 传给子进程）
  - `src/cli/index.ts:469-471`（子进程 `realpath` / `isDirectory` 才失败）
- **失败路径**
  1. 传入相对路径、不存在路径或非目录
  2. 协调器写入 `reserved`→`spawned` 并占用 pending 冲突槽
  3. 子进程在拿锁前/后失败；协调记录需等超时逻辑才变 `failed`（并与 F2 叠加）
- **影响**
  - 错误反馈延迟；短暂阻止其它合法启动
  - 资源限制/组局器已在服务侧校验，workspace 不一致
- **最小修复**
  - 在 `POST /api/launches` 对 workspace 做 `realpath` + `stat.isDirectory`（失败 400）；保持子进程二次校验

### F5 · [中] 规划阶段 SIGINT 与“取消”语义分裂，网页无 pause 入口

- **位置**
  - `src/cli/index.ts:229`（无 pause）
  - `src/web/index.ts` 方案编辑 UI 仅暴露 confirm/replace/regroup/cancel 类动作
  - `src/cli/index.ts:605-612`（`PAUSED`/`CANCELLED` 状态写入依赖异常码）
- **失败路径**
  - 网页用户只能 cancel（终态不可恢复），无法 pause（可恢复）
  - 终端用户期望 Ctrl-C = pause，却触发 F1 挂死
- **影响**
  - 与领域词“暂停审议 ≠ 取消审议”不一致
- **最小修复**
  - 同 F1：支持 pause；网页方案检查点增加“暂停”按钮（可选，至少终端/信号路径必须正确）

### F6 · [低] 失败后的幂等重试 HTTP 语义偏乐观

- **位置**
  - `src/server/observer.ts:243-246`（任意已存在记录统一 200）
  - `src/server/observer.ts:313`（首次 `failed` 返回 500，重试变 200）
- **失败路径**
  - spawn 失败写入 `failed` 后，客户端用同一 `requestId` 重试得到 200 + `status: "failed"`，易被泛化 HTTP 层当成成功
- **影响**
  - 客户端若只看状态码会误判；不破坏磁盘幂等
- **最小修复**
  - 已存在且 `status === "failed"` 时返回 409/422/200 但文档强制要求读 `status`；或统一：首次与重试都按记录状态映射（failed→500/422）

### F7 · [低] 启动记录“创建若无”非跨进程 CAS

- **位置**
  - `src/server/launch-coordinator.ts:41-45`（`atomicJson`：tmp `wx` + `rename` 覆盖目标）
  - `src/server/launch-coordinator.ts:104-109`（仅单进程内 `queue` 串行）
- **失败路径**
  - 同一 `MAD_HOME` 上两个 `mad serve` 并发处理相同/不同 `requestId` 时，映射文件可能互相覆盖或双 spawn；最终仍受 `active.lock` 约束，但协调记录可能脏
- **影响**
  - 本地默认单 serve 风险低；与“持久映射唯一”严格语义有隙
- **最小修复**
  - 预留文件用 `link`/`O_EXCL` 创建目标（参考 `publishExclusiveJson`）；冲突则读已有记录返回

## Residual Risk

1. **PID 复用**：`activeDeliberationId` / observer `activeDeliberation` 仅 `kill(pid,0)`，若锁文件 PID 被无关进程复用，可能误判活动审议（历史问题，本 PR 复用同一模式）。
2. **幂等不校验请求体指纹**：相同 `requestId` 不同 topic/mode 仍返回原映射（at-most-once 常见取舍）；依赖前端每次新 UUID。
3. **`pendingDeliberationId` 不含 `planning`**：依赖子进程先 `lock.acquire` 再 `archive.create` 的顺序；当前 CLI 顺序正确，若未来调换将出现双启动窗口。
4. **detached 子进程与 stdio ignore**：spawn 成功只表示 OS 接受进程，早期崩溃依赖超时/档案探测（与 F2 相关）。
5. **多标签/多控制台**：版本绑定与一次性信箱可防双确认；pause 修复前并发 abort 风险更高。
