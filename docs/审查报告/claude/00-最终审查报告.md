## 2. 总体结论（综合判定）

**整体判定**：**基本符合目标架构，存在少量必须修复的契约偏离，但无阻断级 P0 缺陷。**

四个分册一致指出：`src/cli/`、`src/adapters/`、`src/core/`、`src/archive/`、`src/server/`、`src/web/` 的实现与 `docs/TypeScript目标架构.md` 主体高度一致——CLI 注册表只承载可信 CLI 边界、调用预设按 `AdapterId` 类型化校验、安全参数（`--permission-mode`、`--sandbox` 等）由适配器在运行时强制拼装、`mad init` 仅探测可执行文件、结构化审议 7 阶段与 4 处检查点屏障、自由讨论覆盖周期与窗口边界、全局与 CLI 限流器、统一滚动摘要、归档五类文件结构与原子写、唯一提交、文件权限 0o700/0o600、127.0.0.1 绑定、URL fragment + sessionStorage、文件信箱原子发布、Markdown `marked` + `sanitize-html` 严格白名单等关键契约均在代码中固化。

**已识别但未发现 P0 阻断级偏差**。汇总 P1 偏差 9 项、P2 偏差 16 项。P1 项集中在以下三类：

1. **退出码与 JSON 输出契约缺口**（分册 #1 / P1-1、P1-2）：失败/暂停/取消路径在 `--format json` 时不输出结构化 JSON；`EXIT_CODES.LOCKED = 5` 是死代码。
2. **只读能力硬性要求在项目审议启动路径上未强制**（分册 #1 / P1-3 与分册 #4 / §15 部分符合）：非 Codex 适配器虽标注 `runtime-canary`，但 `verifyProjectReadOnly` 未作为项目审议的硬性门槛。
3. **可观测性与可恢复性边界**（分册 #2 / P1-D1、分册 #3 / P1-1、P1-3、分册 #4 / P1-2/P1-3）：abort 触发到 PAUSED 的快速短路不充分；全局锁在极端 PID 复用下可能误判永久活动；`Cookie` / `Proxy-Authorization` 脱敏盲区；`401` 文案与 SSE 长连接清理不严；`token` 仅打印不自动打开浏览器。

P2 项为风格、UX、可维护性层面的可优化项，可在后续 PR 中分批处理。

## 3. 核心偏差与缺陷汇总

> 评级说明：
> - **P0** 阻断系统正确完成关键功能
> - **P1** 偏离目标架构的安全或正确性契约（必须修复）
> - **P2** 体验或弱一致性、可维护性问题（建议修复）

### 3.1 P1（重要偏差 — 必须修复）

#### P1-1 失败/暂停/取消路径不输出结构化 JSON，违反 §16 "JSON 至少包含状态"与"暂停、取消、配置错误、预检失败和执行失败使用不同退出码"

- **来源**：分册 #1 / §3.1 P1-1
- **架构条款**：§16 stdout/stderr/退出码
- **证据**：`src/cli/output.ts:15-44` `writeCompletedResult` 仅在成功路径写出；`src/cli/index.ts:540` 仅在 Controller 正常返回 `result` 后调用；`catch` 中没有对应 JSON 写出（`src/cli/index.ts:542-555`、`src/cli/index.ts:700-707`）；`src/cli/index.ts:761-765` 顶层 catch 仅写 stderr + 设置 exitCode
- **影响**：调用方使用 `--format json` 并通过退出码判断状态时，无法拿到统一的 JSON 体；CI 集成需要解析 stderr 而非 stdout，违反"JSON 至少包含状态"语义
- **修复建议**：`main().catch` 内根据 argv 中是否含 `--format json` 决定写 stderr JSON + 退出码双通道，或在 stdout 写一个包含 `status` 字段的精简对象（明确不在 §16 覆盖的成功输出范围）

#### P1-2 `EXIT_CODES.LOCKED = 5` 是死代码：未发现任何抛 `MadError("LOCKED", ...)` 的实现路径

- **来源**：分册 #1 / §3.1 P1-2
- **架构条款**：§16、§14
- **证据**：`src/core/errors.ts:27-35` 定义 `LOCKED: 5`，但 CodeGraph 全文索引中无 `MadError("LOCKED", ...)` 实例；`src/archive/store.ts` `ActiveDeliberationLock` 是锁实现入口，未把锁冲突翻译成 `LOCKED` 退出码；`tests-ts/` 没有针对并发锁竞争产生 `LOCKED` 退出码的测试
- **影响**：用户同时启动第二个 `mad deliberate` 时，行为未对齐到 §16 约定的 `LOCKED=5` 退出码；可能导致 CI 或外部脚本无法区分"锁冲突"与"其他配置错误"
- **修复建议**：在 `ActiveDeliberationLock.acquire` 检测到已存在活跃审议时抛 `MadError("LOCKED", ...)`，由顶层 `main().catch` 自然映射到退出码 5；并补 `tests-ts/cli-e2e.test.ts` 用例

#### P1-3 项目审议启动路径未硬性强制 `verifyProjectReadOnly`，与 §15 "最低只读能力不能用于项目审议"不一致

- **来源**：分册 #1 / §3.1 P1-3（与分册 #4 §2.4 "部分符合"同源）
- **架构条款**：§15 工作目录与安全
- **证据**：`src/adapters/read-only.ts:30-67` 实现通用 canary，但仅在 `projectReadOnlyCapability === 'runtime-canary'` 时调用（`src/adapters/codex.ts:52-54`、`src/adapters/generic.ts:70-75`）；`reasonix` 标为 `'unsupported'`，其它 5 个标注 `'runtime-canary'`；CodeGraph 视图中 `verifyProjectReadOnly` 仅被 `OrganizerService.preflightPlan` 调用一次，项目审议启动路径（`src/cli/index.ts:483-493`、`src/cli/index.ts:634-654`）在 `propose`/`preflightPlan` 中是否对 `verifyProjectReadOnly` 做硬性要求并阻止非只读适配器，没有被 §15 明确校验
- **影响**：当用户把未通过 `verifyReadOnlyWithCanary` 的配置用于 `--workspace`，可能进入项目审议，违反 §15
- **修复建议**：在 `OrganizerService.preflightPlan` 中对每个 `plan.participants[*].invocation` + `plan.organizer` 的适配器，若 `projectMode=true` 且 `verifyProjectReadOnly` 未通过，抛 `MadError("PREFLIGHT", "项目审议需要已验证的只读 CLI：...")`；并在 `tests-ts/planning.test.ts` 加覆盖

#### P1-4 abort 触发到 PAUSED 状态的快速短路不充分（pauseAt / atBoundary 不在 AbortSignal 触发时立即短路）

- **来源**：分册 #2 / §3.1 P1-D1
- **架构条款**：§10、§14
- **证据**：`src/core/structured.ts` 第 198-223 行 `pauseAt`：若 `decision.action === "pause"` 抛 `MadError("PAUSED", ...)`，但若 abort 在 `await this.checkpoint(stage, summary)`（第 210 行）等待期间发生，调用层抛 abort 错误而非快速短路的 PAUSED；`coordinatedStructuredCheckpoint`（`src/cli/index.ts:264-309`）在 `mailbox.wait` 内部监听 `signal?.abort` 自动 `submit(checkpointId, "pause")`，最终仍回到一致状态，但与 §10 即时性不完全匹配；`src/core/discussion.ts` 第 222-248 行 `atBoundary` 构造器未接收 `signal`
- **影响**：暂停恢复路径在 abort 注入检查点时仍正确但稍慢；对于非协调模式（无 mailbox）若 checkpoint handler 自身不消费 abort，会出现"信号触发但仍等待响应"的边缘态
- **修复建议**：把 `signal` 显式传播至 `pauseAt` / `atBoundary`；`pauseAt` 入口先判断 `signal?.aborted` 直接抛 `PAUSED`

#### P1-5 全局活动锁 `ActiveDeliberationLock.acquire` 缺少"跨进程去重 + 进程 PID 真实性"校验，OS 高速复用 PID 时可能误判永久活动

- **来源**：分册 #3 / §3.1 P1-1
- **架构条款**：§14 "同一 MAD_HOME 使用原子全局锁限制单个活动审议"
- **证据**：`src/archive/store.ts:271-289` `acquire` 在 `open(path, "wx", ...)` 失败后仅尝试一次 `reclaimStale`；`reclaimStale` 通过 `process.kill(pid, 0)` 探测 PID 是否存在，但若上一个活动进程 PID 已被 OS 复用给另一个无关进程，`process.kill(pid, 0)` 会成功，导致锁永远无法回收
- **影响**：极少数极端场景（OS 高速复用 PID + 锁文件残留）出现永久"活动审议"误判
- **修复建议**：在锁文件中加入"创建时间戳"或"ownerId + nonce"，并在 `reclaimStale` 中额外检查 ownerId 与上次写盘时间间隔；或采用 `fcntl(F_OFD_GETLK)` 风格的心跳

#### P1-6 脱敏对 `Authorization` 之外的 HTTP 风格头（`Cookie`、`Proxy-Authorization`、自定义 `X-Api-Token`）未覆盖

- **来源**：分册 #3 / §3.1 P1-3
- **架构条款**：§13 "diagnostics.jsonl 只追加脱敏调用诊断"
- **证据**：`src/archive/redact.ts:1-17` `SENSITIVE_KEY` 仅匹配 authorization / api_key / access_token / auth_token / token / secret / password；HTTP `Cookie: session=xxxx` 不会触发键名匹配；redactString 的正则也不会匹配
- **影响**：诊断文件可能泄漏会话 Cookie
- **修复建议**：将 `cookie`、`proxy[-_]authorization`、`set-cookie` 加入 `SENSITIVE_KEY`，或在 `redactString` 中加入匹配常见 header 名（大小写不敏感）

#### P1-7 Token 仅打印、不自动打开浏览器（偏离 §12 "自动打开或打印" 的"自动可达性"）

- **来源**：分册 #4 / §3.1 3.1
- **架构条款**：§12 本地认证
- **证据**：`src/cli/index.ts:714-726` `serve()` 仅 `process.stderr.write` 打印 `observer.url`，没有调用 `open`/`xdg-open`/`start` 自动拉起浏览器；`grep child_process|execSync|spawnSync` 在 `src/cli/index.ts` 与 `src/server/observer.ts` 均无命中
- **影响**：低（功能正确但用户体验打折）
- **修复建议**：使用 `open` 包（跨平台）做 `await open(observer.url)`；或在 stderr 输出后再写一行"可复制链接"提高可发现性

#### P1-8 Observation API 对未授权请求统一返回 401 但 body 为纯文本，存在跨源 `<script src>` 试探面

- **来源**：分册 #4 / §3.1 3.2
- **架构条款**：§11、§12
- **证据**：`src/server/observer.ts:120` 直接 `return send(response, 401, "Unauthorized")` 而不带 `WWW-Authenticate: Bearer` 与不返回 JSON
- **影响**：低（CSP `default-src 'self'` 与 `frame-ancestors 'none'` 已阻止跨源读取）；但 401 文案可能被其他本地进程读取来探测 Token 是否被尝试过
- **修复建议**：将 401 文案改为空 body 或 `{"error":"unauthorized"}`，并补 `WWW-Authenticate: Bearer realm="mad"`；更新 `tests-ts/observer.test.ts` 期望

#### P1-9 SSE 长轮询定时器 `setInterval(publish, 500)` 在 `request.close` 后才 `clearInterval`，无显式最大空闲超时

- **来源**：分册 #4 / §3.1 3.3
- **架构条款**：§11 流式接口
- **证据**：`src/server/observer.ts:164`：每个 500ms 周期调用 `publish().catch(() => response.end())`——一旦 `publish` 异常就 `response.end()` 而非 `send 500`，破坏连接语义；正常情况下仅在客户端 `close` 才清理 timer
- **影响**：低-中——SSE 连接长期保持会推迟端口释放
- **修复建议**：在 `publish()` 中维护 `offset` 与最近事件时间，超过 5 分钟无新事件后 `response.end()`；将 `setInterval` 替换成 `setTimeout` 链以便清除
