## 一、变更概述

本 PR 将 `mad serve` 的网页从"只读观察页"升级为"审议控制台"，实现了从网页端发起独立审议进程的完整能力。核心变更包括：

| 变更域 | 说明 |
|---|---|
| **新模块 `LaunchCoordinator`** | 管理审议启动协调，提供幂等键（requestId→deliberationId 持久映射）、冲突检测、进程发现与失败脱敏 |
| **新 API 端点** | `GET /api/launch-options`（安全发起选项）、`POST /api/launches`（发起审议）、`GET /api/launches/:requestId`（查询启动记录）、`POST /api/deliberations/:id/agent-id`（生成 Agent ID） |
| **CLI 扩展** | `deliberate` 新增 `--id`（指定审议 ID）与 `--web-plan`（通过网页确认方案）参数；`confirmPlan()` 重构为通过 `CheckpointMailbox` 双通道（终端 + 网页）接收方案确认 |
| **领域模型** | 新增 `candidateVersion` 追踪方案迭代；`PendingCheckpoint`/`CheckpointResponse` 增加 `data` 字段承载替换方案 |
| **前端** | 三步向导（议题→方案→Agent配置）、草稿持久化（sessionStorage）、活动审议跳转、响应式状态 |
| **术语/文档** | "审议观察页"→"审议控制台"、"问题"→"审议议题"；新增 ADR-0018；更新 CONTEXT.md 与 README.md |
| **测试** | 新增 5 个测试用例覆盖控制台启动、方案确认/替换/重新组局/取消、幂等重试、失败脱敏 |

## 二、逐文件审查

### 2.1 `src/server/launch-coordinator.ts`（新增，211 行）

**正向发现：**

- **幂等设计正确**：`launchExclusive()` 先检查 `requestId` 是否已有记录，再检查活动审议冲突，然后原子写入 `reserved` 记录，最后启动子进程。服务重启后 `read()` 通过 `withArchiveStatus()` 重新解析状态，不会重复派生。
- **原子写入**：`atomicJson()` 使用 `writeFile(flag: "wx")` + `rename`，避免了部分写入和竞态。
- **冲突检测合理**：`activeDeliberationId()` 使用 `process.kill(pid, 0)` 检测进程存活；`pendingDeliberationId()` 扫描所有启动记录中的 `reserved`/`spawned` 状态。
- **错误脱敏**：启动失败时对 `error.message` 调用 `redactAdapterDiagnostic()`。

**问题与建议：**

1. **[中] 轮询上限与超时**：`launchExclusive()` 在子进程 spawn 后轮询最多 50 次 × 100ms = 5 秒等待状态变为 `planning`，超时后返回 `spawned` 状态。如果组局阶段比较慢（比如 CLI 预检耗时较长），5 秒可能不足。建议：
   - 将轮询上限提升至 120 次（12 秒）或与 `withArchiveStatus` 中的 10 秒超时对齐为常量；
   - 考虑在返回 `spawned` 时附带提示信息方便前端展示"正在规划中"。

2. **[低] `pendingDeliberationId` 的竞态窗口**：该函数先 `readdir` 再逐个 `readFile`，若并发启动两个不同 requestId 的审议，第二个可能在第一个写入 `reserved` 记录之前通过冲突检查。但外层 `launch()` 用串行队列（`this.queue`）保证了调用顺序，实际不会触发。建议在 `pendingDeliberationId` 的注释中标注依赖串行队列的假设。

3. **[低] `spawned` 状态回归**：轮询超时返回 `spawned` 后，下一次 `read()` 会通过 `withArchiveStatus()` 尝试解析 archive 状态。但如果子进程在 5 秒后才开始写 archive，中间会有短暂窗口返回 `spawned` 而非 `planning`。这在 UI 上可能短暂显示"启动中"状态，不算 bug，但值得在状态转换文档中明确。

### 2.2 `src/server/observer.ts`（修改，+229 / -2）

**正向发现：**

- **安全公开视图**：`/api/launch-options` 只返回 CLI ID、预设 ID、模型名、上下文预算和可用性，不暴露 `executable`、认证信息或任意参数。预检失败的错误信息经过 `redactAdapterDiagnostic()` 脱敏。
- **输入校验严格**：
  - `requestId` 使用 `REQUEST_ID` 正则（`/^[a-zA-Z0-9_-]{1,80}$/`）；
  - `topic` 长度限制 1-5000 字符；
  - `mode` 仅接受 `structured`/`free`；
  - `organizer` 和 `limits` 通过 `resolveInvocation()` 和 `resolveLimits()` 校验；
  - `workspace` 长度上限 4096；
  - 拒绝未知字段（`allowed` 白名单）。
- **缓存合理**：`launchOptionsCache` 有 30 秒 TTL，避免每次刷新都重新预检所有 CLI 预设。

**问题与建议：**

4. **[中] `plan_confirmation` 检查点的 candidateVersion 校验**：`/api/checkpoints/:id/respond` 中对 `plan_confirmation` 增加了 `candidateVersion` 防陈旧检查，这是正确的。但 `guidance` 字段在非 `replace` 动作时没有额外校验——这在语义上没问题（用户可以附带指导），但建议在 API 文档中明确 `guidance` 与 `action` 的组合语义。

5. **[低] `agent-id` 端点缺少并发保护**：`POST /api/deliberations/:id/agent-id` 使用 `do...while` 循环生成不重复的 agent ID，但未加锁。如果两个请求同时到达同一个审议的规划阶段，可能生成相同 ID。建议：
   - 使用 `randomUUID()` 替代 `randomBytes(6).toString("hex")` 以消除碰撞可能（12 字符 hex 仅有 48 位熵，碰撞概率虽低但可避免）；
   - 或者在循环内重新读取 manifest 以获取最新的 `existing` 集合。

### 2.3 `src/cli/index.ts`（修改，+121 / -52）

**正向发现：**

- **双通道方案确认**：`confirmPlan()` 重构为通过 `CheckpointMailbox.wait()` 统一处理终端和网页两个通道，终端输入被包装为 `local` 回调。架构统一且优雅。
- **candidateVersion 追踪**：每次 `replace` 操作递增 `version`，`regroup` 重置为 0，确保网页端能校验方案版本避免陈旧提交。
- **状态覆盖扩展**：错误处理中增加了 `waiting_checkpoint` 状态的覆盖，避免组局阶段的检查点异常时状态未能正确落盘。
- **`--id` 参数**：允许 CLI 调用方指定审议 ID，使 `LaunchCoordinator` 能预先确定 ID 再启动子进程，是实现幂等映射的关键。

**问题与建议：**

6. **[中] `--web-plan` 与 `--auto` 的交互**：
   - `--web-plan` 会强制 `allowRegeneration = true` 并将 `planConfirmation` 设为 `interactive`，这是正确的。
   - 但如果用户同时传入 `--web-plan --auto`（如测试用例所示），审议进程会以 `auto` 交互策略运行但方案确认仍走网页。这种组合在语义上令人困惑：`--auto` 通常意味着无人值守，但方案确认却需要网页交互。建议在 `--web-plan` 存在时忽略 `--auto` 或至少发出警告。

7. **[低] `confirmPlan` 中的 `validationError` 持久化**：校验失败的错误信息直接写入 `PendingCheckpoint.summary`，会被展示在网页上。建议确认这些错误信息不会泄露内部路径或敏感配置。当前来源是 `parseDeliberationPlan` 和 `preflightPlan` 的错误，需要确认这些函数的错误信息是否都已脱敏。

## 三、架构与安全评估

### 3.1 安全边界 ✓

| 安全检查项 | 状态 | 说明 |
|---|---|---|
| Bearer Token 认证 | ✓ | 所有 `/api/` 端点继续要求 Token |
| 仅监听 loopback | ✓ | 服务固定 `127.0.0.1`，未变更 |
| 凭证不泄漏 | ✓ | `/api/launch-options` 不暴露 executable 路径；预检错误经 `redactAdapterDiagnostic` 脱敏；启动失败同样脱敏 |
| 可执行路径不可修改 | ✓ | 页面只能从 CLI 注册表选择 CLI/预设，不能修改可执行路径或认证 |
| 未知字段拒绝 | ✓ | `/api/launches` 和 `/api/checkpoints/:id/respond` 都使用白名单拒绝未知字段 |
| 审议进程仍是唯一状态所有者 | ✓ | 控制台只写启动协调记录，候选方案和检查点通过一次性文件信箱交给独立进程 |
| 控制台关闭不终止审议 | ✓ | 审议进程以 `detached: true` + `unref()` 启动 |
| candidateVersion 防重放 | ✓ | 网页提交检查点响应时校验版本号匹配 |

### 3.2 架构一致性 ✓

- **ADR-0012 修订**：原先"网页不能发起审议、命令行是主要发起入口"被修订为"控制台可以发起独立审议进程"，但核心约束不变——审议进程仍是运行状态和正式档案的唯一所有者。
- **ADR-0014 修订**：原先"观察服务只能响应检查点"被修订为"控制台可以发起审议并转发检查点响应"，但文件信箱协议保持一致。
- **状态所有权清晰**：`LaunchCoordinator` 只有自己的协调记录（`runtime/launches/*.json`），不触及审议档案；审议进程独立写入 `deliberations/:id/`。

### 3.3 幂等性设计 ✓

```
requestId → deliberationId 的映射：
1. 检查是否存在已持久化记录 → 存在则直接返回
2. 检查是否存在活动审议 → 存在则返回 409
3. 检查是否存在 pending 启动（reserved/spawned）→ 存在则返回 409
4. 原子写入 reserved 记录
5. spawn 子进程
6. 写入 spawned 记录
7. 轮询等待子进程进入 planning 状态
8. 返回 planning 或 spawned 或 failed
```

服务重启后步骤 1 保证幂等性，`withArchiveStatus()` 通过检查 archive 重新解析状态。

## 五、总体评价

**该 PR 是一次设计良好的架构升级**。核心决策——"控制台发起独立进程，而非将审议引擎嵌入服务进程"——保持了 ADR-0012 和 ADR-0014 确立的核心约束（审议进程是唯一状态所有者），同时显著提升了网页端的能力边界。

**安全方面**保持了高强度：所有新 API 继续要求 Bearer Token、不暴露可执行路径/凭证、使用白名单拒绝未知字段、错误信息脱敏、候选方案版本校验防重放。

**测试覆盖充分**：覆盖了正常路径、幂等重试、服务重启恢复、方案替换/重新组局/取消、失败脱敏等关键场景。

**主要建议在合并前处理 #1（轮询超时）和 #2（`--web-plan --auto` 语义）**。#4（测试超时）可在后续迭代中优化。

**审查结论：建议有条件通过（Approved with Suggestions）**，上述 2 个中等问题解决后即可合并。
