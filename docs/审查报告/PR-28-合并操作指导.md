## 1. 操作结论

**当前结论: REQUEST_CHANGES。保持 Draft，不得直接合并。**

PR 的架构方向正确，且已有真实 HTTP -> 独立审议进程 -> 规划检查点 -> 完成的 E2E 证据。但当前实现存在两个可达的正确性缺陷:

1. `plan_confirmation` 不接受 `pause`，而信箱在 Abort/SIGINT 时会写入 `pause`，可造成规划检查点永久等待，并污染恢复路径。
2. 子进程启动超过 10 秒未建档时，启动记录会被粘性标记为 `failed`。即使档案稍后出现，同一 `requestId` 也不会自愈，破坏幂等查询的真实性。

合并的最小充要条件是: 修复上述两项，补齐对应回归测试，并完成全量自动化验证。鉴于 PR 声明关闭 #18 - #27，Draft 转 Ready 前还应完成本文第 4 节的 API 契约收口和第 6 节的手工 UI 验收。

## 3. 合并阻断项: 必须在当前 PR 修复

### G1. 修复规划检查点的 pause/Abort 死等

**代码位置**

- [`src/cli/index.ts`](../../src/cli/index.ts): `confirmPlan()` 中 `plan_confirmation.actions`
- [`src/server/mailbox.ts`](../../src/server/mailbox.ts): `CheckpointMailbox.wait()` 的 Abort 响应和动作过滤

**已证实的失败时序**

1. guided 或 `--web-plan` 进入 `mailbox.wait()`。
2. SIGINT/Abort 调用 `submit(checkpointId, "pause")`。
3. `plan_confirmation.actions` 只含 `confirm|replace|regroup|cancel`。
4. 轮询拒绝 `pause`，但一次性响应文件已被占用，合法响应无法覆盖。
5. 进程不能正常进入 `PAUSED`，残留响应还可能影响 resume。

**实现要求**

- 在 `plan_confirmation.actions` 中增加 `pause`。
- `confirmPlan()` 收到 `pause` 后抛出 `MadError("PAUSED", ...)`，使现有 catch 路径将档案写为 `paused` 并在 finally 释放 `active.lock`。
- 加固 `CheckpointMailbox.wait()`: Abort 不应写入 `pending.actions` 不接受的动作。
- 清理或正确消费已完成的 request/response 文件，避免 resume 被旧 `pause` 响应卡住。

**必须新增的回归测试**

- guided 规划等待中 SIGINT -> `state.status === "paused"`。
- `active.lock` 已释放。
- `mad resume <id>` 能继续同一检查点，旧响应不会造成死等。
- 无效 Abort 动作不得占用一次性响应位。

### G2. 区分 spawn 硬失败和建档超时，允许启动记录自愈

**代码位置**

- [`src/server/launch-coordinator.ts`](../../src/server/launch-coordinator.ts): `read()`、`withArchiveStatus()`、启动轮询常量

**已证实的失败时序**

1. `spawn` 成功，子进程因冷启动或慢 IO 超过 10 秒未创建档案。
2. `read()` 或 pending 扫描调用 `withArchiveStatus()`，将记录改为 `failed` 并写回磁盘。
3. `withArchiveStatus()` 对所有 `failed` 记录直接返回，不再探测档案。
4. 子进程稍后正常进入 planning，但控制台对同一 `requestId` 永久返回失败。

**实现要求**

- 为 spawn 硬失败与档案未就绪分配不同的可机器识别状态或错误码。
- spawn 硬失败可保持终态; 档案超时应保持可探测，档案出现后可升级为 `planning`/`finished`。
- 将 5 秒启动等待与 10 秒超时整理为具名常量，明确它们的不同用途。超时不应单独作为永久失败证据。
- 保持 `requestId -> deliberationId` 映射不变，不得因重试派生第二个进程。

**必须新增的回归测试**

- 用 fake timer 或可注入时钟模拟建档超过阈值。
- 超时期间查询不得产生不可恢复的假失败。
- 档案出现后，原记录升级为 `planning`，且 deliberation ID 不变。
- 真实 spawn 异常仍为终态失败，且诊断已脱敏。

## 4. Draft 转 Ready 前的必修契约项

下列问题不都是独立的高危缺陷，但它们直接关系 #18 - #27 的验收声明。建议与 G1/G2 在同一 PR 收口; 若不处理，必须将对应 Issue 从 `Closes` 改为 follow-up 而非随 PR 自动关闭。

### R1. 统一活动审议与 pending 启动的公开状态

- `/api/launch-options` 的 `canLaunch` 和 POST 前置检查仅读 `active.lock`，而 coordinator 还会读 `reserved/spawned`。
- 将 `active.lock 或未终态 launch record` 统一定义为不可再发起。
- `GET /api/launch-options`、`POST /api/launches` 和 coordinator 必须共用同一判定，避免 UI 显示可发起而 POST 返回 409。
- 补测试: 仅有 `spawned` 记录、无 `active.lock` 时 `canLaunch === false`。

### R2. 在 API 边界预检 workspace

- 当前 API 仅校验字符串类型和长度，相对路径、不存在路径和非目录直到子进程才失败。
- POST 中要求绝对路径，再执行 `realpath` + `stat().isDirectory()`; 将规范化后的绝对路径传给 coordinator。
- 非法 workspace 返回 400，不得写 launch record，不得 spawn。
- 子进程保留二次校验，防止 TOCTOU 和非 HTTP 入口绕过。

### R3. 统一 API 错误 envelope 并脱敏顶层 500

- 当前 400/404/409/500 同时存在纯文本和 JSON; 前端在非 2xx 时统一将 body 当文本，会展示整段 JSON。
- API 错误统一为 `{ "code": "...", "message": "..." }`，业务附加字段保持结构化。
- 前端 `api()` 按 `Content-Type` 解析 JSON，优先显示 `message`/`error`。
- observer 顶层 catch 对外固定文案或使用 `redactAdapterDiagnostic`，不得原样返回未知 `error.message`。
- 补测试: 409 活动冲突、500 启动失败、404 launch 不存在均有稳定 JSON 契约且不泄漏 token/路径。

### R4. 修正以 `-` 开头议题的 argv 边界

- 当前 topic 位于选项之前，`--auto` 等议题会被 `parseArgs` 当作选项，导致合法请求失败。
- 这个问题当前是 fail-closed 的参数歧义，不是已证实的权限提升或 shell 注入。
- 正确的 argv 顺序是: 先放置所有选项，然后放置 `--`，最后放 topic。不得使用“`--`, topic, `--mode`, ...”，因为 `--mode` 也会被解析为位置参数。
- 补测试: topic 为 `--auto`、`-foo` 时不改变 mode/id/workspace，且能稳定发起。

### R5. 修正方案校验失败的 UI 语义

- 当前 `validationError` 与“当前候选方案已通过校验”同时显示。
- 错误态应明确说明: 本次修改未通过，上一版有效候选方案仍可确认。
- 不得同时展示成功与失败状态。

## 8. 争议审查项的整合裁决

| 原报告主张 | 裁决 | 理由 |
| --- | --- | --- |
| `pendingDeliberationId` 忽略 `planning` 会在“已建档但未建锁”时双派生 | **当前实现不成立，作为未来时序约束保留** | CLI 先执行 `ActiveDeliberationLock.acquire(id)`，后执行 `archive.create()`。只要此顺序不改，`planning` 档案存在时锁已先创建。应补并发测试保护不变量，但不应单独作为当前 P1。 |
| `launchExclusive` 持 queue 5 秒阻塞所有 HTTP | **降级为启动请求延迟** | queue 只包装 `launch()`; `GET /api/launches/:requestId` 直接调用 `read()`，不在 queue 上。它会串行后续 launch POST，这与单活动审议的目标一致，但可优化响应延迟。 |
| 公开 `--web-plan` 可绕过 `--auto-confirm-plan` 安全授权 | **不作为安全阻断** | `--web-plan` 不会自动接受方案，而是强制通过信箱显式确认。它是内部契约暴露/用法语义问题，可隐藏或更名，但不是无人值守方案的静默确认。 |
| `active.lock` 原子创建有空文件窗口 | **保留为低优先级加固** | 锁通过 `open("wx")` 抢占后再写内容，读方可短暂解析失败。但 coordinator 的 pending 记录可作第二层防护，且窗口极短。应优先改进读取重试/锁协议，不使用未证实的双派生表述。 |
| 草稿恢复、SSE 竞态和同档案脏编辑是 P1 | **缺陷成立，严重度调整为 UI 验收项** | 代码路径支持这些风险，但报告未提供浏览器级复现。在 PR 声明关闭 #27 时应手工验收；若复现则同 PR 修复，若不复现则留证据关闭。 |
| `validationError` 未脱敏是 P1 安全问题 | **降级为统一脱敏加固** | 多数错误来自结构校验，预检诊断通常已脱敏，尚无密钥直出复现。但该文本会入 checkpoint/event 并通过详情 API 回显，应在持久化前统一脱敏。 |
| 48 位随机 Agent ID 无并发保护是中危 | **降级为维护性加固** | 当前仅生成候选 ID，未写档案；48 位空间在本地交互负载下碰撞概率极低。可换 `randomUUID()` 或在最终方案校验中保持唯一性，不阻塞合并。 |
