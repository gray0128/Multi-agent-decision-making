# 分离结构化审议与自由讨论调度器

目标产品同时支持结构化审议和自由讨论：前者保留固定阶段、阶段内并行和明确检查点，后者不设预定义语义阶段，每个发言回合只调用一名参与者，并由主持 Agent 根据当前权威记录动态规划后续发言和建议结束时机。主持 Agent 与报告 Agent 都必须是参与者，可以由同一个审议 Agent 兼任；参与者不少于三名时组局器应优先分配给不同实例，但不作硬校验。持续运行的 TypeScript `DiscussionController` 是自由讨论的状态与硬限制控制者；主持 Agent 不维护 CLI 私有会话，只在覆盖周期开始及每个检查窗口边界以一次性调用规划下一个窗口，并在边界调用中同时评估上一窗口的收敛情况。主持调度调用计入总调用次数和成本，但不计为发言回合或参与者观点。自由讨论先执行覆盖周期，由主持 Agent 决定顺序并让每位参与者恰好发言一次；此后进入开放讨论，允许重复选择参与者但禁止连续发言。开放讨论每完成与参与者数量相同的发言回合就关闭一个检查窗口，执行收敛检查并允许用户暂停、继续或终止，窗口不要求覆盖所有参与者。应用以内置保守默认值、用户按次覆盖值和不可由普通命令突破的安全最大值共同约束参与者数量、检查窗口、总调用次数、超时和上下文预算，最终值进入审议方案并在恢复时保持不变；组局器可以看到但不能提高这些上限，具体默认数值在 Codex 纵向验证后校准。讨论结束后，两种模式使用相同的共同成果流水线：报告 Agent 生成草稿，其他参与者并行审阅，报告 Agent 完成一次最终修订；共同成果明确区分共识、未决争议、假设和风险，不把主持调度内容当作参与者观点。两种模式共享 CLI 注册表、CLI 适配器、组局阶段、上下文管理、审议档案和共同成果生成，却使用相互独立的调度器，避免把固定阶段和动态对话塞入同一个万能状态机。迁移时先完成结构化审议的 TypeScript 等价实现并验证基础设施，再增加自由讨论模式；自由讨论不依赖 Agent Framework、GroupChat 或任何 CLI 的私有会话。
