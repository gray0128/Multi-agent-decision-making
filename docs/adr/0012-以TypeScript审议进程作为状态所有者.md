# 以 TypeScript 审议进程作为状态所有者

应用以 TypeScript CLI 实现审议引擎，并移除 Microsoft Agent Framework 及其 DevUI。TypeScript 审议进程是运行状态的唯一所有者和审议档案的唯一写入者；命令行是发起、恢复、取消和管理审议的主要入口，审议观察页通过 GET 查询和 SSE 事件流展示历史与实时过程，并且只能使用带有一次性 `checkpoint_id` 的请求响应当前检查点，不能发起、恢复、删除审议或修改 CLI 注册表。观察页首版不使用前端框架，以静态 HTML/CSS 和少量原生 TypeScript 实现，使用带认证的流式 `fetch` 接收阶段、回合、调用生命周期、耗时、警告和检查点事件；参与者正文只有在 CLI 完成、解析并提交权威记录后才整体展示，不提供 token 级文本流，不发送 thinking/reasoning 事件或原始 stdout/stderr，并以独立解析和净化步骤渲染 Markdown。服务固定监听 `127.0.0.1`，不提供可配置监听地址、局域网或公网模式；`mad serve` 每次启动生成新的 Bearer Token，通过自动打开或打印的 URL fragment 交给页面，页面读取后立即从地址栏移除并仅保存在 `sessionStorage`，所有档案、事件和检查点请求都使用认证 `fetch`，不得把 Token 放入查询参数、长期 Cookie 或磁盘文件。首版要求用户安装 Node.js，以包含 CLI、核心引擎、适配器、档案、本地服务和页面资源的单个 npm 包分发，不建立 monorepo；只有在新架构通过可行性验证后，才考虑补充不依赖 Node.js 的独立可执行程序。迁移先以 Codex 适配器完成端到端验证，再逐个迁移 Claude Code、Reasonix、Grok、Pi、CodeBuddy 和 `agy`；每个适配器都必须覆盖调用预设校验、实际预检、只读调用、超时、取消、输出解析和脱敏。Python 与 TypeScript 在迁移期间允许并存，但配置与档案互不兼容且不双写；只有七个适配器、结构化审议、自由讨论、组局与预检、透明档案与恢复、本地观察服务、检查点、纯文本与工作目录模式、机器可读输出及文档全部通过接管验收后，才一次性删除 Python、`pyproject.toml`、Agent Framework 和 DevUI。新版本使用 `config/clis.toml` 保存 CLI 注册表，不兼容原有 `agents.toml`，也不读取、展示、迁移或恢复 Python 版本产生的审议档案；旧数据由用户自行保留或删除，迁移过程不得自动清理。CLI 成功时默认只向 stdout 输出最终 Markdown，`--format json` 时只输出一个完整 JSON 对象，进度、警告、Token 地址和档案路径进入 stderr；非 TTY 不输出 ANSI，中途失败不产生半截 JSON，不同失败状态使用不同退出码，机器结果包含审议 ID、状态、模式、报告、参与者、预算使用、警告和档案路径。该选择以自行实现受限检查点交互为代价，换取单一状态所有者、清晰的恢复边界，以及业务引擎和观察界面的独立演进。
