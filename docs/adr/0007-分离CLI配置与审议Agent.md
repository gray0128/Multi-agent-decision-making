# 分离 CLI 配置与审议 Agent

长期可信配置只描述 CLI 适配器、可执行程序、类型化允许选项、能力限制和模型调用预设，不预设角色，也不允许 `extra_args` 等原始命令参数透传；每个适配器定义并验证自己的配置 schema，且任何配置都不能覆盖只读、审批、工具限制、输出格式或工作目录等安全参数。`mad init` 只探测已安装 CLI 和可执行路径并生成带适配器选项说明的 `clis.toml` 骨架，不猜测模型、思考等级或默认组局器，也不覆盖已有配置，除非用户显式传入 `--force`；用户填写调用预设与 `[defaults.generator]` 后，使用 `mad config validate` 做静态校验，使用 `mad config check` 执行实际预检。每个预设固定模型及思考等级等允许推理设置，并由对应 CLI 适配器验证和映射，审议方案不得携带裸模型名或任意 CLI 参数。加载 `clis.toml` 时，适配器必须静态验证预设字段、类型、枚举和组合；运行时先实际预检组局器，方案生成后再对其中每个不同的 CLI 配置与调用预设组合预检一次，共享组合的多个审议 Agent 不重复预检。任何配置缺失、静态验证或运行时预检失败都阻止审议启动，不自动替换模型或思考等级。应用同时执行全局并发限制和 CLI 配置级并发限制；每个 CLI 配置首版默认并发为 1，其全部调用预设和审议 Agent 共享同一限流器，结构化阶段可以保持逻辑并行但只能在两层限流器允许时启动实际进程。每次审议都在组局阶段基于这些 CLI 配置生成只属于本次任务的审议 Agent，并为每个实例确定唯一 ID、模型调用预设和角色描述。同一 CLI 与预设可以生成多个不同角色的审议 Agent 并同时参与，它们按参与者身份正常计入流程规则，但页面和共同成果必须保留共享来源，不能把一致意见描述为独立模型的交叉验证。该分层让权限边界保持稳定、角色贴合具体问题，也允许单模型多角色审议；代价是每次审议都必须生成并持久化完整方案。
