# 分离观察服务与审议进程

`mad serve` 作为独立、长期运行的本地观察服务，负责展示审议档案、推送实时变化并接收受限的检查点响应；`mad deliberate --mode structured|free` 作为独立审议进程拥有运行状态、校验响应并写入正式档案，两者通过 `MAD_HOME/runtime` 下仅限当前用户访问的文件信箱协作。两种模式共享组局、预检、方案确认和公共命令参数，默认使用结构化审议与 guided 交互；`--auto` 显式切换自动模式，`mad resume` 从档案恢复原审议和交互模式，不允许切换。审议进程原子发布待处理检查点，观察服务只能原子写入带一次性 `checkpoint_id` 的响应文件，审议进程拒绝过期、重复或不匹配的响应并独占正式状态更新；终端与观察页的第一份有效响应获胜。guided 模式既无交互终端也无在线观察服务时立即报错，不无限等待。同一 `MAD_HOME` 首版通过原子全局锁只允许一个活动审议，历史档案浏览不受该锁限制。观察服务未启动或重启都不得中断已有审议：guided 模式可以退回终端检查点，自动模式继续按既定策略执行；审议进程退出也不得妨碍已落盘历史的浏览。相比为每次审议嵌入临时 Web 服务或维护 Unix Socket，这一分离增加了文件信箱协议和过期清理职责，但避免端口、连接和页面生命周期绑定到单次命令，并使自动化 CLI 不依赖页面存在。
