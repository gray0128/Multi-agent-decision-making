# 每次审议固定执行组局阶段

每次审议都在正式交流前执行组局阶段：应用使用默认的 CLI 配置与模型调用预设组合，或用户为本次任务显式指定的允许组合，根据问题和 CLI 注册表安全视图生成审议 Agent、角色及完整审议方案，并在方案确认后启动。默认组局器在 `clis.toml` 的 `[defaults.generator]` 中引用 `cli` 与 `preset`，用户可以通过单次命令参数覆盖；默认或覆盖组合无效、不可用或预检失败时直接报错，不自动降级。交互调用可以确认方案、增删审议 Agent、修改角色或调用预设、改选主持或报告 Agent，也可以附带指导重新组局或取消；所有引用仍受白名单与资源上限约束，新增组合重新预检，修改或重新组局后的完整方案必须再次展示并确认。非交互调用没有显式 `--auto-confirm-plan` 时直接退出，传入该参数后也只能接受第一次生成且通过 schema、白名单和运行时预检的方案，并把自动确认方式写入档案，不能自动修改或重新组局。组局器只能引用 `clis.toml` 中的 CLI 配置和模型调用预设，不能生成可执行路径、CLI 参数、权限、裸模型名或任意推理设置。相比允许用户跳过组局直接选择长期 Agent，这一固定步骤增加了一次模型调用和确认，但让角色始终围绕当前问题生成，并使长期配置只承担可信执行边界。
