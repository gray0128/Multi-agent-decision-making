# Pool CLI 适用性分析

> 状态：分析报告。评估 Poolside `pool` CLI（v1.0.14）是否适合作为 [Multi Agent Decision](https://github.com/gray0128/Multi-agent-decision-making) 项目的 CLI 适配器，或作为项目开发工具。

---

## 1. 背景

### 1.1 项目概述

**Multi Agent Decision（mad）** 是一个本地、可审计、可恢复的多 Agent 审议工具。它通过用户已安装并认证的 AI 编程 CLI 执行一次性非交互调用，支持结构化审议、自由讨论、透明档案和本地审议控制台。

项目的核心设计原则：

- **TypeScript 单包架构**，Node.js 22+，`mad` 可执行入口
- **多 CLI 适配器**：Codex、Claude Code、Reasonix、Grok、Pi、CodeBuddy、AGY
- **严格安全边界**：非持久会话、禁用记忆/历史、禁用 Web 搜索、固定只读工具集、禁用子 Agent、禁用 MCP
- **一次性非交互调用**：通过 `runProcess` 执行子进程，不依赖 CLI 私有会话
- **项目只读门禁**：通过 `verifyReadOnlyWithCanary` 验证 CLI 的只读能力

### 1.2 Pool CLI 概述

**Pool** 是 Poolside 的 ACP（Agent Client Protocol）客户端/服务器 CLI，当前版本 v1.0.14。

主要功能：

| 功能 | 命令 | 说明 |
|---|---|---|
| 交互式会话 | `pool` / `pool acp` | 启动 ACP 客户端或服务器 |
| 非交互执行 | `pool exec` | 一次性任务执行 |
| 认证 | `pool login` / `pool logout` | 登录登出 |
| 配置管理 | `pool config` | 显示配置和日志目录 |
| MCP 管理 | `pool mcp` | 管理 MCP 服务器 |
| 密钥管理 | `pool secrets` | 管理密钥 |
| 更新 | `pool update` | 更新版本 |

## 2. Pool 作为 mad CLI 适配器的可行性

### 2.1 能力映射

将 `pool` 添加为 `mad` 的 CLI 适配器，需要满足 `CliAdapter` 接口的所有要求。以下是能力映射：

| mad 适配器要求 | Pool 支持情况 | 详情 |
|---|---|---|
| **非交互调用** | ✅ `pool exec -p "prompt"` | 支持通过 `-p` 参数传递提示，或 `-` 从 stdin 读取 |
| **探测命令** | ✅ `pool --version` | 返回版本号 `1.0.14` |
| **模型指定** | ✅ `-m <model>` | 通过 `--model` 标志覆盖默认模型 |
| **工作目录指定** | ✅ `-d <dir>` | 通过 `--directory` 标志 |
| **JSON 输出** | ✅ `-o json` | NLJSON 格式，每行一个 JSON 对象 |
| **沙箱/只读模式** | ✅ `--sandbox required` | 沙箱模式可选 `required` / `disabled` |
| **stdin 输入** | ✅ `pool exec -` | 从 stdin 读取提示 |
| **超时控制** | ✅ 通过 `runProcess` | mad 的 `runProcess` 提供超时机制 |
| **退出码语义** | ✅ 0=成功，4=失败 | 其他退出码表示意外错误 |

### 2.2 安全边界对比

mad 对每个 CLI 适配器都强加固定的安全边界参数。以下是对比：

| 安全边界 | mad 要求 | Pool 当前支持 | 差距 |
|---|---|---|---|
| **非持久会话** | `--no-session-persistence` | ❌ `pool exec` 似乎是一次性执行，但未明确保证不持久化会话 | ⚠️ 需验证 |
| **禁用记忆/历史** | `--no-memory` / `--no-context-files` | ❌ 无对应参数 | ⚠️ 需验证 |
| **禁用 Web 搜索** | `--disable-web-search` | ❌ 无对应参数 | ⚠️ 需验证 |
| **固定工具集** | `--tools Read,Glob,Grep` | ❌ 无对应参数 | ⚠️ 需验证 |
| **禁用子 Agent** | `--no-subagents` | ❌ 无对应参数 | ⚠️ 需验证 |
| **禁用 MCP** | `--strict-mcp-config` + `--mcp-config '{"mcpServers":{}}'` | ❌ 无对应参数 | ⚠️ 需验证 |
| **禁用计划/自省** | `--no-plan` | ❌ 无对应参数 | ⚠️ 需验证 |
| **沙箱模式** | 各 CLI 自有 | ✅ `--sandbox required` | ✅ |
| **自动审批** | `--permission-mode dontAsk` | ✅ `--unsafe-auto-allow` | ✅ |

**关键问题**：Pool 缺少 mad 所需的细粒度安全边界控制参数。mad 的设计 philosophy 是通过显式的 CLI 参数固定安全边界，而 Pool 的 `pool exec` 模式似乎将这些控制留给了沙箱机制本身，而非通过命令行参数。

### 2.3 输出格式兼容性

Pool 的 JSON 输出格式（NLJSON）如下：

```json
{"reasoning":"...","type":"reasoning"}
{"message":"hello","type":"assistantMessage"}
{"thought":"...","type":"thought"}
{"args":{"success":true},"name":"exit","type":"toolCall"}
{"result":"","type":"toolCallResult"}
```

mad 的 `publicText` 函数目前处理的格式：

- 直接 JSON 对象（包含 `result`、`message`、`text` 等键）
- JSONL 事件流（`type` 为 `result`、`item.completed`、`message_end`、`assistant`、`agent_message`、`message`）

**差距**：Pool 的事件类型（`assistantMessage`、`toolCallResult`、`reasoning`、`thought`）与 mad 当前处理的类型不完全匹配。需要扩展 `publicText` 以识别 `type: "assistantMessage"` 的事件并提取 `message` 字段。

### 2.4 项目只读能力

mad 通过 `verifyReadOnlyWithCanary` 验证 CLI 的只读能力：

1. 创建临时目录，写入 `readable.txt`（含随机 nonce）
2. 调用 CLI 读取 `readable.txt` 并尝试写入 `must-not-exist.txt`
3. 检查文件是否被创建，验证 nonce 匹配

Pool 的 `--sandbox required` 模式可能满足这一要求，但需要实际测试验证：

- 沙箱模式是否真正阻止文件写入？
- 沙箱模式下是否允许读取指定目录？
- 沙箱模式是否允许工具调用（Read、Glob、Grep）？

### 2.5 适配器实现难点

如果要将 Pool 添加为 mad 适配器，需要解决以下问题：

1. **安全边界验证**：必须通过实际测试验证 `pool exec --sandbox required` 是否真正提供只读保证，不能仅依赖沙箱模式声明。
2. **输出解析**：需要扩展 `publicText` 和 `publicError` 以处理 Pool 的 NLJSON 格式。
3. **参数映射**：Pool 的参数与 mad 的安全边界参数不直接对应，需要找到等效的控制方式。
4. **错误处理**：Pool 的退出码语义（0=成功，4=失败）需要映射到 mad 的错误处理逻辑。

## 3. Pool 作为项目开发工具的可行性

### 3.1 优势

| 优势 | 详情 |
|---|---|
| **ACP 协议支持** | Pool 基于 ACP 协议，可以与多种 IDE 集成（Zed、JetBrains） |
| **沙箱模式** | `--sandbox required` 提供安全的开发环境 |
| **JSON 输出** | `-o json` 支持结构化输出，便于自动化 |
| **非交互模式** | `pool exec` 支持 CI/CD 集成 |
| **MCP 管理** | 内置 MCP 服务器管理，便于扩展工具链 |
| **Trajectories 记录** | 提供 trajectory 记录，便于调试和审计 |

### 3.2 劣势

| 劳节 | 详情 |
|---|---|
| **登录要求** | 需要 `pool login` 认证，无法完全离线 |
| **网络依赖** | 默认 API URL 为 `https://inference.poolside.ai`，需要网络访问 |
| **版本新** | v1.0.14 仍是早期版本，生态和文档可能不够成熟 |
| **平台限制** | 可能仅在特定平台上提供完整功能 |

### 3.3 开发工作流建议

如果将 Pool 用于本项目开发：

```bash
# 非交互执行开发任务
pool exec -p "运行单元测试" --sandbox required --unsafe-auto-allow

# 使用 JSON 输出进行自动化
pool exec -p "检查 TypeScript 类型" -o json --sandbox required

# 在工作目录执行
pool exec -d /path/to/project -p "运行 npm run build" --sandbox required
```

## 4. 兼容性评估总结

### 4.1 作为 mad CLI 适配器

| 评估维度 | 得分 | 说明 |
|---|---|---|
| **基本调用能力** | ✅ 8/10 | 支持非交互调用、JSON 输出、模型指定、工作目录控制 |
| **安全边界** | ⚠️ 5/10 | 缺少细粒度安全参数，需要通过沙箱模式验证 |
| **输出解析** | ⚠️ 6/10 | 需要扩展 `publicText` 以处理 Pool 的 NLJSON 格式 |
| **只读验证** | ⚠️ 6/10 | 需要实际测试沙箱模式的只读保证 |
| **错误处理** | ✅ 7/10 | 退出码语义清晰，易于映射 |
| **整体可行性** | ⚠️ 6/10 | 可行但需要额外的安全验证和输出解析工作 |

**结论**：Pool **可以**作为 mad 的 CLI 适配器，但需要：

1. **安全验证**：通过 `verifyReadOnlyWithCanary` 实际测试 `pool exec --sandbox required` 的只读能力
2. **输出扩展**：扩展 `publicText` 和 `publicError` 以处理 Pool 的 NLJSON 格式
3. **参数映射**：找到 Pool 等效的安全边界控制方式，或接受沙箱模式作为唯一安全保证
4. **测试覆盖**：添加 Pool 适配器的单元测试、配置测试、只读验证测试

### 4.2 作为项目开发工具

| 评估维度 | 得分 | 说明 |
|---|---|---|
| **开发体验** | ✅ 8/10 | ACP 协议、沙箱模式、JSON 输出 |
| **CI/CD 集成** | ✅ 7/10 | 非交互模式、结构化输出 |
| **安全性** | ✅ 8/10 | 沙箱模式、自动审批控制 |
| **网络要求** | ⚠️ 5/10 | 需要登录和网络访问 |
| **整体适用性** | ✅ 7/10 | 适合作为辅助开发工具 |

**结论**：Pool **适合**作为项目开发工具，特别是在需要沙箱隔离、结构化输出和 CI/CD 集成的场景下。

## 5. 建议

### 5.1 短期建议

1. **作为开发工具使用**：将 Pool 用于项目开发任务（代码生成、测试执行、代码审查等），充分发挥其 ACP 协议和沙箱模式的优势。
2. **安全验证**：通过 `verifyReadOnlyWithCanary` 测试 `pool exec --sandbox required` 的只读能力，为未来作为适配器做准备。
3. **输出格式分析**：分析 Pool 的 NLJSON 输出格式，为 `publicText` 扩展做准备。

### 5.2 长期建议

如果决定将 Pool 添加为 mad 的 CLI 适配器：

1. **创建 `src/adapters/pool.ts`**：实现 `CliAdapter` 接口，处理 Pool 特有的调用逻辑。
2. **扩展 `publicText`**：添加对 `type: "assistantMessage"` 事件的处理。
3. **添加配置支持**：在 `config.ts` 中添加 `pool` 到 `ADAPTER_IDS`，在 `buildConfigTemplate` 中添加模板。
4. **添加测试**：在 `adapters-ts.test.ts`、`config.test.ts`、`read-only.test.ts` 中添加 Pool 相关测试。
5. **更新文档**：在 `docs/各agent信息收集/` 中添加 Pool 的模型与参数信息。

### 5.3 风险提示

- **安全边界不确定**：Pool 缺少 mad 所需的细粒度安全参数，沙箱模式是否足以保证只读访问尚需验证。
- **输出格式不兼容**：Pool 的 NLJSON 格式需要 `publicText` 扩展，可能影响现有适配器的输出解析。
- **登录要求**：Pool 需要登录认证，无法完全离线使用，这可能不适合 mad 的本地优先设计。
