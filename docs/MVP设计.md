# 本地多 Agent 审议应用 MVP 设计

## 目标

构建一个仅在 macOS 本机运行的多 Agent 审议应用。应用通过 Microsoft Agent Framework 编排已经安装和认证的 AI CLI 运行时，使多个 Agent 配置围绕一个问题进行结构化、多阶段交流，最后生成可核查的高质量报告。

应用既可通过原生 DevUI 供用户交互，也可通过 `mad` 命令被 Codex、Claude Code 等 Coding Agent 在开发过程中直接调用。两个入口共用同一审议引擎，不重新接入各 CLI 背后的模型 API。

## MVP 范围

### 包含

- Python 3.12、`uv` 和锁定版本的 Microsoft Agent Framework、DevUI。
- Codex、Claude Code、Reasonix、Grok、Pi、CodeBuddy 六类内置 CLI 适配器。
- 同一 CLI 适配器注册多个模型或服务商组合的 Agent 配置。
- 纯文本问题，以及带可选工作目录的项目问题。
- 结构化审议、并行阶段、用户检查点、报告审阅、失败降级、取消和恢复。
- 原生 DevUI 与可供其他 Coding Agent 调用的 `mad` 命令行入口。
- 透明文件形式的审议档案、临时文件管理及本地诊断。

### 不包含

- Windows 或 Linux 的正式支持。
- 自定义 Web 前端或远程访问。
- 任意 shell 命令模板形式的自定义适配器。
- 多个审议同时运行。
- 修改项目文件或执行会改变外部状态的操作。
- 自由讨论式 Group Chat；可在后续作为独立模式增加。
- 数据库、远程知识库、图片或通用附件解析。

## 核心模型

- **CLI 运行时**：本机已经安装和认证的 AI CLI 程序。
- **CLI 适配器**：负责安全启动某类 CLI、传递输入、归一化输出、取消进程和映射错误。
- **Agent 配置**：可被选择的独立身份，由适配器、模型或服务商、名称、参数及稳定角色组成。
- **参与者**：一次审议实际选中的 Agent 配置。
- **组局 Agent**：可选的 Agent 配置，只能从注册表中建议参与者、临时角色和报告 Agent。
- **报告 Agent**：负责报告草拟和最终修订的参与者。
- **审议记录**：由应用统一维护的权威交流历史。
- **审议档案**：一次审议的状态、记录、报告和诊断资料集合。

完整定义见项目根目录的 `CONTEXT.md`。

## 总体架构

```text
原生 DevUI                    mad 命令行
    │                              │
    └────────── 应用用例层 ────────┘
                   │
          审议 Workflow 状态机
                   │
     ┌─────────────┼─────────────┐
     │             │             │
 Agent 注册表   审议档案仓库   材料快照服务
     │
 CLI 适配器注册中心
     │
 Codex / Claude Code / Reasonix / Grok / Pi / CodeBuddy
```

UI、CLI 和持久化层不得自行复制审议规则。所有入口都调用同一用例层和 Workflow。

## Agent 注册表

全局 `agents.toml` 位于用户应用数据目录。它只保存非秘密配置；API Key、令牌及登录状态继续由各 CLI 或环境变量管理。

示例：

```toml
[[agents]]
id = "pi-deepseek"
name = "DeepSeek 分析员"
adapter = "pi"
model = "deepseek-v4-pro"
role = "侧重严谨推理、识别假设和构造反例"
enabled = true
default_report = false
context_budget = 64000

[[agents]]
id = "pi-minimax"
name = "MiniMax 审稿人"
adapter = "pi"
model = "minimax-M3"
role = "侧重发现遗漏、矛盾和不可执行的建议"
enabled = true
default_report = true
context_budget = 64000
```

允许覆盖可执行文件路径、模型、额外参数、超时和上下文预算，但参数必须以数组形式传给子进程，禁止 shell 解析。修改注册表后重启应用生效。

同一适配器可以注册多个 Agent 配置。Agent ID 必须唯一且稳定，用于审议记录、恢复和机器输出。

## 审议输入

一次审议包含：

- 问题文本；
- 可选工作目录；
- 工作目录访问模式：默认材料快照，或显式直接只读访问；
- 参与者；
- 报告 Agent；
- 自动模式或引导模式；
- 可选组局 Agent；
- 可选输出语言覆盖，默认跟随问题主要语言。

不提供工作目录时进行纯文本审议。命令行不会隐式使用当前目录，项目分析必须显式传入 `--workspace/-w`。

## 组局流程

组局功能默认关闭并按次启用。启用后：

1. 用户选择一个已注册且预检成功的 Agent 配置作为组局 Agent。
2. 组局 Agent 读取问题和 Agent 注册表的安全视图。
3. 组局 Agent 建议参与者、临时角色和报告 Agent。
4. 用户确认或修改审议方案。
5. 应用验证方案后启动正式审议。

组局 Agent 不能修改 `agents.toml`，不能创建新的 CLI 参数、模型名称、命令或秘密。

## 审议状态机

```text
配置与确认
    ↓
参与者预检
    ↓
独立陈述（并行）
    ↓
检查点一：继续 / 指导 / 取消
    ↓
质疑与补充（并行）
    ↓
检查点二：继续 / 指导 / 取消
    ↓
修订意见（并行）
    ↓
报告 Agent 草拟报告
    ↓
检查点三：继续 / 指导 / 取消
    ↓
其他参与者审阅报告（并行）
    ↓
报告 Agent 修订一次
    ↓
形成共同成果
```

引导模式默认启用三个检查点。自动模式跳过正常检查点，但参与者不足或报告 Agent 失败等异常仍可请求用户处理。DevUI 默认使用引导模式；程序化 CLI 调用默认使用自动模式。交互式 CLI 只有显式传入 `--interactive` 才启用检查点。

同一阶段的参与者不能看到彼此尚未完成的输出。应用在所有调用成功、失败或超时后越过阶段屏障，并将完整阶段结果提供给下一阶段。

## 输出长度与上下文

阶段提示词给出以下建议长度：

- 独立陈述：约 1,500 字；
- 质疑与补充：约 1,000 字；
- 修订意见：约 1,000 字；
- 报告审阅：约 1,000 字；
- 最终报告：约 5,000 字。

完整原始输出始终保存。后续参与者优先接收完整审议记录；若预计超过任一参与者声明的上下文预算，则由报告 Agent 生成一份全员共用的审议摘要。摘要必须保留原问题、全部用户指导、关键论点、分歧、假设和资料来源，并作为正式事件进入审议记录。

## 最终报告质量要求

报告结构可以根据问题类型调整，但必须覆盖：

1. 执行摘要；
2. 明确结论或建议；
3. 支撑结论的关键理由；
4. 尚存分歧、假设和不确定性；
5. 可核查资料来源，或明确说明缺少来源；
6. 建议的下一步行动。

参与者、失败、重试、耗时等运行信息保存在诊断文件中，不混入正式报告正文。

项目文档和审议档案的结构化字段使用中文；Agent 发言和最终报告默认跟随问题语言，用户可在开始前覆盖。

## 工作目录与安全

### 纯文本模式

没有工作目录时不创建材料快照，参与者只接收问题、阶段指令和审议记录。

### 材料快照模式

提供工作目录时默认使用快照：

- Git 仓库默认包含已跟踪文件，以及用户明确允许的未跟踪文件；
- 非 Git 目录默认排除隐藏内容；
- 始终排除版本库元数据、环境变量文件、密钥、证书、认证目录、依赖和构建产物；
- 内置敏感规则不能关闭，用户只能追加排除规则；
- 创建前展示包含与排除统计；
- 所有参与者读取同一份快照。

### 直接工作目录模式

用户可以显式传入 `--direct-workspace` 或在 DevUI 中选择直接模式。应用必须显示警告并在审议档案中记录该选择。该模式仍启用各适配器的只读、计划或禁用写工具选项，但不提供快照级隔离。

所有模式都禁止修改用户文件和执行有副作用的操作。疑似秘密不得进入 UI、正式记录或原始诊断。

## CLI 适配器要求

每个内置适配器负责：

- 定位或使用配置的可执行文件；
- 校验模型和非交互调用参数；
- 使用参数数组且设置 `shell = false`；
- 传入问题、阶段指令、统一上下文和只读工作位置；
- 在能力允许时解析流式 JSON，否则安全处理纯文本；
- 只把公开回答写入正式记录，原始运行事件进入脱敏诊断；
- 强制超时、取消整个子进程组并回收资源；
- 使用对应 CLI 的最严格可用只读或计划模式；
- 阻止参与者再次递归调用 `mad`。

若某类 CLI 无法满足最低只读约束，适配器应在项目模式中标记不可用，而不是仅依赖提示词承诺。

## 预检与容错

正式审议开始前，对全部参与者执行极短的真实预检，并缓存结果 10 分钟。失败时允许用户重试、移除该参与者或取消。

每次正式调用：

- 默认超时 5 分钟，可配置；
- 失败后自动重试一次；
- 再次失败则记录并跳过；
- 仍有至少两名成功参与者时继续；
- 少于两名时暂停并请求处理；
- 报告 Agent 失败时请求用户选择替代者。

默认最多同时执行 6 个 CLI 调用，允许配置为 1–6。MVP 同一时间只允许一个活动审议，但可以浏览历史档案。

## 取消与恢复

取消时终止当前阶段全部仍在运行的 CLI 子进程。已完整完成的阶段继续保留；当前阶段的部分输出只进入脱敏诊断，不进入正式审议记录。

恢复时从未完成阶段整体重跑，不混用取消前的部分结果。仍可恢复的审议所需材料快照不能被临时文件清理功能删除。

## DevUI 入口

- 通过工作流结构化输入 schema 生成启动表单。
- 展示阶段、参与者输出、检查点、错误和最终报告。
- 固定监听 `127.0.0.1`。
- 保留启动时生成的随机 Bearer Token 认证。
- 不提供局域网或公网监听选项。
- DevUI 只作为本地开发工具界面，不视为生产级服务。

## 命令行入口

建议命令名为 `mad`，当前机器未发现同名命令。

```bash
# 启动 DevUI
mad serve

# 纯文本自动审议
mad deliberate "比较事件驱动和请求响应架构"

# 分析当前项目，默认使用材料快照
mad deliberate "检查当前设计的并发风险" --workspace .

# 显式直接读取当前项目
mad deliberate "检查未提交改动" --workspace . --direct-workspace

# 供 Coding Agent 调用并返回机器可读结果
mad deliberate \
  "比较两个重构方案" \
  --workspace . \
  --agents pi-deepseek,pi-minimax,claude-opus \
  --report-agent claude-opus \
  --non-interactive \
  --format json
```

非交互调用未指定参与者时，选择所有启用且预检成功的 Agent 配置；未指定报告 Agent 时，先使用注册表中唯一的默认报告 Agent，否则使用注册表顺序中第一个预检成功的参与者。只有传入 `--organizer <agent-id>` 才启用组局功能。

进度和警告写入标准错误，最终 Markdown 或 JSON 写入标准输出。JSON 至少包含审议 ID、状态、最终报告、警告、实际审议方案和档案路径。

退出码：

- `0`：最终报告成功生成，包括带警告的降级成功；
- `1`：工作流失败或无法生成最终报告；
- `2`：参数或配置错误；
- `3`：可用参与者不足；
- `130`：用户取消或收到终止信号。

## 存储布局

默认使用 macOS 用户应用数据目录：

```text
~/Library/Application Support/MultiAgentDecision/
├── config/
│   └── agents.toml
├── deliberations/
│   └── <审议ID>/
│       ├── metadata.json
│       ├── plan.json
│       ├── transcript.jsonl
│       ├── report.md
│       └── diagnostics.jsonl
├── logs/
└── temp/
```

根路径允许通过环境变量覆盖。审议档案默认永久保留且只能由用户二次确认后删除；普通应用日志默认保留 14 天；预检缓存保留 10 分钟；完成或取消后的材料快照立即清理。

异常遗留的孤立临时文件可通过 DevUI 和 CLI 的独立入口查看路径、所属审议、创建时间及占用空间，并支持二次确认后的单项或批量清理。临时清理不得删除审议档案，也不得删除活动或可恢复审议正在使用的快照。

## 工程约束

- Python 3.12。
- 使用 `uv` 管理虚拟环境和锁文件。
- 精确锁定 Agent Framework 与 DevUI 版本，升级需先通过适配器与工作流回归测试。
- 使用异步子进程和进程组管理，不通过 shell 执行命令。
- 仅正式支持 macOS，路径、信号和快照实现封装为平台服务。
- 所有项目文档使用中文。

## 验收标准

MVP 至少应证明：

1. 可以从 `agents.toml` 加载同一 Pi 或 Reasonix 适配器下的两个不同模型 Agent 配置。
2. DevUI 可以通过结构化表单发起纯文本审议和项目审议。
3. `mad deliberate` 可以被另一个 Coding Agent 非交互调用，并只在标准输出返回约定结果。
4. 六个参与者能够在阶段内并行执行，并在阶段边界共享一致上下文。
5. 引导模式的三个检查点可以继续、添加指导或取消。
6. 单个参与者超时或失败不会阻断仍满足最低人数的审议。
7. 取消后能够从整个未完成阶段恢复。
8. 报告经过草拟、一次集体审阅和一次最终修订。
9. 不指定工作目录时不会读取当前目录；快照为项目模式默认值，直接模式必须显式启用。
10. 审议档案、日志和临时文件遵循既定保留与清理边界。
