# 审查结果综合分析与修改意见

## 1. 文档目的

本文综合分析 `docs/审查/` 下 19 份审查报告，以 `docs/TypeScript目标架构.md` 和当前源码为裁决依据，形成可执行的修改意见。本文不是对既有报告的简单汇总：对相互冲突、证据过时或严重度失真的结论进行了重新核验，并区分为“采纳”“调整后采纳”“不采纳”和“后续优化”。

审查基线：2026-07-21 当前工作树，Git 基线提交 `7e90f6a`。

## 2. 综合结论

当前 TypeScript 实现的主架构已经成立，组局、双模式 Controller、透明档案、恢复、CLI 注册表、本地观察服务和工作目录门禁均有明确代码落点。现状不属于需要推倒重做的实现，但还不能据此宣称所有目标架构约束均已闭环。

建议结论为：**有条件通过**。

优先修改集中在四个方面：

1. 项目只读能力目前是“配置了安全参数并做 READY 冒烟”，还不是“运行时证明写操作被阻断”。这是安全承诺的验证缺口。
2. 调用预设的 `context_budget` 只有正整数校验，没有安全最大值；最终 prompt 的预算校验也只对照预设预算，尚未完整纳入按次计划上限。
3. 警告写入 stderr 的行为不一致：首次项目审议会立即显示工作区提示，但共享来源警告不显示；恢复路径也没有统一输出最终警告集合。
4. 观察页 Markdown 采用自研正则子集，安全上先转义是正确的，但表格、围栏代码、链接和嵌套列表等报告常见结构无法完整呈现，尚未充分满足“独立解析与净化”的产品语义。

其余问题主要是纵深防御、可观测性、测试覆盖和可维护性，不应与上述四项混为同一优先级。

## 3. 既有审查结果的可信度判断

### 3.1 可直接采纳的结论

- 项目模式显式使用 `--workspace`，路径经 `realpath` 规范化，并写入 manifest；纯文本模式使用私有 scratch 目录。
- `CodexAdapter` 和 Generic adapters 的命令行参数由代码生成，`spawn` 使用 `shell: false`，不存在 `extra_args` 原始参数透传入口。
- `InvocationScheduler` 实现全局与 CLI 级两层限流；同一 CLI 的角色共享 CLI 级限流器。
- 调用在执行前冻结，完成后原子提交；恢复可复用已提交逻辑调用，权威 transcript 按 `logicalCallId` 去重。
- `mad serve` 固定监听 `127.0.0.1`，Bearer Token 通过 URL fragment 交付并只存入 `sessionStorage`，档案 API 和 SSE 均需鉴权。
- 页面在 Markdown 转换前先执行 HTML 转义，当前未发现直接注入未净化原始 HTML 的路径。

### 3.2 调整后采纳的结论

| 议题 | 既有报告说法 | 复核结论 |
|---|---|---|
| warnings 与 stderr | 有报告称警告完全不可见，也有报告称已全部进入 stderr | 首次项目审议在解析 workspace 后会单独写一条 stderr 提示；`warnings` 数组中的共享来源警告没有统一写 stderr。恢复时也仅单独输出 workspace 提示。应按“部分符合”处理。 |
| 诊断脱敏 | 有报告称所有 Generic adapter 都把原始 stderr 写入档案 | 当前 `GenericCliAdapter.invoke()` 已调用 `redact()`；该具体漏洞已不存在。但 `ArchiveStore.appendDiagnostic()` 接受任意对象且没有末端兜底，纵深防御建议仍成立。 |
| 上下文摘要恢复 | 有报告认为 `summarizedEntries` 未持久化会导致恢复重复计费 | Controller 恢复时会重放确定性逻辑，已提交摘要通过稳定 `logicalCallId` 命中缓存，不会直接重复调用。仍应增加“摘要后崩溃再恢复”的验收测试，但不应把它列为已确认缺陷。 |

## 4. 修改意见与验收标准

### P1-1：把项目只读能力从参数声明提升为可验证契约

**现状证据**

- Codex 固定使用 `--sandbox read-only`；Claude、Grok、CodeBuddy、Pi、agy 使用各自的 plan/sandbox/禁写工具参数；Reasonix 标记为 `supportsProjectReadOnly=false`。
- `check()` 的正式调用只要求“不要使用工具，只回复 READY”。它能证明 CLI 可应答，但不能证明写操作在运行时确实被拒绝。

**修改建议**

1. 将适配器能力从单一布尔值升级为明确的 capability contract，例如区分 `projectReadOnly: unsupported | declared | verified`。
2. 为项目模式增加 CLI 专属的只读验证流程。优先采用可确定验证的 CLI 能力/版本检查；需要负向 canary 时只在应用创建的临时目录执行，绝不在用户 workspace 中试写。
3. 负向 canary 必须能证明“确实尝试了写操作且被运行时拒绝”。模型仅口头回复“不会写”不能算验证通过；证据不充分时应 fail closed。
4. 项目组局器和方案中的每个唯一 `CLI + preset` 组合都必须通过该验证；Reasonix 继续在项目模式中被拒绝。

**验收标准**

- fake adapter 在 canary 目录成功写入时，项目预检失败且正式审议不启动。
- fake adapter 的写请求被沙箱拒绝时，项目预检通过。
- 非项目模式不执行项目写入 canary。
- `supportsProjectReadOnly=false` 的组合在组局器、参与者、主持和报告角色中均无法进入项目审议。
- 预检日志只记录脱敏后的能力证据，不记录凭证或原始 thinking。

### P1-2：闭合上下文预算的安全上限与最终 prompt 校验

**现状证据**

- `resolveLimits()` 对按次 `contextBudget` 强制 `SAFE_MAX_LIMITS.contextBudget = 1_000_000`。
- `parsePreset()` 对 `context_budget` 只做正整数校验，没有最大值。
- `SharedContextManager` 使用 `min(plan limit, participant preset budgets)` 生成共享摘要，这是合理的保守策略。
- `InvocationRunner` 对最终 prompt 只比较当前 preset 的预算，没有同时比较 plan 的按次上限；摘要返回后追加的角色说明、问题、JSON schema 和用户指导也可能使最终 prompt 超过计划预算。

**修改建议**

1. 在配置解析时对 preset `context_budget` 施加受支持的硬上限。若直接复用 `SAFE_MAX_LIMITS.contextBudget`，应把该常量移动到无循环依赖的共享策略模块。
2. `InvocationRunner` 的有效输入上限应为 `min(plan.contextBudget, preset.contextBudget)`，组局调用也必须受本次 plan limits 约束。
3. 不建议把整个摘要策略直接塞进 `InvocationRunner`。更清晰的做法是让上下文构建器接收 `reserveTokens` 或目标调用预算，在组装完整 prompt 前为固定模板、问题、schema 和 guidance 预留空间；Runner 继续承担最终硬拒绝。
4. 保留共享摘要的单一版本，不按模型生成不同摘要。

**验收标准**

- 配置中的 preset 预算超过安全最大值时，`config validate` 失败。
- prompt 大于 plan 上限但小于 preset 上限时，在启动 CLI 前被拒绝或先完成统一摘要。
- 固定模板与 guidance 加入后仍不超过有效输入上限。
- 在摘要成功提交后模拟崩溃并恢复，不新增同一摘要逻辑调用的计费尝试。
- 增加中文、英文、代码块和超长问题的预算边界测试；估算算法可以保持保守，但必须记录误差策略。

### P1-3：统一 deliberate/resume 的 warning 生成、归档与 stderr 输出

**现状证据**

- 首次传入 workspace 时会立刻写 stderr。
- 共享 CLI/preset 来源警告只进入 `events.jsonl` 与 JSON 结果字段，默认 Markdown 模式不可见。
- `resume()` 重新生成 warning 集合，但在执行完成前没有统一输出；workspace 只通过另一条硬编码信息提示。

**修改建议**

1. 提取单一 `emitWarnings()`：同一 warning 集合在进入正式审议前写入 stderr，并追加为 archive event。
2. deliberate 与 resume 共享该逻辑；避免 workspace 警告被两套文案重复输出。
3. JSON 模式继续保留 `warnings` 字段，同时仍写 stderr。目标架构明确要求警告进入 stderr，机器可解析 stdout 的完整性不会因此受影响。

**验收标准**

- Markdown 与 JSON 两种格式的 stderr 都包含共享来源警告。
- 项目审议及其恢复都在任何参与者正式调用前显示 workspace 授权提示。
- stdout 仍分别只包含最终 Markdown 或单个 JSON 对象。
- 警告事件在一次运行中不重复追加；恢复时如需再次提醒，应有明确的 `resumed` 语义或去重规则。

### P1-4：完善观察页 Markdown 解析与净化

**现状证据**

- 当前实现先 `esc()` 再进行正则替换，XSS 基线是正确的。
- 解析器不完整支持围栏代码、表格、链接、引用和嵌套列表；报告恰好大量使用这些结构。

**修改建议**

1. 使用经过审计的 Markdown parser 与 sanitizer 组合，并通过构建时打包继续产出静态页面资源；不要直接信任 parser 的 HTML 输出。
2. 禁止原始 HTML，限制链接协议，外链增加安全属性；代码块、表格和列表应稳定显示。
3. 若暂不引入依赖，应在目标架构和用户文档中明确“仅支持 Markdown 子集”，并补齐当前子集的安全与渲染测试。但这只能作为过渡方案。

**验收标准**

- `<script>`、事件属性、`javascript:` URL 和畸形标签不能产生可执行 DOM。
- 标题、粗体、行内代码、围栏代码、表格、链接、引用及嵌套列表有快照测试。
- transcript 与 report 使用同一条解析/净化路径。
- 不展示 thinking、原始 stdout 或原始 stderr。

### P2-1：在档案边界增加统一诊断脱敏

Generic 与 Codex adapters 当前都做了脱敏，因此这不是已确认泄漏。但档案契约承诺 `diagnostics.jsonl` 为脱敏诊断，而 `appendDiagnostic(record: unknown)` 本身不保证这一点。建议保留 adapter 侧上下文相关脱敏，并在档案入口增加递归、限深、限长的兜底脱敏，覆盖 token、secret、password、Bearer、常见 key 前缀以及已加载的敏感环境变量值。

验收应包含嵌套对象、数组、错误消息、环境变量原值和超长字符串；同时避免把普通模型 ID、逻辑调用 ID误删。

### P2-2：补充运行时 schema 与审议结束原因

1. 服务端目前对 manifest/state/event 主要依赖 TypeScript 强制转换。建议让 archive writer、resume reader 和 observer reader 共享最小运行时 schema，至少校验 `schema_version`、状态枚举、plan、checkpoint 和事件公共字段。
2. 自由讨论在主持判定收敛、用户结束、达到窗口上限、暂停和取消时写入明确的 `discussion.ended` 或等价事件，并记录 `end_reason`。这不会改变 Controller 语义，但能让档案和页面解释“为什么停止”。

### P2-3：补齐契约型测试，而不是按报告逐项堆测试

建议优先补充以下测试：

- 项目只读负向 canary 与不支持适配器门禁。
- preset 安全最大值、plan/preset 双预算、完整 prompt 预留空间和摘要后恢复。
- deliberate/resume × Markdown/JSON 的 stdout/stderr 契约矩阵。
- 所有 adapter 的非零退出、损坏输出、超时、中止和脱敏；真实 CLI 验证作为显式启用的本地/发布验收套件，不放入默认离线单测。
- 两个不同 CLI 的全局并发上限、同一 CLI 的共享上限，以及并行阶段失败后等待 sibling settle。
- 自由讨论 `cancel`、observer 400/404/409、Markdown XSS 与复杂结构渲染。

### P3：完成测试保护后再做可维护性重构

以下建议成立，但不应抢在契约修复和测试之前：

- 拆分 `src/cli/index.ts` 中的参数解析、组局确认、运行编排、warning 输出和结果序列化。
- 将 Generic adapter 的命令构建与能力声明拆为 adapter-owned definition，配置层只负责聚合严格 schema。
- 将 Reasonix 特有的输出清理从通用 `publicText()` 中下沉到 adapter-specific normalizer。
- 让 observer 监听地址引用 `SERVER_HOST` 单一常量，消除字面量漂移。
- 保持 SSE 500ms 文件轮询；单用户本地服务没有证据表明需要引入 `fs.watch` 或额外依赖。

## 5. 建议实施顺序

1. 先补契约测试骨架：warnings、预算、项目只读、Markdown 安全。
2. 修复 P1-3 warnings，改动小且能立即闭合 CLI 契约。
3. 修复 P1-2 预算上限和完整 prompt 预算。
4. 实现 P1-1 adapter 能力验证；真实 CLI 的差异需要逐适配器验收。
5. 完成 P1-4 Markdown parser/sanitizer。
6. 增加档案末端脱敏、运行时 schema 和 `end_reason`。
7. 在上述测试全部稳定后拆分 CLI 与 adapter 结构。

每个阶段应独立提交，避免把安全语义修改、依赖引入和大规模重构放在同一个变更中。

## 6. 审查文档治理建议

当前 `docs/审查/` 同一主题常同时存在“详细报告”和“审查报告”，其中部分结论彼此冲突，且若干报告引用的源码行号与当前实现不一致。建议：

1. 将本文作为当前基线的综合入口，并在 `docs/审查/README.md`（如后续创建）维护“结论状态、基线 commit、是否已修复、替代报告”索引。
2. 每份新报告必须记录 Git commit、实际执行的命令、未执行项和证据边界；不要只记录日期。
3. 已证伪项保留原报告作为历史证据，但标记 `superseded`，不要直接删除，以免失去审计链。
4. 修复完成后更新综合表的状态，不反复重写原始独立报告。
