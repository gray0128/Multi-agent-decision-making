## 2. 目标架构要求摘要

- 第 5 节：每次审议先执行组局阶段（默认/覆盖组局器、提供问题/资源上限/安全视图、生成仅本次审议 Agent、指定报告/主持 Agent、唯一组合预检、确认后进入）；交互式确认支持确认/修改/重组局/取消且修改不破白名单与资源上限；非交互须显式 `--auto-confirm-plan` 且仅接受首份通过校验方案、不降级；同一 CLI+预设允许多角色且须保留共享来源。
- 第 6 节：默认 `structured`；`mad resume <id>` 从档案恢复、不允许切换；默认 guided，`--auto` 才跳过；自动模式仍需独立 `--auto-confirm-plan`；guided 无通道即失败。
- 第 7 节：结构化七阶段与屏障（独立陈述→质疑补充→修订与争议信号→一次争议收敛→报告草稿→并行审阅→最终修订），同阶段逻辑并行但受全局+CLI 限流器约束，同阶段看不到他人未完成输出；guided 在独立陈述/质疑补充/争议判定/报告草稿后等待检查点。
- 第 8 节：持续运行 DiscussionController 而非模型进程；先覆盖周期（每人恰好一次）再开放讨论（可重复但同参与者不连发）；每完成与人数相等的回合数结束一个检查窗口；窗口边界收敛评估；主持按窗口规划、首版无长期主持会话；主持调度计入成本但不算发言回合、非参与者观点；结束复用共享报告流水线并区分共识/未决争议/假设/风险。
- 第 10 节：检查点动作（继续/指导后继续/结束讨论(仅自由)/暂停/取消）；第一次 Ctrl-C 按暂停并终止当前 CLI 子进程；终端与页面第一份有效检查点响应获胜，过期/重复/ID 不匹配被拒。

## 3. 当前实现对照

### 要点 1：组局阶段前置（默认/覆盖组局器、安全视图、生成仅本次 Agent、指定报告/主持、唯一组合预检、确认后进入）

- 符合。
- 默认组局器与 `--organizer` 覆盖：`src/cli/index.ts:414-421`；默认取自 `registry.defaults.generator`，覆盖时解析为 `cli/preset`（`src/core/planning.ts:132`）。
- 向组局器提供问题、资源上限、CLI 注册表安全视图：`src/core/planning.ts:211-227`（`buildPrompt` 注入 `question`、`request.limits`、`registryView`，且明确禁止输出模型名/路径/秘密；项目模式通过 `cwd=workspace` 直接只读授权）。
- 生成仅属于本次审议的 Agent（唯一 ID、CLI 配置、调用预设、角色）：`src/core/planning.ts:66-80`（`parseDeliberationPlan` 校验 ID 格式与唯一性、解析 `cli/preset/role`）。
- 指定报告 Agent；自由讨论还指定主持 Agent：`src/core/planning.ts:83-101`（report_agent_id 必为参与者；free 模式额外要求 moderator_agent_id 且为参与者）。
- 对唯一调用组合执行运行时预检：`src/core/planning.ts:186-203`（`preflightPlan` 去重组合并 `requireReady` 校验就绪）。
- 方案确认后才进入正式审议：`src/cli/index.ts:493-510` 确认/写入 manifest 后，才于 `src/cli/index.ts:519-537` 启动对应 Controller。

### 要点 2：交互式确认支持确认/修改/附带指导重新组局/取消；修改不破白名单与资源上限；新增组合重新预检

- 符合。
- 确认（回车）、修改（完整 JSON）、重新组局（`/regroup 指导`）、取消（`/cancel`）：`src/cli/index.ts:217-260`（`confirmPlan` 循环）。
- 修改不突破白名单与资源上限：`src/core/planning.ts:61-65`（人数上限）、`:74`（`resolveInvocation` 校验白名单）、`:83-84`（报告 Agent 必须为参与者）；方案本身不含可改的资源上限字段，上限来自 `request.limits`，无法经修改突破。
- 新增组合重新预检：`src/cli/index.ts:255`（修改后再次 `preflightPlan`）；修改后回到循环再次确认（`src/cli/index.ts:249-256`）。

### 要点 3：非交互须显式 `--auto-confirm-plan`，仅接受首份通过校验方案，不自动修改/重组局；默认组局器无效/不可用/预检失败直接报错不降级

- 符合。
- `--auto` 必须同时显式传入 `--auto-confirm-plan`：`src/cli/index.ts:386-388`。
- 仅自动接受第一次生成并通过校验的方案、不重组局：非 guided 时 `allowRegeneration=false`，`propose` 仅尝试一次（`src/core/planning.ts:160-180`）；CLI 直接采用 `proposed.plan`，无修改/重组局分支（`src/cli/index.ts:502-503`）。
- 默认组局器无效/不可用/预检失败直接报错不降级：`src/core/planning.ts:133-138`（`resolveInvocation`/`requireReady` 抛错）、`:208`（预检失败抛 `PREFLIGHT`）、`preflightPlan` 抛错（`:194-208`）；全程无 fallback/降级分支。

### 要点 4：同一 CLI+预设可生成多个不同角色审议 Agent；页面/报告须保留共享来源，不得描述为独立模型交叉验证

- 符合。
- 允许多角色共享同一 cli/preset：`src/core/planning.ts:220` 组局提示明确「同一 cli/preset 可创建多个不同角色实例，但它们共享模型来源」。
- 报告/上下文保留共享来源、禁止交叉验证描述：`src/core/outcome.ts:5-13`（`sharedOriginWarning` 注入草稿与最终修订提示）；`src/core/context.ts:58-59` 滚动摘要指示「不得把共享来源说成独立模型验证」；CLI 亦将共享来源警告写入事件（`src/cli/index.ts:513-517`）。

### 要点 5：默认模式 structured；`mad resume <id>` 从档案恢复原模式/交互策略/方案/预算，不允许切换

- 符合。
- 默认 `structured`：`src/cli/index.ts:363`。
- resume 恢复且不切换：`src/cli/index.ts:575-735` 全程使用 `manifest.mode`（`680/690`）、`manifest.interaction`、`planning.limits`/`plan.limits`，未提供 `--mode` 等切换入口；终态档案拒绝恢复（`598-600`）。
- 默认 guided、`--auto` 才跳过、自动仍需独立 `--auto-confirm-plan`、guided 无通道即失败：`src/cli/index.ts:385`、`386-388`、`397-399`。

### 要点 6：结构化七阶段与屏障；同阶段逻辑并行但受全局+CLI 限流器约束；同阶段看不到他人未完成输出；guided 在独立陈述/质疑补充/争议判定/报告草稿后等待检查点

- 符合。
- 七阶段：独立陈述 `src/core/structured.ts:90-94`；质疑补充 `:97-102`；修订与争议信号 `:105-125`（`revisions` + `findDisputes`）；一次争议收敛 `:128-136`（仅在检测到争议时执行，符合「按策略」）；报告草稿 `src/core/outcome.ts:48-57`；并行审阅 `src/core/outcome.ts:63-74`；最终修订 `src/core/outcome.ts:79-89`。
- 同阶段并行 + 限流器：并行经 `settleAllOrThrow`（`src/core/structured.ts:156-174`），底层 `InvocationRunner` 使用全局 + 每 CLI 限流器（`src/core/execution.ts:138` `scheduler.run(resolved.cli.id, resolved.cli.maxConcurrency, ...)` 嵌套全局信号量）。
- 同阶段看不到他人未完成输出：各阶段 prompt 使用「上一阶段已完成的快照」（`src/core/structured.ts:96,104,127`），独立陈述阶段 prompt 仅含问题、不含任何同伴输出（`:91`）；并发 prompt 闭包捕获的是阶段开始前固定的上下文字符串，不会泄漏同阶段在途输出。
- guided 四检查点：独立陈述后 `:94`、质疑补充后 `:102`、争议判定后 `:125`、报告草稿后 `:143`（经 `onDraft` 回调）；与「独立陈述、质疑补充、争议判定、报告草稿」一致。

### 要点 7：自由讨论——DiscussionController 权威状态；覆盖周期；开放讨论不连发；每 N 回合一窗口；窗口边界收敛；主持按窗口规划、无长期会话；主持调度计入成本但非观点

- 符合。
- 持续运行 DiscussionController 而非模型进程：`src/core/discussion.ts:84`（`DiscussionController` 持有权威状态 `speeches`/`windows`/`converged`）。
- 覆盖周期每人恰好一次：`src/core/discussion.ts:117-127` + `parseCoverage`（`:48-57` 强制 `order` 长度=人数、不重复、全包含）。
- 开放讨论可重复但同一参与者不连发：`parseModeratorPlan` `:71-76` 禁止相邻重复并禁止首位是上一位。
- 每完成与人数相等的回合数结束一个检查窗口：`run` 循环 `:130-142`，未收敛时 `parseModeratorPlan` 强制 `speakers.length === participantIds.length`（`:68-70`），完成即一个窗口并在边界 `atBoundary`。
- 窗口边界收敛评估 + guided 允许动作：`atBoundary` `:197-223`，检查点动作含 continue/guide/end/pause/cancel（`src/cli/index.ts:320`）。
- 主持按窗口规划而非每回合、首版无长期会话：`evaluate` 仅在各窗口边界调用（`:129,141`、`:174-195`），每次均为一次性 CLI 调用，无持久会话。
- 主持调度计入调用/成本但不算发言回合、非参与者观点：`evaluate` 调用 `kind:"moderator"`（`:183-188`），`DiscussionResult.rounds = speeches.length`（仅发言，`:150`）；报告证据明确标注「主持调度不作为参与者观点」（`src/core/discussion.ts:144`）。

### 要点 8：自由讨论结束复用共享报告流水线；报告区分共识/未决争议/假设/风险；不把主持调度当作参与者观点

- 符合。
- 复用共享报告流水线：`src/core/discussion.ts:146` 使用 `OutcomePipeline`（与结构化共用 `src/core/outcome.ts`）。
- 报告强制区分共识/未决争议/假设/风险：`src/core/outcome.ts:19-31`（`validateFinalReport` 校验四类必需章节）。
- 不把主持调度当观点：草稿提示注入 `sharedOriginWarning` 且要求「不得虚构一致意见」（`src/core/outcome.ts:54-55`）；讨论证据已声明主持调度非观点（`src/core/discussion.ts:144`）。

### 要点 9：检查点动作（继续/指导后继续/结束讨论(仅自由)/暂停/取消）；第一次 Ctrl-C 按暂停并终止当前 CLI 子进程；第一份有效响应获胜，过期/重复/ID 不匹配被拒

- 基本符合（存在一项低危偏差：无绝对时间过期机制，仅以 checkpointId 作废实现「过期」）。
- 动作枚举：结构化 `CheckpointDecision` 为 continue/pause/cancel（`src/core/structured.ts:11-14`），"指导后继续"以 `continue + guidance` 表示（CLI 终端/观察页动作含 `guide`，`src/cli/index.ts:272,283,295`）；自由讨论额外含 `end`（`src/core/discussion.ts:10-13`），结构化信箱动作不含 `end`（`src/cli/index.ts:272`），满足「结束讨论仅自由讨论可用」。
- 暂停/取消进入终态、保留档案：controller 抛 `PAUSED`/`CANCELLED`（`src/core/structured.ts:211-212`、`src/core/discussion.ts:219-220`），CLI 将档案置为 paused/cancelled（`src/cli/index.ts:562-565`、`726-728`），不写 report。
- 第一次 Ctrl-C 按暂停并终止当前 CLI 子进程：`src/cli/index.ts:446` 注册 `SIGINT→interrupt.abort()`，runner 在调用前检测 `signal.aborted` 抛 `PAUSED`（`src/core/execution.ts:127`）；适配器在 abort 时终止子进程组（`src/adapters/process.ts:46-60`），满足「终止当前 CLI 子进程」。
- 第一份有效响应获胜：响应经原子 `link` 独占写入（`src/server/mailbox.ts:21-33`），先到者胜，迟到者 `EEXIST` 失败。
- ID 不匹配/重复被拒：`src/server/mailbox.ts:78` 仅当 `response.checkpointId === checkpointId && pending.actions.includes(action)` 才接受；重复 `submit` 因原子 `link` 失败被拒（`:96-103`）。
- 偏差（低）：`过期`仅通过「checkpointId 不匹配即拒绝」实现陈旧/过期响应过滤，缺少基于绝对时间的过期窗口（如请求 `createdAt` 超时校验）。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| D1 | 第 10 节 | 检查点响应无绝对时间过期机制，仅以 checkpointId 匹配作废「过期」响应 | 低 | 历史/陈旧响应仅在 ID 不符时被拒；若极端情况下 ID 被复用存在理论误接受风险（当前每次新建 checkpointId，实际风险极低） | 在 `CheckpointMailbox.wait` 增加基于 `createdAt` 的绝对过期校验，超时响应直接忽略 |
| D2 | 第 10 节 | 结构化 `CheckpointDecision` 未将「指导后继续」建模为独立动作枚举，仅以 `continue + guidance` 表示（功能等价） | 低 | 序列化/审计时无法直接区分「纯继续」与「指导后继续」 | 可在类型层显式增加 `guided-continue` 枚举或在决策记录中标注来源 |
| D3 | 第 7 节 | 争议收敛仅在检测到争议信号时执行（结构化） | 低 | 若策略要求「无论是否有争议都执行一次收敛」，当前会跳过收敛阶段 | 当前实现与「按策略执行一次争议收敛」相容；如需强制收敛，可在 `findDisputes` 为空时仍触发收敛，建议以测试明确策略 |
| D4 | 第 8 节 | 覆盖周期后的首次「检查窗口边界」在开放讨论发言之前触发（windows=0 边界） | 低 | 窗口计数语义（边界先于窗口发言）需在文档/测试中明确，避免与「完成 N 回合结束一个窗口」产生歧义 | 在 `discussion.ts` 注释或测试中明确窗口编号与边界时机 |

## 5. 结论与优先建议

- 结论：当前实现对照第 5/6/7/8/10 节要求，整体为「符合」，仅存在 4 项低危偏差，无「不符合」或「缺失」项。组局前置、交互确认与约束、非交互不降级、共享来源保留、模式/恢复策略、结构化七阶段与屏障、自由讨论 DiscussionController 权威状态与窗口机制、共享报告流水线、检查点动作枚举与 Ctrl-C 暂停终止子进程等核心要求均已落地且证据明确。
- 优先建议：
  1. （低优先）补充检查点响应的绝对时间过期校验（D1），进一步收紧第 10 节「过期被拒」的语义。
  2. （低优先）在类型/审计层显式区分「指导后继续」与「纯继续」（D2）。
  3. （低优先）以单元测试固化结构化收敛策略与自由讨论窗口计数语义（D3、D4），防止后续重构偏离目标架构。
