# 档案与恢复审查报告

## 1. 审查范围与依据

- **目标架构**：`docs/TypeScript目标架构.md` 第13节（透明档案）、第14节（恢复与失败）。
- **当前实现源码**：
  - `src/archive/index.ts`、`src/archive/store.ts`（档案目录、文件写入、原子替换、只追加、锁）
  - `src/core/execution.ts`（逻辑调用、冻结输入、原子提交、重试、恢复）
  - `src/cli/index.ts`（resume 命令、暂停/取消终态、全局原子锁）
  - `src/core/structured.ts`、`src/core/discussion.ts`（并行/逐回合提交、恢复重放）
  - `src/core/context.ts`（共享上下文滚动摘要）、`src/core/types.ts`（manifest 类型）
  - `src/adapters/generic.ts`、`src/adapters/codex.ts`、`src/core/errors.ts`（瞬时/可重试错误分类与脱敏）

> 本报告严格以目标架构文档与源码为准，未读取 `docs/审查/` 下其它子目录及 `docs/TypeScript当前实现架构审查.md`。

## 2. 目标架构要求摘要

- **第13节 透明档案**：每次审议独立目录，含 `manifest.json`（身份/模式/参与者/方案/schema_version）、`state.json`（原子替换可恢复状态）、`events.jsonl`（只追加生命周期）、`transcript.jsonl`（只追加权威发言）、`diagnostics.jsonl`（只追加脱敏调用诊断）、`report.md`（最终共同成果）；首版不引入数据库。
- **第14节 恢复与失败**：以逻辑调用为最小边界，启动前冻结输入+逻辑调用ID+尝试信息，成功后原子提交；结构化保留已完成结果只重跑未完成调用，自由讨论逐回合提交；瞬时与 schema 错误各重试一次，再失败则暂停为可恢复状态；不自动移除参与者/降低最低人数/替换 CLI 或预设；CLI 已完成未提交崩溃可重复计费但权威记录只提交一次；每次尝试独立尝试 ID 并在脱敏诊断中关联同一逻辑调用；恢复不重新组局/不切换模式/交互策略/角色/预设/预算；同一 `MAD_HOME` 用原子全局锁限制单个活动审议。

## 3. 当前实现对照

### 要点 1：第13节档案目录与六文件结构
- **判定：符合**
- 证据：
  - `store.ts:57-76` `create()` 创建独立目录（`mkdir` 模式 `0o700`），写入 `manifest.json`、`state.json`，并创建 `events.jsonl`/`transcript.jsonl`/`diagnostics.jsonl` 三个只追加文件。
  - `store.ts:216-219` `writeManifest` 写入 `schema_version: 1` 及清单内容；`types.ts:67-87` `DeliberationManifest` 含 `schemaVersion`/`id`(身份)/`mode`/`plan`(参与者与方案)/`registrySnapshot` 等。
  - `store.ts:95-98` `writeState` 经 `atomicJson` 原子替换 `state.json`；`store.ts:248-254` `writeReport` 原子写入 `report.md`。
  - `store.ts:234-236` `appendDiagnostic`、`store.ts:221-232` `ensureTranscript`、`store.ts:238-246` `appendEvent` 均为只追加。
  - 首版未引入数据库，全部文件化（源码中无 SQLite/DB 依赖）。

### 要点 2：以逻辑调用为最小边界（冻结输入 + 尝试信息；成功原子提交）
- **判定：符合**
- 证据：
  - `execution.ts:108-116` 在调度适配器前构造 `FrozenInvocation`（含 `logicalCallId`/`kind`/`agentId`/`prompt`/`invocation` 冻结输入）并调用 `freezeInvocation` 持久化。
  - `execution.ts:128-136` 在调用前 `beginAttempt` 持久化尝试计数，并 `appendEvent("invocation.attempt_started", { attemptId, logicalCallId, attemptNumber, ... })` 记录尝试信息。
  - `execution.ts:156-175` 成功返回后 `commitInvocation` 原子提交规范化输出 `InvocationResult`（含 `text`/`completedAt`/`durationMs`）。
  - `store.ts:125-145` `commitInvocation` 通过状态变更保证同一 `logicalCallId` 只提交一次（幂等）。

### 要点 3：结构化保留已完成结果只重跑未完成；自由讨论逐回合提交
- **判定：符合**
- 证据：
  - `execution.ts:117-122` 每次调用前读取 `state.completedInvocations`，若已完成直接返回缓存结果（`newlyCommitted:false`），即重放已提交结果。
  - `structured.ts:85-154` 控制器 `run()` 从首个阶段重放，每个 `runner.run` 对已完成参与者走缓存、仅对未完成/失败调用实际重跑；`structured.ts:156-174` `parallel()` 对阶段内所有参与者并发，已完成者命中缓存。
  - `discussion.ts:158-172` `speak()` 每回合独立 `runner.run`（冻结+提交），满足“逐回合提交”；`discussion.ts:107-156` 恢复时重放各发言调用，已完成者走缓存。
  - 注：恢复以“逻辑调用缓存”实现阶段级续跑，未对阶段本身设独立检查点，但已完成结果语义上被完整保留。

### 要点 4：瞬时错误与 schema 输出错误各自动重试一次；再失败则暂停为可恢复状态
- **判定：符合**
- 证据：
  - `execution.ts:125` `for (let retry = 0; retry < 2; retry += 1)` 固定两次尝试（自动重试一次）。
  - `execution.ts:147-155` schema 解析失败抛出 `RetryableMadError`。
  - `adapters/generic.ts:74` 与 `adapters/codex.ts:76` 通过 `isLikelyTransientFailure(detail) ? RetryableMadError : MadError` 将瞬时失败映射为可重试错误；`adapters/process.ts:83` 超时亦为 `RetryableMadError`。
  - `execution.ts:176-197` 捕获到 `RetryableMadError`（瞬时或 schema）时记诊断并可重试；`PAUSED`/`CANCELLED` 立即抛出，非可重试 `MadError` 立即抛出。
  - 再次失败：`execution.ts:199` 抛 `EXECUTION 连续两次失败`，向上由 `structured.ts:149-153` / `discussion.ts:151-155` 捕获并 `setStatus("paused")`（除 `CANCELLED` 外），即暂停为可恢复状态。

### 要点 5：不自动移除参与者、降低最低人数、替换 CLI 或调用预设
- **判定：符合**
- 证据：
  - `execution.ts` 中 `freezeInvocation`/`commitInvocation` 仅读写调用结果，不触碰参与者列表。
  - `cli/index.ts:575-735` `resume()` 从 `manifest.plan`/`manifest.planning` 复原方案，参与者、CLI、预设均来自档案快照（`cli/index.ts:162-195` `registryFromManifest`），未实现任何自动移除/降额/替换逻辑。
  - 调用解析严格使用方案内 `invocation`（cli/preset），无回退到其它 CLI/预设（`execution.ts:100` `resolveInvocation`，失败时抛错而非替换）。

### 要点 6：CLI 已完成但未提交时崩溃，可重复计费但权威记录只提交一次
- **判定：符合**
- 证据：
  - `execution.ts:156-175` 适配器返回后才提交；若在“返回后—`commitInvocation` 前”崩溃，重启后该 `logicalCallId` 不在 `completedInvocations`，将重跑（可能产生重复计费）。
  - `store.ts:125-145` `commitInvocation` 幂等：已存在则跳过，不产生重复权威记录。
  - `store.ts:221-232` `ensureTranscript` 按 `logicalCallId` 去重，权威 `transcript.jsonl` 只写入一次。

### 要点 7：每次尝试独立尝试 ID，并在脱敏诊断中关联同一逻辑调用
- **判定：符合**
- 证据：
  - `execution.ts:129` `const attemptId = randomUUID()` 在每次重试循环内生成独立尝试 ID。
  - `execution.ts:163-171`（成功）与 `execution.ts:187-194`（失败）均将 `{ attemptId, logicalCallId, attemptNumber, ... }` 写入 `diagnostics.jsonl`，关联同一逻辑调用。
  - 脱敏：`adapters/generic.ts:81` 诊断仅含 `executable`/`exitCode`/`stderr`，且 `stderr` 经 `redact()`（`:84-93`）脱敏；`errors.ts:23` `isLikelyTransientFailure` 仅用于分类，不影响脱敏。

### 要点 8：恢复不重新组局、不切换模式/交互策略/角色/预设/预算
- **判定：符合（含一处边缘备注）**
- 证据：
  - `cli/index.ts:595-598` 读取 `manifest` 复原 `registry`、`state`，`manifest.mode`/`manifest.interaction`/`manifest.plan` 直接复用，无切换逻辑。
  - `cli/index.ts:613-678` 若 `plan` 已存在则仅对计划重新预检；缺失 `plan` 时仅在“组局从未确认”（`status==="planning"`）场景下用**同一** `organizer` 与 `planning.limits` 重建方案（`:629-667`），未切换模式/角色/预设/预算。
  - `structured.ts:85-88` / `discussion.ts:107-110` 恢复以相同 `plan` 进入，并将已存档的 `guidance` 重新载入（`:87`/`:109`），交互策略与预算不变。
  - 边缘备注：当组局从未确认而恢复时，重新 `propose()` 可能生成与中断前不同的参与者组合（因组局器非确定性），但所用 CLI/预设/组织者/资源上限严格一致，未“切换”既有参数。

### 要点 9：同一 MAD_HOME 使用原子全局锁限制单个活动审议
- **判定：符合**
- 证据：
  - `store.ts:266-339` `ActiveDeliberationLock` 以 `open(path,"wx",0o600)` 原子创建 `runtime/active.lock`，保证同一 `MAD_HOME` 仅一个活动审议。
  - `store.ts:307-338` `reclaimStale()` 通过 `process.kill(pid,0)` 检测并回收僵尸锁（崩溃后恢复可用）。
  - `cli/index.ts:442-443`（deliberate）与 `:589-590`（resume）均在开启档案前 `lock.acquire(id)`，`finally` 中 `lock.release()`（`:571`/`:733`）。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| R1 | 14（要点8） | 组局从未确认（status=planning）时 `resume` 会重新调用组局器 `propose()`，可能生成与中断前不同的参与者组合 | 低 | 恢复后的方案构成可能偏离用户最初所见，但 CLI/预设/预算/模式均未切换 | 可考虑在 `planning.candidatePlan` 已存在时直接复用，仅在确实缺失时重跑组局 |
| R2 | 13/14（要点3） | 结构化与自由讨论恢复均从阶段 0 重放，依赖“逻辑调用缓存”续跑，未对已完成阶段设独立跳过标志 | 低 | 重放会重新计算上游共享上下文快照（含可能触发滚动摘要），存在额外本地计算开销，但结果确定且正确 | 可优化为按阶段记录已完成标记以跳过整段重放；当前正确性无碍 |
| R3 | 13（要点1 脱敏） | `diagnostics.jsonl` 的脱敏完全依赖各适配器 `redact()` 实现 | 中 | 若新增适配器未实现/未调用 `redact()`，敏感信息（密钥、Token）可能被写入档案 | 在 `appendDiagnostic`（`store.ts:234`）入口统一做一层兜底脱敏，降低适配器缺漏风险 |
| R4 | 14（要点9） | 全局锁释放依赖运行进程正常退出（finally），`SIGKILL`/断电场景靠 `reclaimStale()` 的 pid 存活探测 | 低 | 极少数情况下 pid 复用可能导致僵尸锁误判为存活而拒绝新审议，或反之 | 可补充写入启动时间戳/心跳，但当前实现已覆盖绝大多数崩溃恢复场景 |
| R5 | 14（要点4） | 重试策略对“可重试错误”一律重试一次，未区分瞬时与 schema 是否应独立计数（如瞬时可多次、schema 仅一次） | 低 | 与目标“各自动重试一次”一致，但 schema 错误重试一次价值有限（输入未变，重跑大概率仍失败） | 当前符合规格，可后续按需将 schema 错误改为不重试以省开销 |

## 5. 结论与优先建议

**结论**：当前 TypeScript 实现与 `docs/TypeScript目标架构.md` 第13、14节高度一致。九个审查要点中，**要点1–7、9 均判定为“符合”**，要点8 为“符合（含边缘备注）”。档案六文件结构、原子替换 `state.json`、只追加 `events/transcript/diagnostics`、以逻辑调用为最小边界的冻结/提交、瞬时与 schema 错误各重试一次、幂等提交与去重、独立尝试 ID 关联脱敏诊断、原子全局锁等关键要求均有明确源码证据支撑。

**优先建议（按严重度）**：
1. **（中）R3**：在 `ArchiveStore.appendDiagnostic` 增加统一兜底脱敏，避免未来适配器缺漏导致敏感信息落盘。
2. **（低）R1**：组局未确认场景下 `resume` 优先复用 `planning.candidatePlan`，减少非确定性重组局带来的方案漂移。
3. **（低）R2/R4/R5**：属效率与极端边界优化，正确性不受影响，可在后续迭代处理。

整体上，档案与恢复模块满足目标架构的透明、可审计、可恢复设计意图，无高严重度偏差。
