# 工作目录与安全审查报告

## 1. 审查范围与依据

- 审查对象：当前 TypeScript 实现中与「工作目录与安全」相关的代码。
- 目标依据：`docs/TypeScript目标架构.md`

  - 第 1 节约束 #15（项目审议只读访问显式指定原始工作目录，不创建快照，不隐式使用当前目录）、约束 #18（本地服务只绑定 `127.0.0.1`）。

## 2. 目标架构要求摘要

第 15 节及第 1 节约束 #15/#18 的核心要求：

1. 纯文本审议不提供工作目录；项目审议必须显式传入 `--workspace <path>`，参数即读取授权；不提供 `--direct-workspace`，不额外确认。
2. 组局器与所有参与者从组局阶段起共同直接只读访问规范化后的原始目录。
3. 应用向 stderr 输出风险提示，并把绝对路径与读取模式写入档案。
4. CLI 适配器必须启用并验证对应运行时的只读/计划/禁用写工具模式；不满足最低只读能力的适配器或预设不能用于项目审议。
5. 直接只读不限制 CLI 可见文件；应用不通过快照排除秘密/依赖/构建产物。
6. 本地服务只绑定 `127.0.0.1`（约束 #18）。

## 3. 当前实现对照

### 3.1 要求 1：纯文本不提供工作目录；项目审议显式 `--workspace`，无 `--direct-workspace`，无额外确认

- 符合。
- 证据：
  - `src/cli/index.ts:364` 仅声明 `workspace: { type: "string" }` 选项，全仓 Grep `direct-workspace`/`directWorkspace` 无任何命中（`src/cli/index.ts:357-376` 的 `options` 列表中无该参数）。
  - `src/cli/index.ts:408-413`：传入 `--workspace` 时通过 `realpath` 规范化并将其设为 `cwd`；未传入时回退到 `MAD_HOME/runtime/scratch/<id>` 私有临时目录（`src/cli/index.ts:405-406`），即纯文本审议不提供工作目录，也不隐式使用进程当前目录（契合约束 #15）。
  - 代码中不存在针对工作目录的额外确认步骤；`--workspace` 的传入即被视为授权（组局方案确认属于方案层，非目录授权层）。

### 3.2 要求 2：组局器与所有参与者从组局阶段起共同直接只读访问规范化后的原始目录

- 符合。
- 证据：
  - 规范化：`src/cli/index.ts:409` `cwd = await realpath(parsed.values.workspace)`。
  - 组局器访问：组局阶段 `OrganizerService.propose` 将 `cwd` 透传给组局器调用（`src/core/planning.ts:131-167`，`request.cwd` 在 `propose` 调用处由 `src/cli/index.ts:485` 传入），`generatorAdapter.invoke({ cwd: request.cwd, ... })`（`src/core/planning.ts:162-167`）。
  - 参与者访问：`InvocationRunner` 以同一 `cwd` 构造（`src/cli/index.ts:474`），并在每次调用透传给适配器（`src/core/execution.ts:138-144`，`adapter.invoke({ cwd: this.cwd, ... })`）。
  - 恢复路径同样复用规范化原始目录（`src/cli/index.ts:606-611`）。

### 3.3 要求 3：向 stderr 输出风险提示，并把绝对路径与读取模式写入档案

- 符合。
- 证据：
  - stderr 风险提示：`src/cli/index.ts:412` `警告：所有审议 CLI 将直接只读访问完整工作目录：${cwd}`；恢复时 `src/cli/index.ts:611` 同样输出警告。
  - 写档案（绝对路径 + 读取模式）：`src/cli/index.ts:407-411` 构造 `workspace = { path: cwd, mode: "direct-read-only" }`，并在 `src/cli/index.ts:467` 通过 `...(workspace ? { workspace } : {})` 写入 `manifest.json`。`cwd` 来自 `realpath`，为绝对路径。
  - 额外档案事件：`src/cli/index.ts:513-517` 将「参与 CLI 已获完整目录只读授权：<path>」作为 `warning` 事件追加到 `events.jsonl`；恢复路径 `src/cli/index.ts:699-703` 同口径。

### 3.4 要求 4：适配器启用并验证只读/计划/禁用写工具模式；不满足最低只读能力的适配器或预设不能用于项目审议

- 部分符合。
- 启用情况（符合）：
  - Codex：`src/adapters/codex.ts:54-55` 固定 `--sandbox read-only --ephemeral`，明确只读沙箱。
  - Claude：`src/adapters/generic.ts:21` `--permission-mode plan` 且 `--tools Read,Glob,Grep,WebSearch,WebFetch`，工具被限制为只读集合。
  - Grok：`src/adapters/generic.ts:23` `--permission-mode plan --no-subagents --no-memory`。
  - CodeBuddy：`src/adapters/generic.ts:25` `--permission-mode plan --tools Read,Glob,Grep`。
  - Pi：`src/adapters/generic.ts:24` `--no-approve` 且 `--tools read,grep,find,ls`，排除写工具。
  - agy：`src/adapters/generic.ts:26` `--mode plan --sandbox`。
  - 配置层封锁：预设 schema 仅允许 `reasoning_effort`/`effort`/`thinking`，无任何可开启写工具或关闭沙箱的选项（`src/adapters/config.ts:78-117`），符合第 4 节「禁止配置覆盖只读、审批、工具限制」约束。
- 能力门禁（符合）：
  - `supportsProjectReadOnly` 标志在 `src/adapters/types.ts:27` 声明；`reasonix` 显式置为 `false`（`src/adapters/generic.ts:35`）。
  - 项目模式下强制校验：组局器 `src/core/planning.ts:135-137` 拒绝不支持只读的组局器；参与者 `src/core/planning.ts:193-195` 在 `preflightPlan` 中拒绝不支持只读的组合。即 `reasonix` 及其预设均无法进入项目审议。
- 验证不足（偏差）：
  - 各适配器的预检 `check()` 仅发送「只回复 READY，不要执行任何工具」（如 `src/adapters/codex.ts:28-48`、`src/adapters/generic.ts:51-60`），断言响应为 `READY`，属于冒烟测试；并未主动验证运行时只读/计划沙箱确实阻断了写操作。也就是说，适配器的「验证」停留在「CLI 可应答」，而非「写能力被实际禁用」的断言级验证。
  - 因此「启用」充分、「能力门禁」充分，但「验证只读模式真正生效」这一子要求仅部分满足。

### 3.5 要求 5：直接只读不限制 CLI 可见文件；不通过快照排除秘密/依赖/构建产物

- 符合。
- 证据：
  - 项目模式下将规范化后的原始目录作为 `cwd` 直接透传给所有 CLI（`src/cli/index.ts:409`、`src/core/execution.ts:141`），无任何文件清单、`.gitignore`、依赖目录或构建产物的排除/掩码逻辑。
  - 全仓不存在「创建工作目录材料快照」或「按规则裁剪可见文件」的实现；`src/core/context.ts` 的 `snapshot()` 是上下文滚动摘要机制（针对审议上下文预算），与工作目录文件快照无关。
  - 用户显式传入的目录被当作完整读取授权（`src/cli/index.ts:514` 警告文案「完整目录只读授权」），符合「视为完整读取授权」的语义。

### 3.6 要求 6：本地服务只绑定 127.0.0.1（约束 #18）

- 符合。
- 证据：
  - `src/server/observer.ts:190` `server.listen(port, "127.0.0.1", resolve)` 硬编码绑定回环地址。
  - 回调 URL 同样使用回环地址 `src/server/observer.ts:202` `http://127.0.0.1:${address.port}/#token=...`；文件信箱连接也指向 `127.0.0.1`（`src/server/mailbox.ts:116`）。
  - 启动日志明确提示「服务仅监听 127.0.0.1」（`src/cli/index.ts:742`）。
  - 注意：`src/server/index.ts:1` 定义了 `SERVER_HOST = "127.0.0.1"` 常量，但 `observer.ts` 实际绑定处使用字符串字面量 `"127.0.0.1"` 而非该常量（Grep `SERVER_HOST` 仅命中定义处，未被引用）。功能上仍满足约束，但存在常量未使用的维护隐患（见偏差 D2）。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| D1 | 15 / 要求4 | 适配器预检 `check()` 仅做 READY 冒烟测试，未主动断言运行时只读/计划沙箱确实阻断写操作；「验证只读模式生效」这一子要求未达断言级。 | 中 | 若某运行时升级后改变默认权限语义，应用无法在组局前发现「只读未真正生效」，存在越权写入原始工作目录的潜在可能。 | 在 `check()` 中增加正向验证：在临时子目录尝试一次写操作并断言被拒绝（或断言沙箱/plan 模式激活信号），失败即阻断项目审议。 |
| D2 | 18 / 要求6 | `SERVER_HOST` 常量（`src/server/index.ts:1`）未被 `observer.ts` 实际引用，绑定处为字面量 `"127.0.0.1"`。 | 低 | 当前仍绑定 127.0.0.1，功能正确；但若后续误改常量而不知绑定未联动，可能产生配置漂移风险。 | 让 `observer.ts:190` 改为 `server.listen(port, SERVER_HOST, resolve)`，统一单一真相来源。 |
| D3 | 15 / 要求4 | `reasonix` 适配器未启用任何只读/计划模式（`src/adapters/generic.ts:22` 仅 `run --dir . --max-steps 3`），其 `supportsProjectReadOnly=false` 仅靠门禁阻止进入项目审议；若该门禁未来被绕过或 `reasonix` 被误设为组局器默认，将以写权限运行。 | 低 | 现有门禁有效，风险被控制；但属于「能力缺失靠门禁兜底」而非「默认安全」。 | 保持并补充测试，确保 `supportsProjectReadOnly=false` 的适配器在任何项目模式路径下都被拒绝；可在配置校验阶段对预设标注只读能力。 |

## 5. 结论与优先建议

整体结论：当前实现在第 15 节与约束 #15/#18 上**基本符合**，6 条要求中 5 条符合、1 条部分符合（要求 4 的「验证」子项）。

- 要求 1（显式 `--workspace`、无 `--direct-workspace`、无额外确认）、要求 2（规范化原始目录被组局器与参与者共同直接只读访问）、要求 3（stderr 提示 + 绝对路径与读取模式入库）、要求 5（不创建快照、不裁剪可见文件）、要求 6（绑定 127.0.0.1）均符合目标架构。
- 要求 4 的「启用只读/计划/禁用写工具模式」与「能力门禁」均符合，但「验证只读模式真正生效」仅做到 CLI 可应答级，未达到断言级验证（偏差 D1，中）。

优先建议（按重要度）：

1. 优先补强 D1：在适配器运行时预检中增加「写操作被拒绝」或「沙箱/plan 模式激活」的断言级验证，使只读能力可被实际证明，而非仅证明 CLI 可应答。
2. 顺手修复 D2：将 `observer.ts:190` 改为引用 `SERVER_HOST` 常量，消除绑定地址与常量不一致的配置漂移隐患。
3. 保留 D3 的回归测试：确保任何 `supportsProjectReadOnly=false` 的适配器/预设绝不会进入项目审议路径。
