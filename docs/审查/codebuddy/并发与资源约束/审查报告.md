# 并发与资源约束审查报告

## 1. 审查范围与依据

本审查对照 `docs/TypeScript目标架构.md` 第 9 节「资源、并发与上下文」，对当前 TypeScript 实现做并发与资源约束审查。

依据与阅读对象：

- 目标架构文档：`docs/TypeScript目标架构.md`（第 9 节，并参考第 4/5/6/7/8 节涉及预算与并发处）。
- 限制默认值与安全最大值、用户覆盖：`src/core/limits.ts`。
- 调用执行、限流器使用、并发控制：`src/core/execution.ts`。
- 上下文预算估算、滚动摘要：`src/core/tokens.ts`、`src/core/context.ts`。
- 预算/并发在档案中的写入与恢复：`src/archive/store.ts`、`src/cli/index.ts`、`src/core/planning.ts`。
- CLI 配置与类型声明（max_concurrency 默认、上下文预算声明）：`src/adapters/config.ts`、`src/adapters/types.ts`。

审查严格遵守约束：未读取 `docs/TypeScript当前实现架构审查.md` 及 `docs/审查/` 下其它子目录报告，一切结论仅以目标架构文档与源码为准。

## 2. 目标架构要求摘要

第 9 节核心要求：

1. 参与者数量、自由讨论检查窗口、总调用次数、超时、上下文预算同时受三层约束：内置保守默认值、用户按次覆盖值、普通命令不能突破的安全最大值；组局器可见但不能提高限制。
2. 并发由全局限流器 + CLI 配置级限流器共同控制；CLI 配置首版默认并发为 1；同一 CLI 下的全部调用预设和审议 Agent 共享限流器。最终预算与并发设置写入档案，恢复时不改变。
3. 每个模型调用预设声明上下文预算；Controller 在调用前估算实际输入；任一后续调用预算不足时，由报告 Agent 生成统一滚动摘要。
4. 所有参与者、主持 Agent、报告 Agent 使用同一摘要 + 摘要后的最近发言，不按模型生成不同版本；摘要生成计入预算且可独立恢复，完整原始记录始终保留。

## 3. 当前实现对照

### 第 9 节第 1 条：三层约束 + 组局器可见但不能提高

- 符合。内置保守默认值：`src/core/limits.ts:4-11`（`DEFAULT_LIMITS`，含 maxParticipants 4、maxCalls 60、maxDiscussionWindows 6、timeoutSeconds 300、contextBudget 128000、globalConcurrency 6）。
- 符合。安全最大值：`src/core/limits.ts:13-20`（`SAFE_MAX_LIMITS`，上下文预算上限 1_000_000、globalConcurrency 上限 16 等）。
- 符合。用户按次覆盖 + 普通命令不能突破安全最大值：`src/core/limits.ts:22-31`（`resolveLimits` 以用户覆盖值合并默认值，并对每个字段校验 `1 ≤ value ≤ SAFE_MAX`，超出即抛错）；覆盖入口见 `src/cli/index.ts:432-439`。
- 符合。上下文预算受三层约束：计划层级 `plan.limits.contextBudget` 由 `resolveLimits` 钳制于安全最大值；`src/core/planning.ts:90` 进一步 `effectiveLimits.contextBudget = Math.min(limits.contextBudget, ...预设预算)`，只能调低、不能调高。
- 符合。组局器可见限制：`src/core/planning.ts:223` 在组局提示中注入 `资源限制：${JSON.stringify(request.limits)}`。
- 符合。组局器不能提高限制：`src/core/planning.ts:63-64` 校验参与者数不超过 `limits.maxParticipants`；`src/core/planning.ts:87-103` 的 `effectiveLimits` 完全由 `options.limits` 派生并取 min，模型输出无法抬高任何限制；自由讨论窗口上限在 `src/core/discussion.ts:136` 校验，总调用数上限在 `src/core/execution.ts:128`（`beginAttempt`）强制。
- 偏差（见风险 R1）：预设级 `context_budget` 在配置解析时未受 `SAFE_MAX_LIMITS.contextBudget` 约束（`src/adapters/config.ts:114` 仅要求为正整数），而 `src/core/execution.ts:102` 的逐调用输入估算直接对照「未钳制」的预设预算。普通命令虽不能突破 `plan.limits.contextBudget`，但单个预置一个超大预算可让单次调用输入检查越过安全最大值语义。该值属于可信 CLI 配置边界，影响为低到中。

**结论：符合（含一处低风险偏差 R1）。**

### 第 9 节第 2 条：并发控制、默认并发 1、CLI 共享限流器、写入档案并恢复

- 符合。全局限流器 + CLI 配置级限流器：`src/core/execution.ts:40-56`（`InvocationScheduler` 持有 `global` 信号量与 `perCli` 按 CLI ID 的信号量，二者嵌套生效）。
- 符合。同一 CLI 下全部调用预设与审议 Agent 共享限流器：`src/core/execution.ts:48-54` 以 `resolved.cli.id` 作为限流器键，与预设/角色无关；调度入口在 `src/core/execution.ts:138`。
- 符合。CLI 配置首版默认并发为 1：`src/adapters/config.ts:141`（`max_concurrency` 缺省 1），模板生成同样默认 1（`src/adapters/config.ts:217`）。
- 符合。全局并发默认值接入：控制器与组局器均以 `plan.limits.globalConcurrency ?? 6` 构造 `InvocationScheduler`（`src/core/structured.ts:78`、`src/core/discussion.ts:100`、`src/core/planning.ts:199`；执行器构造见 `src/cli/index.ts:474`、`:622`）。
- 符合。最终预算与并发写入档案：`plan`（含 `limits`，即上下文预算与 globalConcurrency）写入 `manifest.plan`（`src/cli/index.ts:668-674`）；每 CLI 的 `maxConcurrency` 写入 `registrySnapshot`（`src/cli/index.ts:141,154`，并经一致性校验恢复 `:165-187`）。
- 符合。恢复时不改变：`src/cli/index.ts:613` 直接取用 `manifest.plan`，`:676-677` 仅对已有 plan 做预检、不重新求解限制；恢复路径沿用 `plan.limits.globalConcurrency` 与 `registrySnapshot` 中的 `maxConcurrency`，未见重新解析或放宽。

**结论：符合。**

### 第 9 节第 3 条：每层预设声明上下文预算；调用前估算输入；预算不足时报告 Agent 生成统一滚动摘要

- 符合。每个模型调用预设声明上下文预算：`src/adapters/types.ts:24`、`src/adapters/config.ts:114`（`contextBudget` 为必填正整数）。
- 符合。Controller 在调用前估算实际输入：`src/core/execution.ts:101`（`estimateTokens(call.prompt)`），`:102-107` 在输入超出预设预算时抛错。
- 部分符合。预算不足时由报告 Agent 生成统一滚动摘要：统一滚动摘要由 `SharedContextManager.snapshot` 负责，并确实由报告 Agent 调用（`src/core/context.ts:51-91`，`reportAgent` 身份、kind 为 `summary`）；控制器在每阶段/回合前主动调用 `snapshot`（`src/core/structured.ts:96,104,127,138`；`src/core/discussion.ts:144,160,182`），当累计上下文超过约 45% 预算时压缩为统一摘要（`src/core/context.ts:49-50,56-87`）。
- 偏差（见风险 R2）：摘要触发是「控制器主动调用 `snapshot`」而非「`InvocationRunner` 在检测到预算不足时自动触发」。若某调用路径忘记调用 `snapshot`，单次超预算输入会直接抛错（`src/core/execution.ts:102-107`）而非回退到滚动摘要，与「任一后续调用预算不足时由报告 Agent 生成统一滚动摘要」的自动兜底语义存在落差。
- 偏差（见风险 R3）：`src/core/execution.ts:102` 的逐调用输入估算对照的是「预设自身预算」而非「共享上下文预算（plan 级 / SharedContextManager）」，两处预算口径不统一，可能造成「共享预算已紧张但单次检查仍放行」或反之的割裂。

**结论：部分符合（双层预算口径不一致，且摘要兜底非自动触发）。**

### 第 9 节第 4 条：同一摘要共享、计入预算且可恢复、原始记录保留

- 符合。所有参与者、主持 Agent、报告 Agent 使用同一摘要 + 摘要后的最近发言：`src/core/context.ts:15-99` 维护单一 `summary` 与 `entries`，`renderCurrent` 返回「统一滚动摘要 + 摘要后的最近权威记录」，所有 Agent 经由同一 `snapshot()` 结果获取，不按模型分版本。
- 符合。不按模型生成不同版本：摘要仅由报告 Agent 一个身份生成（`src/core/context.ts:51-52,70-75`），单一实例。
- 符合。摘要生成计入预算：摘要调用本身走 `runner.run`（`src/core/context.ts:70`），其提示 token 计入共享预算估算（`src/core/context.ts:61-62` 的 `availableTokens`），并作为逻辑调用计入总调用数（`beginAttempt`）。
- 符合。摘要生成可独立恢复：摘要是 kind=`summary` 的逻辑调用，经 `freezeInvocation`/`commitInvocation` 持久化（`src/core/execution.ts:108-116,162`），恢复时按 `completedInvocations` 复用（`src/core/execution.ts:119-122`）。
- 符合（含备注 R4）。完整原始记录始终保留：原始发言写入 `transcript.jsonl`（`src/core/execution.ts:120,173,179`），`SharedContextManager` 仅以 `summarizedEntries` 偏移读取、不删除条目（`src/core/context.ts:93-99`）。备注：`SharedContextManager` 的运行时状态（entries/summary/summarizedEntries）未持久化到 `state.json`，恢复时需由 transcript 重建，摘要调用虽可独立恢复，但管理器偏移状态可能重算；当前可接受，但对「可独立恢复」的持久化保证偏弱。

**结论：符合（含一处低风险备注 R4）。**

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| R1 | 第 9 节第 1 条 | 预设级 `context_budget` 解析未受 `SAFE_MAX_LIMITS.contextBudget`（1_000_000）约束；`execution.ts:102` 的逐调用输入检查对照未钳制的预设预算，单个超大预设可越过安全最大值语义。 | 中 | 可信配置若误填超大预算，单次调用输入校验失去安全上限保护，可能突破「普通命令不能突破的安全最大值」。 | 在 `parsePreset` 或 `resolveLimits` 处对预设 `contextBudget` 施加与 `SAFE_MAX` 一致的硬性上界校验。 |
| R2 | 第 9 节第 3 条 | 统一滚动摘要由控制器主动调用 `snapshot` 触发，非 `InvocationRunner` 在预算不足时自动兜底；遗漏调用时超预算输入直接抛错而非生成摘要。 | 中 | 与「任一后续调用预算不足时由报告 Agent 生成统一滚动摘要」的自动兜底语义不符，存在失败路径不一致。 | 在 `InvocationRunner.run` 内当估算输入超共享上下文预算时，自动请求统一滚动摘要并重试，而非仅抛错。 |
| R3 | 第 9 节第 3 条 | 逐调用输入估算（`execution.ts:101-107`）对照预设预算，与共享上下文预算（`SharedContextManager`）口径不统一。 | 低 | 两层预算可能互相脱节，摘要触发与单次校验结论可能矛盾。 | 统一以共享上下文预算作为调用前估算的权威阈值，预设预算仅作为预设声明下限参考。 |
| R4 | 第 9 节第 4 条 | `SharedContextManager` 运行时状态（entries/summary/summarizedEntries）未写入 `state.json`，恢复依赖 transcript 重建，偏移状态可能重算。 | 低 | 「摘要生成可独立恢复」的持久化保证偏弱，极端恢复路径下可能重复生成摘要（已计费的摘要调用理论上可复用，但偏移丢失会重跑）。 | 将 `summarizedEntries`/当前 `summary` 摘要指针随状态原子持久化，恢复时直接复用。 |

## 5. 结论与优先建议

整体结论：当前实现已基本落实第 9 节资源、并发与上下文要求——三层限制（保守默认 / 用户覆盖 / 安全最大值）在 `limits.ts` 与 `planning.ts` 中闭环，组局器可见但代码侧无法抬高限制；并发由全局 + 按 CLI 两级限流器控制、默认并发 1、同 CLI 共享限流器；预算与并发写入档案并在恢复时保持不变；每层预设声明上下文预算、调用前估算输入、统一滚动摘要由报告 Agent 生成且全角色共享、摘要计入预算并可独立恢复、原始记录保留。

主要缺口集中在「预算口径统一」与「摘要兜底自动化」：

1. 优先修复 R1：为预设 `context_budget` 增加安全最大值硬校验，闭合「安全最大值不可突破」的唯一缺口。
2. 其次处理 R2/R3：将滚动摘要触发下沉到 `InvocationRunner`，并以共享上下文预算为统一阈值，使「预算不足自动摘要」成为默认兜底而非控制器自觉行为。
3. 最后完善 R4：持久化上下文管理器偏移状态，强化摘要的可独立恢复保证。
