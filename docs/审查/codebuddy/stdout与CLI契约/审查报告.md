# stdout 与 CLI 契约审查报告

## 1. 审查范围与依据

- 审查对象：当前 TypeScript 实现的 stdout / stderr 输出契约与退出码设计。
- 目标依据：`docs/TypeScript目标架构.md` 第 16 节（stdout、stderr 与退出码），并参考第 6 节（CLI 与交互模式）。
- 审查源码：
  - `src/cli/index.ts`（命令解析、stdout/stderr 输出、退出码、`--format json`）
  - `src/core/errors.ts`（错误类型与退出码定义）
  - `src/core/outcome.ts`（最终成果输出与 JSON 序列化；本文件不直接负责 stdout 契约，仅作为报告生成来源核对）
- 审查方式：逐条对照第 16 节七项要求，给出 符合 / 部分符合 / 不符合 / 缺失 判定，并附 `文件:行号` 证据。
- 不读取 `docs/TypeScript当前实现架构审查.md` 及 `docs/审查/` 下其它子目录报告，结论仅以目标架构文档与源码为准。

## 2. 目标架构要求摘要

第 16 节共七条契约（原文）：

1. 默认成功时，stdout 只包含最终 Markdown 报告。
2. `--format json` 时，stdout 只包含一个完整 JSON 对象。
3.  progress、警告、Token 地址和档案路径进入 stderr。
4. 非 TTY 输出不含 ANSI 控制字符。
5. 失败时不向 stdout 写半截 JSON。
6. 暂停、取消、配置错误、预检失败、执行失败使用不同退出码。
7. JSON 至少包含：审议 ID、状态、模式、报告、参与者、预算使用、警告和档案路径。

## 3. 当前实现对照

### 3.1 默认成功时 stdout 只包含最终 Markdown 报告 —— 符合（deliberate/resume），但 config 子命令存在例外

- 证据：
  - `src/cli/index.ts:367`：`format` 默认值为 `"markdown"`。
  - `src/cli/index.ts:555-557`（deliberate 成功分支）：`else` 分支执行 `process.stdout.write(\`${result.report}\n\`)`，stdout 仅含最终报告。
  - `src/cli/index.ts:721`（resume 成功分支）：同上，`process.stdout.write(\`${result.report}\n\`)`。
- 例外（低严重度）：`mad config validate` / `mad config check` 成功时向 stdout 写入状态摘要而非报告，见 `src/cli/index.ts:121`（`process.stdout.write("配置有效，…预检通过。\n")`）、`src/cli/index.ts:125`。严格按第 16 节字面，这些成功响应也落在 stdout 契约范围内，不属于"最终 Markdown 报告"。考虑到它们是命令自身的状态输出，建议明确纳入契约豁免或改为 stderr。

### 3.2 `--format json` 时 stdout 只包含一个完整 JSON 对象 —— 符合

- 证据：
  - `src/cli/index.ts:538-554`（deliberate）：单次 `JSON.stringify({…})` 后接一个 `\n`，整段写一次 stdout，无其它 stdout 输出。
  - `src/cli/index.ts:704-720`（resume）：同上结构，单一完整 JSON 对象。
- 说明：JSON 经一次 `JSON.stringify` 原子生成，不存在分片写入。

### 3.3 进度、警告、Token 地址和档案路径进入 stderr —— 部分符合（警告未进入 stderr）

- 符合项证据：
  - 进度信息全部走 stderr：`src/cli/index.ts:277`（检查点进度）、`:325`（讨论窗口进度）、`:412`（工作目录警告/进度）、`:472`（审议已创建）、`:479`（正在组局）、`:558`（审议档案路径）、`:722`（resume 档案路径）。
  - Token 地址走 stderr：`src/cli/index.ts:742`（`mad serve` 将含 Bearer Token 的 URL 写入 stderr）。
  - 档案路径走 stderr：见上 `:472`、`:558`、`:722`。
- 不符合项证据（警告）：
  - `warnings` 数组仅在 `src/cli/index.ts:513-516` 构建，于 `:517` 写入档案事件，并在 JSON 模式作为 `warnings` 字段进入 stdout（`:552`、`:718`）。
  - 全代码无任何将 `warnings` 写入 `process.stderr` 的语句。因此在默认的 markdown 模式下，警告既不在 stdout（符合第 1 条），也不在 stderr（违反第 3 条），用户对警告完全无感，只能事后查档案。
  - 这与第 3 条"警告进入 stderr"直接冲突，且与第 7 条（JSON 含警告）在 JSON 模式下形成张力：JSON 模式下警告位于 stdout 而非 stderr。
- 结论：进度、Token、档案路径三条满足；警告一条不满足，整体判定为 部分符合。

### 3.4 非 TTY 输出不含 ANSI 控制字符 —— 符合

- 证据：全项目代码中（含 `src/cli/index.ts`）未引入任何颜色/样式库（无 `chalk` 等），所有 `process.stderr.write` / `process.stdout.write` 均为纯文本字面量，不含 `\x1b[` 等转义序列。
- 说明：实现采用"从不注入 ANSI"而非"TTY 检测后剥离"的策略。由于从不生成 ANSI，无论 TTY 与否输出均无控制字符，满足要求。此策略稳健，但缺少显式 TTY 守卫，若后续引入着色库需补检测逻辑（见偏差清单 D1）。

### 3.5 失败时不向 stdout 写半截 JSON —— 符合

- 证据：
  - JSON 仅在 `result` 完整生成后于 `src/cli/index.ts:538`、`:704` 单次写出，位于 try 块最末段，紧随其后的 `:558`、`:722` 仅是 stderr 档案路径。
  - 失败路径：`src/cli/index.ts:559-568`（deliberate）与 `:723-730`（resume）的 catch 仅更新档案状态并 `throw error`；顶层 `src/cli/index.ts:784-788` 的 `.catch` 仅向 stderr 写错误信息并设置退出码，从不写 stdout。
  - 因 JSON 与 `result.report` 在同一原子 `JSON.stringify`/`write` 中完成，失败发生在写入之前则 stdout 完全为空，不会产出半截内容。

### 3.6 暂停、取消、配置错误、预检失败、执行失败使用不同退出码 —— 符合

- 证据：`src/core/errors.ts:27-35` 定义 `EXIT_CODES`：
  - `USAGE: 2`、`CONFIG: 3`、`PREFLIGHT: 4`、`LOCKED: 5`、`PAUSED: 20`、`CANCELLED: 21`、`EXECUTION: 30`。
  - 五种目标码映射为：暂停=20、取消=21、配置错误=3、预检失败=4、执行失败=30，彼此不同。
- 使用证据：
  - 暂停：`PAUSED` 抛出于 `src/cli/index.ts:226`（方案确认被中断）、`:564`、`:727`（catch 中按状态置 `paused`）。
  - 取消：`CANCELLED` 抛出于 `:230`、`:563`、`:726`。
  - 配置错误：`CONFIG` 广泛使用（如 `:83`、`:378`、`:382`、`:410`、`:418`、`:582`、`:604`、`:740`、`:781` 等）。
  - 预检失败：`PREFLIGHT` 于 `:115` 抛出。
  - 执行失败：`EXECUTION` 于 `:616`、`:758` 抛出。
- 退出码落地：顶层 `src/cli/index.ts:787` 以 `EXIT_CODES[madError.code]` 设置 `process.exitCode`，五类错误各归其码。符合。

### 3.7 JSON 至少包含审议 ID、状态、模式、报告、参与者、预算使用、警告、档案路径 —— 符合

- 证据（deliberate，`:539-554`；resume，`:705-720`）：
  - 审议 ID：`deliberation_id: id` ✓
  - 状态：`status: "completed"` ✓
  - 模式：`mode`（deliberate） / `mode: manifest.mode`（resume） ✓
  - 报告：`report: result.report` ✓
  - 参与者：`participants: plan.participants` ✓
  - 预算使用：`budget_usage` 含 `call_attempts`/`max_calls`/`timeout_seconds`/`context_budget`/`global_concurrency` ✓
  - 警告：`warnings` ✓
  - 档案路径：`archive_path: archive.path` ✓
- 备注（低严重度）：`status` 在代码中硬编码为 `"completed"`（`:541`、`:707`），不参与 paused/cancelled 等真实终态（这些情况下 JSON 根本不写出）。当前不触发 bug，但若未来对 paused/cancelled 也生成 JSON 摘要，需改为真实状态。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| D1 | 第16节 第4条 | 输出策略"从不注入 ANSI"，但缺少显式 TTY 守卫；一旦后续引入着色库，非 TTY 场景可能泄漏控制字符。 | 低 | 当前无 ANSI 风险，属未来回归隐患。 | 抽象一个 `writeErr`/`writeOut` 封装，依据 `isTTY` 决定是否着色，集中守卫。 |
| D2 | 第16节 第3条 | 警告（workspace 授权提示、共享来源约束）仅写入档案事件与 JSON 字段，未写入 stderr；markdown 模式下用户完全看不到警告。 | 中 | 违反"警告进入 stderr"契约；安全相关提示（完整目录只读授权）在默认模式下不可见，存在安全感知缺失。 | 在 markdown/自动模式向 `process.stderr` 同步输出 `warnings` 内容；与 JSON 模式字段保持同源。 |
| D3 | 第16节 第3/7条 | JSON 模式下警告位于 stdout（`warnings` 字段），与"警告进入 stderr"存在字面张力。 | 低 | 机器调用时警告随 JSON 正常获取，不丢失；但契约本身两处表述需统一口径。 | 在架构文档或本实现注释中明确：json 模式警告以字段形式承载于 stdout，stderr 不重复；markdown 模式警告进 stderr。 |
| D4 | 第16节 第1条 | `mad config validate`/`mad config check` 成功向 stdout 写状态摘要，非"最终 Markdown 报告"。 | 低 | 对机器解析 stdout 的调用方可能误判；属契约边界模糊。 | 明确 config 子命令 stdout 契约豁免，或改为 stderr 输出状态摘要。 |
| D5 | 第16节 第7条 | JSON `status` 硬编码 `"completed"`，未承载 paused/cancelled 真实终态。 | 低 | 当前因失败不写 JSON 而不暴露；若扩展 JSON 摘要场景会失真。 | 状态由档案 `state.status` 驱动，避免硬编码。 |

## 5. 结论与优先建议

总体结论：当前实现与第 16 节七项契约基本一致，第 1、2、4、5、6、7 条均判定为 符合，第 3 条因"警告未进入 stderr"判定为 部分符合。

优先级建议：
1. 高优先（契约合规 + 安全感知）：修复 D2——将 `warnings` 同步写入 stderr（markdown 与自动模式），确保默认模式下用户可见工作目录只读授权与共享来源约束提示，满足第 3 条。
2. 中优先（契约清晰度）：处理 D3/D4——在代码或文档中统一"警告在 json 模式随字段、markdown 模式进 stderr"以及 config 子命令 stdout 豁免的口径。
3. 低优先（稳健性）：推进 D1 的 TTY 守卫封装与 D5 的状态真实化，防止未来回归。
