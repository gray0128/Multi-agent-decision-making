## 2. 目标架构要求摘要

第 1 节（目标与约束）关键约束：

- 移除 Python、Microsoft Agent Framework、Agent Framework DevUI。
- 以单个 npm 包分发，首版要求 Node.js，不建立 monorepo。
- TypeScript 审议进程是运行状态与正式审议档案的唯一写入者。
- 页面不能发起、恢复、删除审议或修改 CLI 注册表，只能响应当前检查点。
- 本地服务只绑定 `127.0.0.1`，不提供局域网或公网监听模式。

第 2 节（总体结构）关键关系：

- `mad deliberate`（TS 审议进程，唯一状态所有者）与 `mad serve`（本地观察服务，只读档案 + 受限响应）通过 `MAD_HOME/runtime` 下的文件信箱通信。
- 审议进程原子发布待处理检查点；观察服务只能原子写入一次性响应文件，不能直接修改审议档案。

第 3 节（单包代码结构）关键声明：

- 单个 `package.json`，`src/` 下划分 `cli/`、`core/`、`adapters/`、`archive/`、`server/`、`web/`。
- CLI、服务端、页面共享同一套 schema 与领域类型；页面资源随 npm 包发布。

## 3. 当前实现对照

### 要点 1：是否为单个 package.json（非 monorepo）

符合。

- `package.json:1-32` 为单一包，无 `workspaces` 字段，无 monorepo 配置。
- 命令入口 `bin.mad` 指向 `./dist/cli/index.js`（`package.json:6-8`），符合「单个 npm 包分发」。
- 运行时依赖仅 `@iarna/toml`（`package.json:23-25`），开发依赖为 `tsx`、`typescript`、`vitest`、`@types/node`（`package.json:26-31`），无额外框架。

### 要点 2：src/ 实际子目录是否与第 3 节声明的树一致

符合。

- 第 3 节声明树：`cli/`、`core/`、`adapters/`、`archive/`、`server/`、`web/`。
- 实际 `src/` 一级目录（`find src -maxdepth 1 -type d`）：`src/cli`、`src/core`、`src/adapters`、`src/archive`、`src/server`、`src/web`，与声明完全一致，无缺漏、无多余顶层模块。
- 各子目录职责与第 3 节描述吻合：`cli/`（命令解析、检查点、stdout/stderr 契约，见 `src/cli/index.ts`）、`core/`（组局、结构化审议、自由讨论、报告流水线，见 `src/core/types.ts`、`structured.ts`、`discussion.ts`、`planning.ts`、`execution.ts`、`outcome.ts`）、`adapters/`（CLI 适配器与类型化配置校验，见 `src/adapters/config.ts`、`codex.ts`、`generic.ts`、`types.ts`）、`archive/`（透明档案、锁与恢复，见 `src/archive/store.ts`、`index.ts`）、`server/`（HTTP、认证、SSE、文件信箱，见 `src/server/observer.ts`、`mailbox.ts`）、`web/`（静态 HTML/CSS 与原生页面脚本，见 `src/web/index.ts`）。

### 要点 3：CLI、服务端、页面是否共享同一套 schema / 领域类型

符合（含一处设计性边界说明）。

- 领域类型的单一来源是 `src/core/types.ts`，定义了 `DeliberationMode`、`DeliberationStatus`、`DeliberationPlan`、`DeliberationAgent`、`ResourceLimits`、`WorkspaceAccess`、`InvocationConfigSnapshot` 等（`src/core/types.ts:1-60+`）。
- 该模块被 CLI、归档、适配器共同导入复用：`src/cli/index.ts:36`（从 `../core/types.js` 导入）、`src/archive/store.ts:6`（导入 `DeliberationManifest`、`FrozenInvocation`、`InvocationResult`）、`src/adapters/config.ts`、`src/adapters/generic.ts`、`src/adapters/codex.ts`、`src/adapters/process.ts` 均从 `../core` 导入类型与错误定义。
- 服务端以「档案 JSON」作为共享 schema 边界：观察服务读取的是同一套落盘文件（`manifest.json`、`state.json`、`events.jsonl`、`transcript.jsonl`、`report.md`，见 `src/server/observer.ts:65-79`），与 CLI/归档写入的 schema 完全一致；页面（`src/web/index.ts` 内联 `app.js`）消费同一份 API JSON（`/api/deliberations`、`/api/deliberations/:id`、`/api/checkpoints/:id/respond`）。
- 说明（非偏差）：服务端 `src/server/observer.ts:5` 仅导入 `AppPaths` 类型、页面为浏览器内联脚本，二者并不直接 `import` `core/types.ts`，而是通过 HTTP/JSON 边界消费同一 schema。这属于合理的序列化边界，不影响「共享同一套 schema 与领域类型」的实质。

### 要点 4：是否确实已移除 Python 实现、Microsoft Agent Framework、Agent Framework DevUI

符合。

- 对全仓（排除 `node_modules`）检索 `python|Python|agent framework|Agent Framework|DevUI|devui|microsoft|Microsoft`，在源码 `src/` 中无任何残留引用；命中项仅位于 `docs/`（目标架构与验收文档本身的陈述）、`README.md`（「TypeScript 版不会读取、迁移、展示或删除旧版本产生的 Python 配置和档案」，`README.md:216`）及 `node_modules` 第三方依赖（如 `typescript`、`@types/node` 的版权声明），均非本实现代码。
- 仓库根目录无 `pyproject.toml`、`setup.py` 或源码 `.py` 文件（`git ls-files | grep -i pyproject` 为空）。
- 工作树存在 `.venv/`（Python 虚拟环境），但 `.gitignore` 已忽略 `.venv/`（` .gitignore` 首行），且 `git ls-files --error-unmatch .venv` 返回 NOT TRACKED，说明它未被纳入版本控制，属于本地遗留目录，不影响「已移除 Python 实现」的结论。详见第 4 节风险清单 R1。

### 要点 5：第 2 节总体结构中的通信关系是否在代码中体现

符合。

- CLI 唯一状态所有者：CLI 侧通过 `ArchiveStore`、`ActiveDeliberationLock` 写入档案（`src/cli/index.ts:28` 导入 `ActiveDeliberationLock, ArchiveStore`），且服务端代码从不导入 `archive/store`（`src/server/index.ts:1-3` 仅 `export *` 来自 `observer.js` 与 `mailbox.js`），从依赖上保证观察服务不能写正式档案。
- 文件信箱：审议进程原子发布待处理检查点、观察服务原子写入一次性响应，由 `CheckpointMailbox` 实现。请求/响应文件路径位于 `runtime/checkpoints/`（`src/server/mailbox.ts:40-43`）；`publishExclusiveJson` 用临时文件 + `link` 的原子发布/独占写入（`src/server/mailbox.ts:21-33`）。
- 观察服务只读档案：观察服务详情接口只读取 `manifest.json`、`state.json`、`events.jsonl`、`transcript.jsonl`、`report.md`（`src/server/observer.ts:65-79`），并在响应路由中仅向 `checkpoints/<id>.response.json` 写入一次性响应（`src/server/observer.ts:174-180`），从不写入 `state.json` 或档案目录。
- 受限响应（页面只能响应检查点）：服务端唯一的可写接口为 `POST /api/checkpoints/:id/respond`（`src/server/observer.ts:148-182`），且做过期/无效校验（`checkpointId` 不匹配、`action` 不在允许列表、已应答则返回 409，`src/server/observer.ts:165-180`）；无发起、恢复、删除审议或修改注册表的任何路由，符合「页面只能响应当前检查点」。
- 本地服务仅绑定 `127.0.0.1`：`SERVER_HOST = "127.0.0.1"`（`src/server/index.ts:1`），`server.listen(port, "127.0.0.1", ...)`（`src/server/observer.ts:190`），Token 通过 URL fragment 下发（`src/server/observer.ts:202`）。
- 观察服务不直接写正式档案：与 `README.md:226`「观察服务不直接写正式档案，只通过一次性文件信箱响应检查点」一致，代码证据见上。

### 要点 6：是否存在目标架构未提及的额外模块/依赖需要说明

符合（仅有源码外支撑目录，无需架构变更）。

- `package.json` 运行时依赖仅 `@iarna/toml`，无前端框架、路由框架或全局状态库，符合第 3 节「页面不引入前端框架、路由框架或全局状态库」。
- 仓库根存在 `tests-ts/`（vitest 测试，如 `adapters-ts.test.ts`、`archive.test.ts`、`cli-e2e.test.ts` 等）与空的 `scripts/` 目录；二者均不在 `src/` 树内（`tsconfig.json:21` 仅 `include: ["src/**/*.ts"]`），属于源码外测试/脚本支撑目录，不改变第 3 节声明的 `src/` 结构，无需在目标架构中新增模块。
- 未发现目标架构未提及的运行时依赖或业务模块。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|---|---|---|---|---|---|
| R1 | 第 1 节 / 要点 4 | 工作树存在 `.venv/` Python 虚拟环境目录（含 dotenv、httpx、pytest 等第三方包），虽已被 `.gitignore` 忽略且未被 git 跟踪，但作为本地残留易让人误判「Python 未彻底移除」。 | 低 | 不影响已发布产物与架构一致性；仅影响本地整洁与审查误读。 | 从工作树删除 `.venv/`（或运行 `git clean -fdX` 清理被忽略目录），并在贡献指南中说明本项目不再含 Python 依赖。 |
| R2 | 第 3 节 / 要点 3 | 观察服务（`src/server/observer.ts`）与页面（`src/web/index.ts` 内联 `app.js`）通过 HTTP/JSON 边界消费 schema，不直接 `import` `src/core/types.ts`。 | 低 | 属合理序列化边界，实质满足「共享同一套 schema」；但服务端对档案字段的解析为隐式 JSON 形态，若 `core/types.ts` 演变需靠人工保持两侧一致。 | 建议在 `src/server/` 或文档中保留一份与 `core/types.ts` 对应的档案 JSON 形状说明（或轻量运行时校验），降低 schema 漂移风险。 |
| R3 | 第 3 节 | 页面资源以 TS 字符串常量内联于 `src/web/index.ts`（`INDEX_HTML`/`STYLES_CSS`/`APP_JS`），并非「编译后的原生 TypeScript 文件」随包发布。 | 低 | 仍能随 npm 包发布、无需独立前端构建，功能等价；与目标架构字面表述略有出入。 | 维持现状可接受；若需严格贴合「原生 TypeScript」表述，可将页面脚本拆分为独立 `.ts` 并在构建期编译，但当前内联方案对单包分发更简洁。 |

## 5. 结论与优先建议

结论：当前 TypeScript 实现与目标架构第 1、2、3 节高度一致。六个审查要点全部判定为「符合」（其中要点 3、6 含低严重度的设计性说明）。

- 已严格满足：单 npm 包分发（非 monorepo）、`src/` 六目录与目标树完全一致、Python/Agent Framework/DevUI 已从源码与版本库移除、CLI 为唯一状态写入者、文件信箱与观察服务只读档案、本地服务仅绑定 `127.0.0.1`、页面仅能响应检查点、CLI/服务端/页面共享同一套 schema 与领域类型、未引入额外框架或依赖。

优先建议（均低优先级，不阻塞接管）：
1. 清理工作树遗留的 `.venv/` 目录（R1），消除对「Python 已移除」的误读。
2. 为观察服务侧的档案 JSON 形态补充轻量一致性说明或校验（R2），防范 `core/types.ts` 演变引发的隐蔽漂移。
3. 视需要决定是否将内联页面脚本改为独立原生 TS 文件（R3），以更贴合目标架构字面表述。
