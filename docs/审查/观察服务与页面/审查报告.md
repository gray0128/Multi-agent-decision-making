# 观察服务与页面审查报告

> 本审查仅以 `docs/TypeScript目标架构.md` 第 11、12 节与当前源码为准，未参考任何其它实现/审查报告。

## 1. 审查范围与依据

- 目标架构：`docs/TypeScript目标架构.md` 第 11 节（观察服务与页面）、第 12 节（本地认证）。
- 当前实现源码：
  - `src/server/index.ts`、`src/server/observer.ts`、`src/server/mailbox.ts`、`src/web/index.ts`、`src/cli/index.ts`、`src/core/paths.ts`。
  - 观察服务与页面相关模块（`src/server/*`、`src/web/*`）仅含单一 `web/index.ts`，无独立静态资源目录。
- 对照要点（共 7 项）：服务独立性、文件信箱通信、页面技术栈、Markdown 净化、流式展示边界、本地认证、认证与监听约束。

## 2. 目标架构要求摘要

- 第 11 节：`mad serve` 为独立长期本地进程，审议不依赖其存在；审议进程与观察服务通过 `MAD_HOME/runtime` 下仅当前用户可访问的文件信箱通信；审议进程原子发布待处理检查点、观察服务只能原子写入一次性响应、审议进程校验消费后独占更新正式状态、观察服务不能直接改档案；页面首版为静态 HTML/CSS + 少量原生 TS，不引入框架/路由/全局状态库；Markdown 必须独立解析净化、禁止未净化 HTML 注入；流式 `fetch` 只展示状态事件，参与者正文仅在调用完成并提交权威记录后整体显示，不做 token 级流、不展示 thinking/reasoning、不发原始 stdout/stderr。
- 第 12 节：每次启动生成新 Bearer Token，经自动打开或打印的 URL fragment 交给页面；页面读取后立即从地址栏移除并只存 `sessionStorage`；所有档案 API、流式读取、检查点响应均用 `Authorization: Bearer ...`；Token 不进查询参数/长期 Cookie/磁盘；服务固定监听 `127.0.0.1`。

## 3. 当前实现对照

### 要点 1：`mad serve` 独立长期进程；审议不依赖其存在
- **结论：符合**
- 证据：
  - `mad serve` 启动观察服务后阻塞等待 `SIGINT`/`SIGTERM`，属独立长期进程：`src/cli/index.ts:737-749`（serve 函数，`await new Promise` 等待终止信号，最后 `observer.close()`）。
  - 审议进程与观察服务通过文件系统信箱通信，审议进程直接读 `runtime/checkpoints/<id>.response.json`，并不与 HTTP 服务建立连接：`src/server/mailbox.ts:74-93`（`while(true)` 轮询读取 `responsePath`，200ms 间隔）。
  - 服务未启动时用终端检查点：`src/cli/index.ts:274-287`（`terminalAvailable` 时传入 `local` 终端回调）；`src/cli/index.ts:396-397` 仅在 `guided && 无终端 && 无观察者` 时立即失败，否则不依赖服务。
  - 自动模式按策略继续：`src/core/structured.ts:190`（`if (!this.checkpoint) return;` 跳过检查点）、`src/core/discussion.ts:198`（`if (!this.checkpoint) return plan.converged ? "end" : "continue";`）。
  - 服务重启不中断审议：审议进程仅读写文件信箱，不持有到服务的长连接，重启服务不影响正在进行的审议。

### 要点 2：文件信箱通信（原子发布/一次性响应/独占更新/只读档案）
- **结论：符合**
- 证据：
  - 信箱位于 `MAD_HOME/runtime`：`src/core/paths.ts:34`（`runtime: join(home, "runtime")`）；目录以 `0o700` 创建并 `chmod 0o700`：`src/server/observer.ts:89-92`。
  - 审议进程原子发布待处理检查点：`src/server/mailbox.ts:58-60`（先写 `*.tmp` 再以 `rename` 原子替换请求文件）。
  - 观察服务只能原子写入一次性响应：`src/server/observer.ts:174-180`（`publishExclusiveJson` 用 `link` 原子写入；若已存在返回 `false` 并回 `409 Checkpoint already answered`，保证一次性）。
  - 审议进程校验并消费响应、独占更新正式状态：`src/server/mailbox.ts:77-92`（校验 `checkpointId` 与 `action` 合法后 `onAccepted` → `archive.recordCheckpointDecision`，消费后删除请求与响应文件）；`src/cli/index.ts:294-298` 回调写入权威决策。
  - 观察服务不能直接修改审议档案：`src/server/observer.ts` 中所有档案读取均经 `jsonFile`/`jsonLines`/`readFile`/`readdir`，无任何写入 `manifest/state/events/transcript/report` 的调用；仅向 `runtime/checkpoints/<id>.response.json` 与 `runtime/server.json` 写入心跳。
  - 响应文件与请求文件权限为 `0o600`（仅当前用户）：`src/server/mailbox.ts:23`、`:59`。

### 要点 3：页面首版静态 HTML/CSS + 少量原生 TS，无框架/路由/全局状态库
- **结论：符合**
- 证据：
  - 全部页面资源内联于单一 `src/web/index.ts`（`INDEX_HTML`、`STYLES_CSS`、`APP_JS` 三个字符串常量），无独立 `web/` 静态资源目录、无 `node_modules` 前端依赖。
  - `APP_JS` 为原生脚本：`<script type="module" src="/app.js">`（`src/web/index.ts:13`），逻辑为原生 `fetch`/`DOM` 操作，未 `import` 任何框架（React/Vue 等）或路由库、全局状态库。
  - 仅使用 `sessionStorage` 存取 Token（`src/web/index.ts:21`），属浏览器原生存储，非全局状态库。
  - 说明：源码为 TypeScript，构建后以 `app.js` 名义提供；属“原生 TS→JS”首版形态，未引入框架，符合初衷。

### 要点 4：Markdown 必须经独立解析与净化，禁止直接注入未净化 HTML
- **结论：符合**
- 证据：
  - 渲染函数 `markdown()` 先对整个源做 HTML 转义 `esc()`，再做白名单正则替换为受限标签（仅 `h1/h2/h3`、`strong`、`code`、`li`、`p`、`br`）：`src/web/index.ts:23`。
  - 转义先于解析，结构上不可能注入未净化原始 HTML；报告（`report.md`）与发言正文（`transcript` 内容）均经此路径渲染：`src/web/index.ts:28`（`transcriptHtml`）、`:26`（报告 `markdown(data.report)`）。
  - 说明：解析器为内置轻量实现而非独立第三方净化库，但采用“先转义、后白名单”的构造式净化，满足“禁止未净化 HTML 注入”的硬性约束。

### 要点 5：流式 fetch 仅展示状态事件；正文调用完成提交后整体显示；不做 token 流/thinking/原始 stdout-stderr
- **结论：符合**
- 证据：
  - 流式读取 `events.jsonl` 为 SSE：`src/server/observer.ts:124-147`；浏览器端 `stream()` 解析 `data:` 帧并以 `eventHtml` 展示：`src/web/index.ts:32`、`:29`。
  - 状态事件展示字段受限为类型 + `stage/agentId/logicalCallId/durationMs/message` 白名单：`src/web/index.ts:29`（`eventHtml` 仅取这些字段，`esc` 转义）。
  - 参与者正文仅在调用提交权威记录后整体显示：`transcriptHtml` 渲染 `data.transcript`（即 `transcript.jsonl` 权威记录），且仅在收到 `invocation.committed` 事件时刷新：`src/web/index.ts:28`、`:32`（`if(event.type==='invocation.committed')refreshTranscript(id)`）。
  - 不做 token 级文本流：服务端仅推送结构化事件帧，无增量正文流。
  - 不展示 thinking/reasoning：事件为生命周期结构化事件，浏览器仅渲染白名单字段，obs 端 `detail()` 不读取 `diagnostics.jsonl`：`src/server/observer.ts:65-79`（仅 `manifest/state/events/transcript/checkpoint/report`）。
  - 不把原始 stdout/stderr 发往浏览器：同上，`detail()` 不读取 `diagnostics.jsonl`，且事件渲染层只取白名单字段，原始 stdout/stderr 不会进入页面。

### 要点 6：每次启动生成新 Bearer Token，经 URL fragment 交给页面；读取后立即移除并仅存 sessionStorage
- **结论：符合**
- 证据：
  - 每次启动生成新 Token：`src/server/observer.ts:93`（`randomBytes(32).toString("base64url")`）。
  - URL 以 fragment 携带 Token：`src/server/observer.ts:202`（`url: .../#token=${encodeURIComponent(token)}`），fragment 不会发送至服务器。
  - 页面读取后从地址栏移除并只存 `sessionStorage`：`src/web/index.ts:21`（`fragment.get('token')` → `sessionStorage.setItem` → `history.replaceState(null,'',location.pathname)` 清除 fragment）。
  - 交付方式：服务将含 fragment 的 URL 打印到 stderr（确认模式为“自动打开或打印”的“打印”路径）：`src/cli/index.ts:742`。当前未实现浏览器自动打开，但“或”条件下打印即满足要求。

### 要点 7：所有档案 API/流式读取/检查点响应均用 `Authorization: Bearer`；Token 不进查询参数/长期 Cookie/磁盘；固定监听 127.0.0.1
- **结论：符合**
- 证据：
  - 统一鉴权：`authorized()` 校验 `Bearer`：`src/server/observer.ts:35-40`；所有 `/api/` 路由前置校验，未通过返回 `401`：`src/server/observer.ts:100`。
  - 流式读取受鉴权保护：事件 SSE 路由 `/api/deliberations/<id>/events` 以 `/api/` 前缀命中上述网关：`src/server/observer.ts:124`（`:100` 已先拦截未授权请求）。
  - 检查点响应受鉴权保护：POST `/api/checkpoints/<id>/respond` 同样位于 `/api/` 网关内：`src/server/observer.ts:148-149`。
  - Token 不进查询参数：事件流的查询参数仅含 `after` 偏移：`src/server/observer.ts:128`；Token 仅位于 fragment（不上送）。
  - Token 不进长期 Cookie/磁盘：页面仅存 `sessionStorage`（`src/web/index.ts:21`）；服务端心跳 `server.json` 仅含 `pid/port/startedAt`：`src/server/observer.ts:197`，Token 不落盘。
  - 固定监听 `127.0.0.1`：`src/server/observer.ts:190`（`server.listen(port, "127.0.0.1", resolve)`）；常量 `SERVER_HOST = "127.0.0.1"`：`src/server/index.ts:1`。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| R1 | 第 12 节 | 未实现浏览器“自动打开”，仅将含 fragment 的 URL 打印到 stderr（满足“自动打开或打印”的打印分支） | 低 | 用户需手动复制/点击 URL，体验略弱，但不影响安全与功能 | 后续可加 `--open` 选项调用系统打开命令，仍保持 Token 仅在 fragment |
| R2 | 第 11 节 / 要点 4 | Markdown 采用内置轻量正则解析器，而非独立维护的净化库 | 低 | 转义优先保证无原始 HTML 注入；复杂/嵌套 markdown 或代码块内原始标记渲染可能不符合预期 | 后续若需更强语义，可替换为经审计的解析+净化库，并保留“先转义”原则 |
| R3 | 第 12 节 | 静态外壳 `/`、`/styles.css`、`/app.js` 不经鉴权即返回（仅为无数据的页面骨架与脚本） | 低 | 不泄露档案数据；但理论上可被本机其它进程拉取页面骨架 | 当前 `127.0.0.1` + 仅存内存 Token 已足够；如需深度防御可对静态资源也要求 Bearer |
| R4 | 第 11 节 / 要点 5 | 页面内容经 `innerHTML` 注入；当前所有数据源均先 `esc()` 转义，风险低 | 低 | 若未来 `markdown()` 调整而遗漏转义，可能引入 DOM 注入 | 在 `markdown()`/渲染函数处固化“先转义”约束，并补充针对渲染函数的单测 |
| R5 | 第 11 节 / 要点 2 | 检查点请求文件在被 resume 复用同一 `checkpointId` 时会被重新 `rename` 覆盖 | 低 | 不影响一次性响应语义；同一检查点请求内容相同，覆盖无害 | 保持现状即可，建议在注释中明确 resume 复用语义 |

## 5. 结论与优先建议

对照目标架构第 11、12 节七项要点，当前实现均达到**符合**级别，核心安全与架构约束（文件信箱原子通信、观察服务只读档案、本地认证 Bearer + 固定 127.0.0.1 + sessionStorage、Markdown 先转义、流式仅状态事件且正文提交后整体显示）均已落实并有源码证据支撑。

优先建议（按性价比）：
1. 维持并固化 `markdown()` “先转义、后白名单”的渲染约束（R4），补充单测防止回归。
2. 可选增强：提供 `--open` 自动打开浏览器（R1），进一步贴合“自动打开或打印”的首选体验。
3. 后续若页面 Markdown 复杂度上升，评估替换为经审计的解析+净化库（R2），但须保留当前“禁止未净化 HTML 注入”的硬性边界。
