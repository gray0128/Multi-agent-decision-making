# CLI 适配器与注册表审查报告

审查依据（仅限目标架构与源码，未读取 `docs/TypeScript当前实现架构审查.md` 及 `docs/审查/` 下其它子目录报告）：

## 2. 目标架构要求摘要

第 4 节核心要求：

1. 长期配置 `config/clis.toml` 只保存可信 CLI 执行边界（cli id / adapter / executable / timeout_seconds / max_concurrency / presets），不预设长期 Agent 或角色。
2. 模型调用预设（preset）是「模型 + 思考等级等推理设置」的可信组合，同一模型可有多个预设。
3. 审议方案只能引用「CLI 配置 + 调用预设」，不能携带裸模型名、任意推理设置、可执行路径或 CLI 参数。
4. 不支持 `extra_args` 等原始参数透传。
5. 每个适配器定义自己的类型化配置 schema，并禁止配置覆盖只读、审批、工具限制、输出格式和工作目录等安全参数。
6. 验证分两层：加载时静态验证字段/类型/枚举/组合；运行时实际预检组局器，方案生成后对每个唯一「CLI+预设」组合预检一次，共享组合不重复预检。
7. 任何验证或预检失败都阻止审议启动，不自动替换 CLI/模型/思考等级。
8. `mad init` 只探测已安装 CLI 与可执行路径，生成带适配器说明的配置骨架，不猜测模型/思考等级/默认组局器；已有配置默认不覆盖，用户显式要求时才重建。
9. `mad config validate` 静态验证，`mad config check` 实际预检。

## 3. 当前实现对照

### 要求 1：长期配置只保存 CLI 执行边界，不预设 Agent/角色
**判定：符合**

- `CliConfig` 仅含 `id / adapter / executable / timeoutSeconds / maxConcurrency / presets`（`src/adapters/config.ts:28-35`），`CliRegistry` 仅含 `defaults.generator` 与 `clis`（`src/adapters/config.ts:37-40`）。
- 解析逻辑 `parseCli` 明确只接受 `id / adapter / executable / timeout_seconds / max_concurrency / presets` 六个字段，未知字段直接报错（`src/adapters/config.ts:122`）。
- 配置中无任何 Agent、role、participant 字段；角色由组局阶段在运行时生成（见 `src/core/planning.ts:218-227` 组局提示与 `parseDeliberationPlan`）。
- 落点确认为 `${MAD_HOME}/config/clis.toml`（`src/core/paths.ts:32`），与目标命名一致。

### 要求 2：预设是模型+推理设置的可信组合，同模型可多预设
**判定：符合**

- `InvocationPreset` 由 `id / model / contextBudget / options` 组成，其中 `options` 仅包含推理等级（`reasoningEffort` / `effort` / `thinking`）（`src/adapters/config.ts:15-26`）。
- `parsePreset` 解析每个预设并校验（`src/adapters/config.ts:78-117`），仅对 `preset.id` 做唯一性约束（`src/adapters/config.ts:131-135`），对 `model` 无唯一性约束，因此同一模型可拥有多个不同预设（不同 id / 不同推理等级）。
- 每个 CLI 至少需一个预设（`src/adapters/config.ts:127-129`），满足「可信组合」可枚举、可引用。

### 要求 3：审议方案只能引用 CLI+预设，不能携带裸模型名/参数
**判定：符合**

- `parseDeliberationPlan` 对参与者仅允许 `id / cli / preset / role` 四个键，多余键报错（`src/core/planning.ts:69`）；每个参与者的 `cli/preset` 必须经 `resolveInvocation` 在注册表中存在性校验（`src/core/planning.ts:74`）。
- 组局提示明确禁止组局器输出模型名、命令、可执行路径、CLI 参数、权限、环境变量、秘密或配置修改（`src/core/planning.ts:218-226`）。
- 方案对外表达 `externalPlan` 仅输出 `cli / preset / role`（`src/cli/index.ts:128-139`），不含裸模型名。
- 命令行 `--organizer` 也仅接受 `CLI/PRESET` 格式并经 `resolveInvocation` 校验（`src/cli/index.ts:416-421`），不接受裸模型名。

### 要求 4：不支持 extra_args 等原始参数透传
**判定：符合**

- 全仓搜索 `extra_args / passthrough / passthru / rawArgs / raw_args` 均无匹配。
- 所有 CLI 实际参数在适配器中硬编码：Codex 见 `src/adapters/codex.ts:52-66`（含 `--sandbox read-only --ephemeral --color never --skip-git-repo-check` 等固定参数）；通用适配器按 adapter 分支硬编码于 `src/adapters/generic.ts:18-29`。配置层无任何「透传参数」字段。

### 要求 5：每适配器定义类型化配置 schema，禁止覆盖安全参数
**判定：部分符合**

- 安全参数在适配器中硬编码且不可被配置覆盖：Codex 固定 `--sandbox read-only` 等（`src/adapters/codex.ts:52-66`）；Claude/Grok 固定 `--output-format json --permission-mode plan --tools Read,Glob,Grep,... --no-session-persistence --safe-mode --strict-mcp-config`（`src/adapters/generic.ts:21,23`）；Pi 固定 `--no-session --no-approve --no-extensions ... --tools read,grep,find,ls`（`src/adapters/generic.ts:24`）。只读/审批/工具限制/输出格式均不可配置覆盖。
- 每个适配器的「可配置推理选项」白名单在 `src/adapters/config.ts:82-86` 集中定义（codex→`reasoning_effort`，claude/grok→`effort`，pi→`thinking`，其余为空），越权选项直接报错（`src/adapters/config.ts:86`）。
- 偏差点：目标措辞为「每个适配器定义自己的类型化配置 schema」，但当前实现将**全部**适配器的配置 schema 与解析集中放在 `src/adapters/config.ts`，并未下放到各适配器文件（如 `codex.ts` / `generic.ts`）各自声明。功能上「按适配器限定可配置项 + 禁止覆盖安全参数」已完全满足，仅是代码组织未严格满足「每适配器自含 schema」。详见偏差清单 R1。

### 要求 6：两层验证（静态 + 运行时预检，共享组合不重复）
**判定：符合**

- 第一层（加载时静态验证）：`parseCliRegistry` 校验字段、类型、枚举（adapter 枚举 `src/adapters/config.ts:124-126`；reasoning/effort/thinking 枚举 `src/adapters/config.ts:88-101`）、组合（默认组局器引用的 cli/preset 必须存在 `src/adapters/config.ts:162-168`；cli/preset id 唯一 `src/adapters/config.ts:131-135,157-161`）。
- 运行时先预检组局器：`OrganizerService.propose` 在生成方案前先 `requireReady(generatorAdapter)`（`src/core/planning.ts:138`）。
- 方案生成后对唯一组合预检一次、共享组合不重复：`preflightPlan` 用 `Map` 以 `${cli}/${preset}` 为键去重（`src/core/planning.ts:187-198`），再并发预检（`src/core/planning.ts:199-202`）。
- `mad config check` 同样对每个 cli×preset 组合执行一次 `adapter.check`（`src/cli/index.ts:109-123`）。
- 组局器若未证明支持只读则禁止用于项目模式（`src/core/planning.ts:135-137`）。

### 要求 7：验证/预检失败阻止启动，不自动替换 CLI/模型/思考等级
**判定：符合**

- `mad config check` 任一组合预检失败即抛 `PREFLIGHT` 错误并退出（`src/cli/index.ts:114-119`）。
- 组局阶段 `preflightPlan` 失败抛 `PREFLIGHT` 错误（`src/core/planning.ts:208`），向上经 `propose` → `deliberate` 捕获后写入 `failed` 状态并抛出（`src/cli/index.ts:559-568`），**无任何 CLI/模型/思考等级替换逻辑**。
- 组局方案解析失败仅以同一组局器重试（最多 2 次），不降级或替换推理设置（`src/core/planning.ts:159-180`）。

### 要求 8：mad init 只探测、生成骨架、不猜测、不覆盖
**判定：符合**

- `initialize` 仅探测 PATH 中已安装可执行文件并执行版本探测（`src/cli/index.ts:62-74`），记录可执行真实路径（`findExecutable` 返回 `realpath` `src/cli/index.ts:95-104`）。
- `buildConfigTemplate` 生成骨架，默认组局器与 model 均为占位符（`REPLACE_WITH_CLI_ID` / `REPLACE_WITH_PRESET_ID` / `REPLACE_WITH_MODEL_ID`），不猜测模型、思考等级或默认组局器（`src/adapters/config.ts:200-221`）；并按适配器写入可选推理等级说明注释（`src/adapters/config.ts:208-215`）。
- 已有配置默认不覆盖：写入使用 `flag: force ? "w" : "wx"`，`EEXIST` 时提示用 `--force` 重建（`src/cli/index.ts:75-85`）；`--force` 由命令解析显式传入（`src/cli/index.ts:760-763`）。
- 配置权限为 `0o600`（`src/cli/index.ts:76-86`）。

### 要求 9：config validate 静态 / config check 实际预检
**判定：符合**

- `validateConfig` 在 `check=false` 时仅 `loadCliRegistry`（静态解析校验）并解析默认组局器，不发起任何进程（`src/cli/index.ts:106-126`，静态分支 `src/cli/index.ts:124-125`）。
- `check=true` 时对每个 cli/preset 组合调用 `adapter.check` 做运行时实际预检（`src/cli/index.ts:109-123`）。
- 命令路由正确：`config validate` / `config check` 分别映射到 `validateConfig(false/true)`（`src/cli/index.ts:765-768`）。

## 4. 偏差与风险清单

| 编号 | 对应章节 | 问题描述 | 严重程度 | 影响 | 建议 |
|------|----------|----------|----------|------|------|
| R1 | 第4节 要求5 | 适配器的类型化配置 schema 未下放到各适配器文件，而是集中定义在 `src/adapters/config.ts`（白名单于 `config.ts:82-86`）。功能满足「按适配器限定可配置项 + 禁止覆盖安全参数」，但代码组织未严格满足「每个适配器定义自己的类型化配置 schema」。 | 低 | 新增/修改适配器时必须同步改动 `config.ts` 的白名单，扩展点不直观，易遗漏。 | 将各适配器的 `options` 白名单与枚举声明下放到对应适配器模块（如 `codex.ts`/`generic.ts` 各自导出 schema），由 `config.ts` 聚合，或至少在 `config.ts` 中以显式映射表注明每个 adapter 的扩展位置。 |
| R2 | 第4节 要求5/6 | `reasonix` 适配器未在命令中设置只读/沙箱/plan 等安全标志（`src/adapters/generic.ts:22`），其项目模式安全完全依赖 `supportsProjectReadOnly = cli.adapter !== "reasonix"`（即 false，`src/adapters/generic.ts:35`）并在预检阶段显式拒绝（`src/core/planning.ts:193-195`）。非项目模式下其调用行为取决于用户侧 CLI 自身配置。 | 低 | 当前通过「禁止项目模式」已规避风险，但 reasonix 在非项目自由讨论中不受本应用只读约束，依赖用户环境。 | 在文档/模板中说明 reasonix 的安全边界；后续若该 CLI 提供只读开关，应在适配器中显式启用。 |
| R3 | 第4节 要求6/9 | `mad config check` 运行时预检以 `process.cwd()` 为工作目录（`src/cli/index.ts:113`），未针对特定项目工作目录。项目模式必需的只读能力验证发生在 `deliberate`/`resume` 实际传入 workspace 时（`src/core/planning.ts:193-195`）。 | 低 | 属合理分层（config check 为通用就绪性检查），但用户可能误以为 config check 已覆盖项目只读预检。 | 在 `mad config check` 输出或文档中说明其使用当前目录；项目只读约束由审议时 `--workspace` 触发预检。 |

## 5. 结论与优先建议

整体结论：当前 CLI 适配器与注册表实现**高度符合**目标架构第 4 节要求，9 项要求中 8 项判定为「符合」，1 项（要求 5）判定为「部分符合」。配置边界清晰、方案仅引用 cli/preset、无原始参数透传、两层验证与失败即阻断等核心约束均已落地；`mad init` / `mad config validate` / `mad config check` 行为与文档一致。

优先建议（按优先级）：

1. **（低优先级，R1）** 将适配器配置 schema 下沉到各适配器文件，明确扩展点，降低新增适配器时遗漏白名单的风险。
2. **（低优先级，R2）** 补充 reasonix 安全边界说明，明确其仅支持非项目模式。
3. **（低优先级，R3）** 在 `mad config check` 提示中区分「通用就绪性预检」与「项目只读预检（审议时按 `--workspace` 触发）」，避免用户误解。

上述偏差均为低风险、非功能性偏离，不影响第 4 节安全与约束目标的达成。
