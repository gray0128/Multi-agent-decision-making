# TypeScript CLI 与审议观察页目标架构

> 状态：已确认的目标设计，尚未全部实现。当前 Python MVP 的使用方式见项目 README，历史设计见 [MVP 设计](MVP设计.md)。迁移期间两套实现可以并存，但配置与档案互不兼容且不双写。

## 1. 目标与约束

项目改造为以 TypeScript CLI 为核心的本地多 Agent 审议工具：CLI 调用用户已安装并认证的 AI CLI，执行结构化审议或自由讨论，保存可审计、可恢复的透明档案；独立的本地观察页展示历史和实时过程，并只允许响应当前检查点。

目标架构移除 Python、Microsoft Agent Framework 和 Agent Framework DevUI。首版要求 Node.js，以单个 npm 包分发；独立可执行程序只在新架构验证可行后再考虑。

关键约束：

- TypeScript 审议进程是运行状态和正式审议档案的唯一写入者。
- 每个应用数据根目录同时最多有一个活动审议，历史档案可以并行浏览。
- 模型调用均通过一次性、非交互 CLI 子进程完成，不依赖 CLI 私有会话。
- 项目审议只读访问用户显式指定的原始工作目录，不创建材料快照，也不隐式使用当前目录。
- 页面不能发起、恢复、删除审议或修改 CLI 注册表，只能响应当前检查点。
- 本地服务只绑定 `127.0.0.1`，不提供局域网或公网监听模式。
- Python 版配置、档案和未完成审议不迁移；旧数据不自动删除。

## 3. 单包代码结构

首版使用单个 `package.json`，不建立 monorepo：

```text
src/
├── cli/          命令解析、终端检查点、stdout/stderr 契约
├── core/         组局、结构化审议、自由讨论、报告流水线
├── adapters/     CLI 适配器与类型化配置校验
├── archive/      透明档案、事件、锁与恢复
├── server/       本地 HTTP、认证、SSE 和文件信箱
└── web/          静态 HTML/CSS 与少量原生 TypeScript
```

CLI、服务端和页面共享同一套 schema 与领域类型，页面资源随 npm 包发布。

## 4. CLI 注册表与模型调用预设

长期配置文件为 `config/clis.toml`。它只保存可信 CLI 执行边界，不预设长期 Agent 或角色。

```toml
[defaults.generator]
cli = "codex"
preset = "gpt-x-deep"

[[clis]]
id = "codex"
adapter = "codex"
executable = "codex"
timeout_seconds = 300
max_concurrency = 1

[[clis.presets]]
id = "gpt-x-deep"
model = "gpt-x"
context_budget = 64000

[clis.presets.options]
reasoning_effort = "high"
```

模型调用预设是模型与思考等级等推理设置的可信组合。同一模型可以拥有多个预设。审议方案只能引用 `CLI 配置 + 调用预设`，不能携带裸模型名、任意推理设置、可执行路径或 CLI 参数。

不支持 `extra_args` 等原始参数透传。每个适配器定义自己的类型化配置 schema，并禁止配置覆盖只读、审批、工具限制、输出格式和工作目录等安全参数。

验证分两层：

1. 加载配置时，适配器静态验证字段、类型、枚举和参数组合。
2. 运行时先实际预检组局器，方案生成后对每个不同的 CLI 与调用预设组合预检一次；共享组合的多个审议 Agent 不重复预检。

任何验证或预检失败都阻止审议启动，不自动替换 CLI、模型或思考等级。

`mad init` 只探测已安装 CLI 和可执行路径，生成带适配器选项说明的配置骨架，不猜测模型、思考等级或默认组局器。已有配置默认不覆盖；用户显式要求时才重建模板。配置通过 `mad config validate` 静态验证，通过 `mad config check` 实际预检。

## 5. 固定组局阶段

每次审议在正式交流前都执行组局阶段：

1. 使用默认组局器，或用户按次覆盖的允许组合。
2. 向组局器提供问题、单次资源上限和 CLI 注册表安全视图；项目审议还允许其直接只读查看显式工作目录。
3. 组局器生成只属于本次审议的审议 Agent。每个实例包含唯一 ID、CLI 配置、调用预设和角色描述。
4. 方案指定报告 Agent；自由讨论还指定主持 Agent。两者都必须是参与者，可以由同一实例兼任；三名以上参与者时优先分离但不硬性要求。
5. 对方案引用的唯一调用组合执行运行时预检。
6. 方案确认后才进入正式审议。

交互式确认支持确认、修改、附带指导重新组局和取消。修改可以增删审议 Agent、改变角色或调用预设、改选主持与报告 Agent，但不能突破白名单和资源上限；新增组合重新预检，修改后的完整方案再次确认。

非交互调用必须显式传入 `--auto-confirm-plan`，且只能自动接受第一次生成并通过所有校验的方案，不能自动修改或重新组局。默认组局器无效、不可用或预检失败时直接报错，不自动降级。

同一 CLI 和调用预设可以生成多个不同角色的审议 Agent，并正常计入参与者数量与争议信号；页面与报告必须保留其共享来源，不得把一致意见描述为独立模型交叉验证。

## 6. CLI 与交互模式

```bash
mad deliberate "问题" --mode structured
mad deliberate "问题" --mode free
```

- 默认审议模式为 `structured`。
- 两种模式共享组局、预检、确认、工作目录、输出与恢复参数。
- `mad resume <id>` 从档案恢复原模式、交互策略、方案和预算，不允许切换。
- 默认交互策略为 guided；显式 `--auto` 才跳过正常检查点。
- 自动模式仍需独立的 `--auto-confirm-plan` 才能接受组局方案。
- guided 模式既无交互终端也无在线观察服务时立即失败，不无限等待。

机器调用示例：

```bash
mad deliberate "问题" \
  --mode structured \
  --auto \
  --auto-confirm-plan \
  --format json
```

## 7. 结构化审议

结构化审议保留固定阶段与屏障：

1. 独立陈述；
2. 质疑与补充；
3. 修订意见与关键争议信号；
4. 按策略执行一次争议收敛；
5. 报告 Agent 生成草稿；
6. 其他参与者并行审阅；
7. 报告 Agent 最终修订。

同一参与者阶段保持逻辑并行，但实际 CLI 进程同时受全局限流器和 CLI 配置级限流器约束。同一阶段的参与者看不到本阶段尚未完成的其他输出。guided 模式在独立陈述、质疑补充、争议判定和报告草稿后等待检查点。

## 8. 自由讨论

自由讨论不设陈述、质疑、修订等预定义语义阶段，每个发言回合只调用一名参与者。

持续运行的是 TypeScript `DiscussionController`，不是模型进程。Controller 持有权威状态、硬限制和恢复边界。主持 Agent 不使用长期 CLI 进程或私有会话，而是在覆盖周期开始和每个检查窗口边界通过一次性 CLI 调用规划后续发言，并在窗口边界评估收敛。

主持调度调用计入总调用次数与成本，但不算发言回合，也不作为参与者观点。

自由讨论流程：

1. 先执行覆盖周期：主持 Agent 决定顺序，每位参与者恰好发言一次。
2. 随后进入开放讨论：允许重复选择参与者，但同一参与者不能连续发言。
3. 每完成与参与者数量相同的发言回合，就结束一个检查窗口。
4. 检查窗口不要求覆盖所有参与者。
5. 窗口边界执行收敛评估，并在 guided 模式下允许用户继续、补充指导、结束讨论、暂停或取消。

主持 Agent 按窗口规划而非每回合重新调用。首版不维护长期主持会话；未来即使某个适配器支持持久会话，也只能作为优化，不能改变档案作为权威状态的原则。

自由讨论结束后复用结构化审议的共同成果流水线：报告 Agent 生成草稿，其他参与者并行审阅，报告 Agent 完成一次最终修订。报告明确区分共识、未决争议、假设和风险，不把主持调度内容当作参与者观点。

## 9. 资源、并发与上下文

参与者数量、自由讨论检查窗口、总调用次数、超时和上下文预算同时受三层约束：应用内置的保守默认值、用户按次覆盖值、普通命令不能突破的安全最大值。组局器可以看到但不能提高限制。具体默认数值在 Codex 端到端验证后校准。

并发由全局限流器与 CLI 配置级限流器共同控制。CLI 配置首版默认并发为 1，同一 CLI 下的全部调用预设和审议 Agent 共享限流器。最终预算与并发设置写入档案，恢复时不改变。

每个模型调用预设声明上下文预算。Controller 在调用前估算实际输入；任一后续调用预算不足时，由报告 Agent 生成统一滚动摘要。所有参与者、主持 Agent 和报告 Agent 使用同一摘要，加上摘要之后的最近发言，不按模型生成不同版本。摘要生成计入预算且可独立恢复，完整原始记录始终保留。

## 10. 检查点动作

- **继续**：进入下一阶段或检查窗口。
- **指导后继续**：记录用户指导并继续。
- **结束讨论**：仅自由讨论可用；停止新增发言并进入报告流水线。
- **暂停审议**：持久化可恢复状态并退出活动进程。
- **取消审议**：进入不可恢复终态，不生成共同成果，但保留档案。

第一次 `Ctrl-C` 按暂停处理，并终止当前 CLI 子进程。终端与观察页的第一份有效检查点响应获胜；过期、重复或 ID 不匹配的响应被拒绝。

## 11. 观察服务与页面

`mad serve` 是独立、长期运行的本地进程。审议不依赖它存在：服务未启动时，guided 模式使用终端检查点，自动模式按策略继续；服务重启不应中断审议。

审议进程与观察服务通过 `MAD_HOME/runtime` 下仅限当前用户访问的文件信箱通信：审议进程原子发布待处理检查点；观察服务只能原子写入一次性响应文件；审议进程校验并消费响应，再独占更新正式状态；观察服务不能直接修改审议档案。

页面首版使用静态 HTML/CSS 和少量原生 TypeScript，不引入前端框架、路由框架或全局状态库。Markdown 必须经过独立解析与净化，禁止直接注入未净化 HTML。

流式 `fetch` 只展示阶段、回合、调用开始与完成、耗时、警告和检查点等状态事件。参与者正文只有在调用完成并提交权威记录后整体显示；不做 token 级文本流，不展示 thinking/reasoning，也不把原始 stdout/stderr 发送给浏览器。

## 12. 本地认证

`mad serve` 每次启动生成新的 Bearer Token，通过自动打开或打印的 URL fragment 交给页面。页面读取后立即从地址栏移除，并只保存在 `sessionStorage`。

所有档案 API、流式读取和检查点响应都使用 `Authorization: Bearer ...`。Token 不进入查询参数、长期 Cookie 或磁盘文件。服务始终固定监听 `127.0.0.1`，认证不构成开放远程监听的理由。

## 13. 透明档案

每次审议保存为独立目录：

```text
manifest.json      身份、模式、参与者、方案和 schema_version
state.json         原子替换的可恢复状态
events.jsonl       只追加生命周期事件
transcript.jsonl   只追加权威发言记录
diagnostics.jsonl  只追加脱敏调用诊断
report.md          最终共同成果
```

观察服务读取这些文件提供历史和实时视图。首版不引入数据库；只有跨审议检索或并发需求成立后才评估 SQLite。

## 14. 恢复与失败

恢复以逻辑调用为最小边界。每次参与者发言、主持规划、摘要或报告调用在启动 CLI 前先持久化冻结输入、逻辑调用 ID 和尝试信息，成功返回后原子提交规范化输出。

- 结构化并行阶段保留已完成参与者结果，只重跑失败或未完成调用。
- 自由讨论逐回合提交。
- 瞬时错误和 schema 输出错误各自动重试一次；再次失败则暂停为可恢复状态。
- 不自动移除参与者、降低最低人数、替换 CLI 或调用预设。
- CLI 已完成但输出尚未提交时发生崩溃，恢复可能产生重复计费；权威记录仍只能提交一次。
- 每次尝试使用独立尝试 ID，并在脱敏诊断中关联到同一逻辑调用。
- 恢复不重新组局、不切换模式、交互策略、角色、预设或预算。

同一 `MAD_HOME` 使用原子全局锁限制单个活动审议。

## 15. 工作目录与安全

纯文本审议不提供工作目录。项目审议必须显式传入 `--workspace <path>`，该参数本身就是对所选 CLI 读取该目录的授权；应用不再提供 `--direct-workspace` 或额外确认。

组局器与所有参与者从组局阶段开始共同直接只读访问规范化后的原始目录。应用向 stderr 输出风险提示，并把绝对路径与读取模式写入档案。

CLI 适配器必须启用并验证对应运行时的只读、计划或禁用写工具模式。不满足最低只读能力的适配器或预设不能用于项目审议。

直接只读不限制 CLI 可以看到哪些文件。应用不再通过快照排除秘密、依赖目录或构建产物，用户应把显式工作目录视为完整读取授权。

## 16. stdout、stderr 与退出码

- 默认成功时，stdout 只包含最终 Markdown 报告。
- `--format json` 时，stdout 只包含一个完整 JSON 对象。
- 进度、警告、Token 地址和档案路径进入 stderr。
- 非 TTY 输出不含 ANSI 控制字符。
- 失败时不向 stdout 写半截 JSON。
- 暂停、取消、配置错误、预检失败和执行失败使用不同退出码。
- JSON 至少包含审议 ID、状态、模式、报告、参与者、预算使用、警告和档案路径。

## 17. 迁移与接管

迁移期间 Python 与 TypeScript 实现可以并存，但使用不同配置和档案，不做导入、兼容或双写。

实施顺序：

1. 建立单包 TypeScript 骨架、schema、透明档案和 CLI 基础。
2. 使用 Codex 适配器完成端到端纵向验证，并校准资源默认值。
3. 完成结构化审议等价实现。
4. 完成自由讨论、主持窗口规划和共享报告流水线。
5. 完成本地观察服务、认证、历史页面、事件流和检查点文件信箱。
6. 迁移 Claude Code、Reasonix、Grok、Pi、CodeBuddy 和 `agy`。
7. 完成纯文本、项目只读、恢复、取消、机器输出和安全测试。
8. 更新用户文档，执行整体接管。

只有上述能力全部通过验收后，才一次性删除 Python 源码、`pyproject.toml`、Agent Framework 和 DevUI。旧配置与档案不自动删除。

## 18. 暂缓决定

以下内容等待 Codex 纵向验证结果，不在设计阶段猜测：

- 默认与安全最大的参与者数量；
- 自由讨论默认检查窗口数；
- 默认总调用预算；
- 全局实际并发默认值；
- 是否以及如何发布不依赖 Node.js 的独立可执行程序。
