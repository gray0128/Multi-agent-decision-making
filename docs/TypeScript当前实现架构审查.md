## 1. 结论

原始审查时，TypeScript 实现已经覆盖目标架构的大部分主流程，但尚不能认定为完全符合目标架构。下述 19 项结论现已全部修复；详细章节保留原始问题、触发条件和修复依据，作为后续回归验收清单。

本次审查确认 19 项实质问题：

- P1：11 项，主要集中在并发锁、恢复一致性、检查点原子性、上下文预算和报告证据链。
- P2：8 项，主要集中在观察服务容错、自由讨论边界、错误分类、并发配置归档和工作目录边界。

原始审查中最需要优先处理的是：

1. 并行调用失败后仍有子进程运行，但全局活动锁已经释放。
2. stale lock 回收可能让两个恢复进程同时持有锁。
3. 权威结果提交后的附属 I/O 失败可能让运行结果与档案结果分叉。
4. 检查点响应和已接受动作缺乏可靠的原子提交与恢复边界。
5. 上下文超限时，摘要调用本身仍会收到超限输入。

## 2. 审查范围与方法

审查范围包括：

- `src/cli/`：命令参数、组局确认、恢复、检查点、stdout/stderr 和退出码。
- `src/core/`：组局、结构化审议、自由讨论、上下文管理、成果流水线和调用调度。

- `tests-ts/`：现有测试覆盖及关键缺口。

原始审查以 `16a4f8c` 为准，通过 CodeGraph 定位核心符号和调用链，并结合源码、ADR、目标架构及自动测试结果进行验证。随后按本报告第 7 节顺序完成实现和回归测试修复。

严重级别定义：

- **P0**：阻断发布或造成不可接受的数据/安全后果，必须立即修复。
- **P1**：高优先级正确性、安全性或恢复问题，应在宣称架构完成前修复。
- **P2**：正常优先级的行为偏差、可靠性或可维护性问题。
- **P3**：低风险问题或改进建议。

## 3. 工程质量问题

### 3.1 [P1] 并行阶段失败后过早释放全局锁

**位置：**

- `src/core/structured.ts:159`
- `src/core/outcome.ts:56`
- `src/cli/index.ts:472-474`

结构化并行阶段和报告审阅阶段使用 `Promise.all`。其中一个调用失败时，`Promise.all` 会立即拒绝，但其他同组 CLI 子进程不会被取消，也不会被等待。Controller 随后把状态设为 `paused`，CLI 的 `finally` 立即释放 `active.lock`。

如果用户此时执行 `mad resume`，新进程可以取得锁，而旧进程中未结束的调用仍可能提交结果。两个 `ArchiveStore` 实例拥有彼此独立的内存队列，可能并发覆盖 `state.json`、重复计费，或者在终态之后继续追加档案。

现有端到端恢复测试只覆盖“原进程完全退出后恢复”，没有覆盖“快速失败调用 + 慢速同阶段调用 + 立即恢复”的竞态。

### 3.2 [P1] stale lock 回收不是原子的

**位置：**

- `src/archive/store.ts:216-249`
- `src/archive/store.ts:231-235`

锁回收采用“读取锁文件、检查 PID、删除锁文件、重新创建”的多步流程。两个恢复进程可以同时读取到相同的死亡 PID：进程 A 删除旧锁并创建新锁后，进程 B 仍可能删除 A 的新锁并成功创建自己的锁，最终两个进程都认为自己持有全局锁。

此外，`release()` 无条件删除当前路径，没有验证磁盘上的锁是否仍属于本实例，也可能删除其他进程后来创建的锁。

现有测试只覆盖单个回收者，没有覆盖两个回收者同时竞争死亡锁的情况。

### 3.3 [P1] `mad resume` 的 ID 可穿越档案根目录

**位置：**

- `src/cli/index.ts:485-493`
- `src/archive/store.ts:36-41`

`resume` 只检查 ID 非空，随后直接使用 `join(deliberationsRoot, id)`。传入 `../../target` 等值可以让路径逃逸 `MAD_HOME/deliberations`。

如果目标目录包含形状可接受的 `manifest.json` 和 `state.json`，后续流程可能在档案根目录之外覆盖状态、写入报告、追加事件或执行档案中记录的 CLI 配置。观察服务已经使用 ID 正则限制此类输入，但 CLI 没有复用同一约束。

### 3.4 [P1] 子进程 stdout/stderr 无上限缓存

**位置：**

- `src/adapters/process.ts:25-26`
- `src/adapters/process.ts:48-49`
- `src/adapters/process.ts:60-61`

`runProcess` 将全部 stdout 和 stderr chunk 保留在内存中，直到进程退出后再统一拼接。配置允许单次调用最长运行 1800 秒，异常或行为失控的 CLI 可以持续输出并耗尽主进程内存。

一旦主进程被操作系统杀死，就无法执行正常的暂停、诊断写入和锁释放流程。当前测试夹具只产生短输出，没有输出洪泛或容量上限测试。

### 3.5 [P2] 一份损坏档案会让整个历史列表不可用

**位置：**

- `src/archive/store.ts:43-61`
- `src/server/observer.ts:100-110`

档案目录和各个文件分多步创建，创建过程中崩溃可能留下缺少 `state.json`、JSONL 文件或内容不完整的目录。观察服务使用 `Promise.all` 加载所有档案详情，任何一个目录读取或解析失败都会让整个 `/api/deliberations` 请求返回 500。

因此，一份不完整或损坏档案会阻止用户浏览所有其他健康历史档案。当前观察服务测试只包含一个完整档案。

## 4. 目标架构符合性问题

### 4.1 [P1] 组局阶段没有可恢复的透明档案

**位置：**

- `src/cli/index.ts:387-425`
- `src/core/planning.ts:140-146`
- `src/cli/index.ts:217-219`

全局锁在组局前获取，但正式档案直到组局、方案预检和确认全部完成后才创建。组局器的真实 CLI 调用、重试和失败没有进入透明档案。

组局期间第一次 Ctrl-C 会得到暂停类错误，但此时没有审议 ID 和状态可供恢复；方案确认期间的 Ctrl-C 还会被转换为取消。交互式 `/cancel` 同样不会留下规范要求的取消档案。

现有 Ctrl-C 端到端测试特意等待“审议已创建”后才发送信号，因此没有覆盖规划阶段。

### 4.2 [P1] 检查点响应文件不是原子发布

**位置：**

- `src/server/observer.ts:168-170`
- `src/server/mailbox.ts:57-64`
- `src/server/mailbox.ts:75-82`

观察服务和终端提交路径都直接对最终 `.response.json` 执行 `writeFile(..., { flag: "wx" })`。文件创建后、内容写完前，轮询方可能读到空文件或半截 JSON。

该解析错误不是 `ENOENT`，会直接结束等待并进入清理逻辑，最终让审议暂停。请求文件已经使用临时文件加 rename，响应文件没有满足目标架构所要求的原子一次性写入。

### 4.3 [P1] 检查点阶段和已接受动作没有持久化为恢复边界

**位置：**

- `src/archive/store.ts:7-15`
- `src/core/structured.ts:82-99`
- `src/core/structured.ts:186-197`
- `src/core/discussion.ts:104-138`
- `src/core/discussion.ts:193-204`
- `src/server/mailbox.ts:38-45`

`state.json` 没有记录当前 Controller 阶段、当前检查点 ID 或已经接受的检查点动作。每次 `run()` 都从头重建流程，依靠已完成逻辑调用跳过模型执行。

如果进程在读取响应后、进入下一阶段前崩溃，恢复会再次发布同一语义检查点并允许重新作答；此前已经接受的 `end`、`continue` 或 `cancel` 不再具有权威性。新一轮 `wait()` 还会先删除旧响应，可能抹掉崩溃期间由观察页提交的第一份有效响应。

这与“第一份有效检查点响应获胜”和“审议进程消费响应后独占更新正式状态”的约束不符。

### 4.4 [P1] 权威结果提交后重试可能造成运行结果与档案分叉

**位置：**

- `src/core/execution.ts:127-160`
- `src/archive/store.ts:106-125`

一次模型调用成功后，流程依次执行：提交 `completedInvocations`、写诊断、写 transcript。后两项或 `invocation.committed` 事件写入失败时，异常会落入统一重试分支。

第二次调用可能得到不同输出。此时 `commitInvocation` 因第一份结果已存在而返回 `false`，但 `InvocationRunner` 仍把第二份未提交结果返回给 Controller。后续报告可能基于第二份输出，而恢复和观察页读取的是第一份权威输出，形成不可接受的运行/档案分叉。

### 4.5 [P1] 上下文超限时，摘要调用本身仍然超限

**位置：**

- `src/core/context.ts:48-69`
- `src/core/outcome.ts:45-75`

`SharedContextManager` 在发现当前上下文超过阈值后，仍将完整的超限上下文、问题和摘要指令一起发送给报告 Agent。实现只限制摘要输出长度，没有保证摘要调用的输入能放入上下文窗口。

此外，当前估算只计算共享记录，不包含问题、角色、固定提示和用户指导；成果草稿、审阅与最终修订也完全绕过上下文管理器。

真实 CLI 如果因 context length 拒绝摘要调用，逻辑调用会冻结相同的超长输入，恢复后仍会重复失败，无法通过摘要自救。现有测试使用不会执行上下文限制的 fake adapter，因此只证明“触发过摘要调用”。

### 4.6 [P1] 报告审阅者和最终修订者看不到权威证据

**位置：**

- `src/core/outcome.ts:45-52`
- `src/core/outcome.ts:55-65`
- `src/core/outcome.ts:68-75`

只有报告草稿调用收到完整 `evidence`。其他参与者的审阅调用只收到问题和草稿，最终修订调用只收到草稿和审阅意见。

由于所有 CLI 都是一次性、无私有会话的进程，审阅者无法把草稿与各参与者的原始观点进行比较。如果草稿漏报、歪曲或虚构共识，审阅和最终修订阶段缺少发现与纠正问题所需的材料，不能真正实现目标架构要求的共同成果审阅。

### 4.7 [P1] 报告草稿检查点的用户指导不会影响最终结果

**位置：**

- `src/core/structured.ts:135-141`
- `src/core/outcome.ts:38-75`

进入 `OutcomePipeline.run()` 时，Controller 已经把当时的 guidance 渲染成固定字符串。报告 Agent 生成草稿后才执行 `draft` 检查点；在该检查点新增的指导虽然写入档案，但审阅调用和最终修订调用都不会再次读取它。

因此，用户在目标架构明确要求的报告草稿检查点选择“指导后继续”时，指导对剩余流水线没有实际效果。

### 4.8 [P2] guided 用户不能覆盖主持 Agent 的收敛判断

**位置：**

- `src/core/discussion.ts:126-132`
- `src/core/discussion.ts:193-204`

即使 guided 用户在窗口边界选择“继续”或“指导后继续”，只要主持计划标记 `converged: true`，Controller 仍会立即结束讨论。用户指导会被记录，但不会产生新的发言窗口。

这使模型的收敛判断实际上高于用户检查点动作，与“继续进入下一检查窗口”的动作语义不符。

### 4.9 [P2] 最后一个开放讨论窗口没有 guided 检查点

**位置：**

- `src/core/discussion.ts:127-139`

Controller 在执行第 `maxDiscussionWindows` 个窗口并完成收敛评估后，因为循环条件已经不满足而直接进入报告流水线，不再调用 `atBoundary()`。

guided 用户因此无法在最终窗口边界选择继续、补充指导、结束、暂停或取消。现有自由讨论测试在第一个开放窗口后即收敛，没有覆盖窗口上限分支。

### 4.10 [P2] `resume` 在获取全局锁前执行真实预检

**位置：**

- `src/cli/index.ts:494-515`

恢复流程先读取状态并对方案中的 CLI/preset 执行真实预检，之后才获取 `active.lock`，而且 SIGINT handler 也在预检后才安装。

如果同一 `MAD_HOME` 已有活动审议，第二个 `resume` 仍会先启动多个真实模型检查、产生费用并访问工作目录，最后才报告 `LOCKED`。预检期间 Ctrl-C 也不能通过该恢复流程的 AbortController 终止子进程。

锁前读取终态还形成 TOCTOU：另一个恢复进程可能在本进程预检期间把审议完成，本进程随后取得锁并把终态重新改为 `running`。

### 4.11 [P2] 全局并发设置没有写入档案

**位置：**

- `src/core/execution.ts:32-47`
- `src/core/types.ts:29-35`
- `src/core/types.ts:66-76`

`InvocationScheduler` 将全局并发默认值硬编码为 6，`ResourceLimits` 和 manifest 中没有该字段。恢复时会创建新的 Scheduler 并使用当前代码版本的默认值。

如果后续版本调整默认并发，旧审议恢复时无法保持原始并发设置，违反“最终预算与并发设置写入档案，恢复时不改变”的约束。

### 4.12 [P2] 重试策略没有区分可重试错误

**位置：**

- `src/core/execution.ts:106-161`
- `src/adapters/generic.ts:62-77`

除 `PAUSED` 和 `CANCELLED` 外，所有错误都会自动重试，包括认证失败、权限错误、命令参数错误、模型不可用和其他确定性失败。

目标架构只要求瞬时错误和 schema 输出错误各自动重试一次。当前行为会对确定性错误重复执行和重复计费，并掩盖应立即暴露的配置或安全问题。

### 4.13 [P2] 观察页没有展示规范要求的实时事件详情

**位置：**

- `src/server/observer.ts:130-139`
- `src/web/index.ts:29-32`
- `src/cli/index.ts:428-431`

服务端 SSE 发送完整事件对象，但页面只渲染 `at` 和 `type`，丢弃阶段、Agent、逻辑调用、耗时等数据。workspace 风险和共享来源警告只进入 stderr 或最终 JSON，也没有作为档案事件写入。

因此页面不能按目标架构展示阶段、回合、调用开始与完成、耗时、警告和检查点等实时状态。

### 4.14 [P2] 项目预检的 probe 使用隐式当前目录

**位置：**

- `src/core/planning.ts:183-186`
- `src/adapters/codex.ts:13-18`
- `src/adapters/generic.ts:38-42`

`OrganizerService` 把显式 workspace 传给 `adapter.check()`，但各 adapter 的 `probe()` 仍固定使用 `process.cwd()`。某些 CLI 会在启动时加载当前目录下的配置或上下文文件，因此 probe 可能读取用户没有通过 `--workspace` 明确授权的目录。

这与纯文本审议不隐式使用当前目录、项目审议只访问显式工作目录的边界不一致。

## 5. 已确认符合的关键约束

以下部分在本次审查中没有发现可复现的架构偏差：

- 单 npm 包和 TypeScript 目录结构已经建立，Python 实现已移除。
- CLI 注册表禁止方案携带裸模型、任意 CLI 参数和可执行路径。
- 方案参与者、报告 Agent 和自由讨论主持 Agent 的基本校验完整。
- 结构化审议的独立陈述、质疑、修订、收敛、草稿、审阅和最终修订阶段已经实现。
- 同一参与者阶段采用逻辑并行，并通过 CLI ID 共享配置级限流器。
- 自由讨论覆盖周期保证每位参与者恰好发言一次，开放窗口禁止同一参与者连续发言。
- 本地观察服务固定监听 `127.0.0.1`。
- 档案 API、SSE 和检查点响应均要求 Bearer Token。
- Token 通过 URL fragment 传递，页面读取后从地址栏移除并只保存在 `sessionStorage`。
- 页面 Markdown 在插入 HTML 前进行字符转义，没有直接注入原始 HTML。
- 项目模式会阻止未声明只读能力的适配器。
- 默认成功输出、JSON 输出和错误输出的 stdout/stderr 分离符合基本契约。

1. 并行调用一快失败、一慢运行时，锁不得提前释放。
2. 两个进程同时回收 stale lock 时只能有一个成功。

## 7. 建议修复顺序

建议按以下顺序修复，以减少后续改动建立在不可靠恢复基础上的风险：

1. **锁与唯一写入者：**并行失败收束、stale lock 原子所有权、resume 锁顺序和 ID 边界。
2. **档案事务与恢复：**规划期档案、权威结果提交、检查点状态机和原子响应文件。
3. **上下文与成果证据：**统一完整输入预算、可执行摘要、审阅证据和草稿指导传递。
4. **自由讨论动作语义：**用户覆盖主持收敛、最终窗口检查点和可恢复动作。
5. **进程与服务健壮性：**输出容量、损坏档案隔离、错误重试分类和 SSE 展示。
6. **配置可恢复性：**全局并发归档和 workspace probe 边界。

完成上述修复后，应重新进行一次目标架构逐条验收，而不只依赖现有测试集。
