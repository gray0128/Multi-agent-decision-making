# 审议展示页面 UI/UX 优化建议

## 1. 背景

本建议基于最近一次议题展示页的实际使用体验，范围仅限前端信息架构、视觉层级和交互体验。

当前页面能够完整展示档案、参与 Agent、逐次输出、共同成果和生命周期事件，但长内容阅读成本较高：正文与运行信息混在同一视觉层级中，字体和标题偏大，复杂表格较难扫读，长议题缺少快速定位能力。

页面还需要同时适应两类不同体验：

- 结构化审议：可能存在方案、检查点、审核和最终报告。
- 自由讨论：重点是自然、连续的多方交流，可能没有正式报告，也不应被包装成一板一眼的主持流程。

因此，展示层应以已有档案内容为准，自适应呈现，而不是预设每个议题都必须有报告或固定主持结构。

## 2. 范围与非目标

### 2.1 本轮范围

- 调整页面布局、字体、间距和视觉层级。
- 优化 Markdown 正文、表格、代码块和行内代码的渲染。
- 增加时间线锚点导航和当前位置反馈。
- 区分 Agent 正文、调用元数据、程序化流程输出和生命周期事件。
- 改善长内容下的扫描、跳转、折叠和返回体验。
- 同时适配结构化审议与自由讨论，不依赖最终报告存在。

### 2.2 本轮非目标

- 不调整模型调用次数、提示词、上下文或输出协议。
- 不要求 Agent 额外生成摘要、标签或结构化字段。
- 不改变主持 Agent 的调度方式，也不要求所有讨论由主持人按固定议程推进。
- 不要求所有议题生成正式报告。
- 不对 Agent 自然语言内容做不可靠的语义分类。
- 不修改审议执行、归档和恢复流程。

## 3. 设计原则

### 3.1 模式中立

页面按“实际有什么就展示什么”的原则组织内容：

- 有检查点时显示检查点。
- 有共同成果或报告时显示成果区。
- 没有报告时不显示“尚未生成最终报告”作为主要内容，也不留下空白占位。
- 自由讨论以 Agent 对话时间线为主体，不强行套用主持会议纪要结构。
- 结构化审议可以呈现更多阶段、审核和成果信息，但这些信息默认不压过正文。

### 3.2 正文优先

用户首先需要阅读 Agent 说了什么，其次才是调用了哪个 CLI、属于哪个内部阶段、耗时多少、产生了什么生命周期事件。

视觉权重建议遵循：

> Agent 正文 > 议题和当前检查点 > 可选成果 > Agent 身份 > 调用与流程元数据 > 生命周期事件

### 3.3 渐进披露

首屏只展示理解当前议题所需的信息。运行细节、流程事件、长代码和历史元数据通过折叠或弱化展示，需要时再展开。

### 3.4 只依赖现有数据

当前逐次输出已经包含 `id`、`at`、`stage`、`agentId`、`invocation` 和 `content`；事件包含 `id`、`at`、`type` 和 `data`。时间线、输出分层和弱化展示均可由前端利用这些已有字段完成，无需模型改变输出。

## 4. 建议的信息架构

### 4.1 桌面端布局

建议将当前两栏结构扩展为“档案索引 + 阅读正文 + 时间线导航”的三段布局：

```text
┌──────────────┬──────────────────────────────────┬──────────────────┐
│ 档案索引     │ 议题标题与状态                   │ 本议题时间线     │
│              │ 参与 Agent                       │                  │
│ 可折叠       │ 当前检查点（存在时）             │ 09:30 议题创建   │
│              │                                  │ 09:31 组局       │
│ 历史议题     │ Agent 对话 / 审议内容            │ 09:35 Agent A    │
│              │                                  │ 09:37 Agent B    │
│              │ 可选成果（存在时）               │ 09:48 讨论结束   │
│              │ 弱化的流程记录与事件             │ 09:53 成果       │
└──────────────┴──────────────────────────────────┴──────────────────┘
```

建议尺寸：

- 档案索引：默认 280px，可折叠为 44px。
- 正文区域：宽度随窗口变化，正文阅读宽度控制在 760-920px。
- 时间线：200-240px，吸顶显示。
- 超宽屏将正文与时间线整体居中，避免所有内容挤在左侧、右侧大面积空白。

### 4.2 内容顺序

每个议题建议按以下顺序展示：

1. 议题标题、状态、模式、创建时间和调用量。
2. 当前检查点，仅在存在时显示。
3. 参与 Agent，默认使用紧凑列表，不占用大块首屏。
4. Agent 输出时间线，作为所有模式的主内容。
5. 共同成果或报告，仅在实际存在时显示。
6. 生命周期事件，默认折叠为“流程记录”。

这个顺序不要求报告存在，也不把自由讨论变成正式会议流程。

### 4.3 移动端与窄屏

- 档案索引改为抽屉。
- 时间线改为右下角“定位”浮动按钮，点击后打开底部抽屉。
- 正文保持单栏。
- 表格允许横向滚动，页面本身不得出现横向滚动。
- Agent 元数据折叠到卡片标题下方。

## 5. 字体与阅读密度

当前正文、标题和表格整体偏大，长议题中会显著增加滚动距离。建议默认采用更紧凑但仍易读的密度。

| 元素 | 建议字号 | 建议行高 | 说明 |
|---|---:|---:|---|
| 页面主标题 | 30-34px | 1.15 | 仅议题标题使用 |
| 内容一级标题 | 24px | 1.3 | Markdown `h1` |
| 内容二级标题 | 20px | 1.4 | Markdown `h2` |
| 内容三级标题 | 17px | 1.45 | Markdown `h3` |
| Agent 正文 | 15px | 1.65 | 长文本主要阅读字号 |
| 表格正文 | 13px | 1.45 | 降低表格高度 |
| 代码块 | 12.5-13px | 1.55 | 等宽字体 |
| Agent 元数据 | 11-12px | 1.4 | 弱化显示 |
| 流程事件 | 11px | 1.4 | 默认次要信息 |

其他建议：

- 正文段落间距控制在 `0.65em-0.9em`，避免每段产生过多垂直空白。
- Markdown 标题上间距按层级控制，不使用过大的统一 `1.5em`。
- 正文最大行宽建议为 70-80 个中文字符。
- 可以增加“紧凑 / 舒适”密度切换，默认使用紧凑模式，并通过 `localStorage` 记忆选择。
- 元数据继续使用等宽字体；Agent 正文可使用更适合中文长文阅读的系统无衬线字体，避免所有内容都呈现为程序输出。

## 6. 表格渲染优化

### 6.1 当前问题

- 表格继承较大的正文字号和行高，单元格高度过高。
- 长句在窄列内频繁换行，难以横向比较。
- 表头、数据行和重点结论的层级不够清晰。
- 宽表格和代码路径容易撑破正文区域。
- 表格与普通正文的视觉边界较弱。

### 6.2 建议样式

- 所有表格外包一层可横向滚动的容器。
- 表格宽度使用 `max-content` 与 `min-width: 100%` 组合，短表格铺满，宽表格滚动。
- 表头吸顶，滚动长表格时始终可见列名。
- 表头使用浅深色差，不使用过重的大标题样式。
- 单元格字号 13px，内边距建议 `7px 10px`。
- 单元格顶部对齐，避免多行内容垂直居中造成错读。
- 使用轻微斑马纹和行悬停高亮，帮助横向追踪。
- 长路径、URL 和代码允许在安全位置换行；普通中文句子保持自然换行。
- 第一列为 ID、角色或维度时，可在桌面端设置为粘性列；移动端取消粘性，避免占用空间。
- 表格下方可显示轻量提示：“横向滚动查看更多”。

### 6.3 重点信息表达

表格中的“共识”“未决”“失败”“风险”等文本，不依赖颜色单独表达，可同时使用文字标签：

- `共识`
- `未决`
- `风险`
- `失败`

标签字号保持 11-12px，避免整行使用高饱和背景色。

## 7. 时间线锚点导航

### 7.1 目标

时间线用于快速定位“谁在什么时候输出了什么”以及“流程走到哪里”，不承担总结或重新组织模型内容的职责。

### 7.2 锚点来源

锚点完全由现有档案数据生成：

- 议题创建时间：`manifest.createdAt`。
- Agent 输出：逐次输出的 `id`、`at`、`agentId` 和 `stage`。
- 检查点：当前检查点及已有检查点事件。
- 共同成果：报告实际存在时生成一个锚点。
- 生命周期事件：事件的 `id`、`at` 和 `type`。

不要求模型生成标题或摘要。

### 7.3 时间线结构

时间线建议提供两级视图：

1. 默认视图：只显示议题创建、Agent 正文、检查点和可选成果。
2. “显示流程”视图：补充调用开始、提交、冻结、重试、讨论窗口切换等程序化事件。

建议的条目格式：

```text
09:30  议题创建
09:31  组局流程                     流程
09:35  moderator                    正文
09:36  repo-context                 正文
09:37  docs-archivist               正文
09:48  讨论结束                     流程
09:53  共同成果                     成果
```

### 7.4 交互规则

- 点击条目平滑滚动到对应内容，并保留约 80px 的顶部间距。
- 使用 `IntersectionObserver` 高亮当前阅读位置。
- 页面滚动时，时间线自动跟随当前锚点，但不抢夺用户滚动位置。
- 每个 Agent 输出卡片提供可复制的锚点链接。
- 同一 Agent 连续多次程序化事件可折叠成一组，例如“调用流程 · 3 条”。
- 提供“上一条 / 下一条正文”快捷按钮，跳过流程事件。
- 时间显示优先使用本地化短时间，悬浮时显示完整 ISO 时间。
- 返回页面后，可通过 URL hash 恢复到具体 Agent 输出或成果位置。

### 7.5 长议题降噪

当输出数量较多时：

- 默认只显示 Agent 正文、用户检查点和成果锚点。
- 流程锚点通过“显示流程”开关控制。
- 相同阶段的连续程序事件合并计数。
- 时间线只显示短标签，不复制 Agent 正文标题，避免导航本身变成长文。

## 8. Agent 输出的正文与程序化内容分层

### 8.1 分层依据

前端不猜测自然语言语义，只依据已有结构字段和 Markdown 节点确定层级。

| 内容 | 识别依据 | 默认层级 |
|---|---|---|
| Agent 正文 | `contentHtml`，且阶段属于参与者发言 | 强调、默认展开 |
| Agent 身份 | `agentId` | 卡片标题 |
| 调用来源 | `invocation.cli / preset` | 弱化元数据 |
| 阶段与逻辑调用 | `stage / logicalCallId` | 弱化元数据或折叠详情 |
| 时间 | `at` | 弱化元数据，同时作为锚点 |
| 代码块 | Markdown `pre > code` | 独立技术内容块 |
| 行内代码 | Markdown `code` | 与正文区分但保持高对比度 |
| 生命周期事件 | `event.type / event.data` | “流程记录”区域，默认折叠 |
| 主持调度、覆盖规划等程序输出 | 已知流程型 `stage` | 流程卡，默认收起或紧凑显示 |

### 8.2 Agent 正文卡片

建议每次输出使用统一结构：

```text
┌──────────────────────────────────────────────────────┐
│ repo-context                              09:37:30  │
│ 第 2 次发言 · 自由讨论                              │
├──────────────────────────────────────────────────────┤
│                                                      │
│ Agent 正文内容……                                    │
│                                                      │
├──────────────────────────────────────────────────────┤
│ ▸ 运行信息：discussion_speech · codebuddy/hy3       │
└──────────────────────────────────────────────────────┘
```

视觉规则：

- Agent 名称清晰可见，但不使用与正文标题相同的大字号。
- 时间、阶段、CLI、preset 和逻辑调用 ID 使用小号、低对比度文字。
- 正文使用独立背景或留白，与元数据之间有明确分隔。
- “运行信息”默认收起，展开后查看完整元数据。
- 自由讨论中的普通发言不要显示成系统日志卡片，应更接近连续对话。
- 结构化阶段中的审核、报告草稿和流程判断可以使用同一组件，但通过阶段标签区分。

### 8.3 程序化流程输出

能够由 `stage` 明确识别的主持调度、覆盖顺序、候选方案生成、报告草稿、审核、冻结和调用事件，应采用弱化样式：

- 背景使用比页面底色略深或略浅的中性色。
- 左侧使用细虚线或小型齿轮图标，不使用强调色大边框。
- 默认只显示“阶段名称 + Agent + 时间 + 状态”。
- 原始 JSON、流程理由或完整调用信息放在可展开区域。
- 多条连续流程记录合并为一组。

流程信息仍然可以查看和审计，但不与 Agent 正文争夺视觉注意力。

### 8.4 代码内容

代码是 Agent 正文的一部分，不应一律当作流程日志弱化。建议：

- 代码块保持明显的技术内容容器，字号小于正文。
- 增加语言标签和复制按钮；没有语言信息时只显示“代码”。
- 长代码块默认限制高度，提供“展开全部”。
- 行内代码必须提高文字与背景对比度，避免浅灰字叠加浅灰底。
- 代码块横向滚动，不撑宽正文。
- JSON 等结构化结果若来自流程型阶段，则整个输出卡片弱化；若出现在普通 Agent 正文中，则仍按正文代码块展示。

## 9. 自由讨论与结构化审议的差异化呈现

差异仅发生在 UI 组织方式，不改变实际调度和模型行为。

### 9.1 自由讨论

- 默认使用连续对话流。
- 强调“Agent 名称 + 正文 + 时间”。
- 主持调度信息放入折叠的流程轨道，不穿插成与发言同权重的大段内容。
- 不显示固定“最终报告”占位。
- 若存在共同成果，则作为对话结束后的附加内容，而不是页面必须完成的步骤。

### 9.2 结构化审议

- 可以按规划、参与者意见、审核、成果等阶段分组。
- 阶段分隔使用紧凑标签，不重复渲染大型标题。
- 检查点保持突出，因为它需要用户操作。
- 报告存在时显示成果锚点；不存在时自然结束在最后一条有效输出。

### 9.3 共用能力

- 两种模式都使用同一套时间线锚点。
- 两种模式都使用同一套 Agent 正文卡片。
- 两种模式都可以展开运行信息和生命周期事件。
- 页面不根据模式推断“必须有主持总结”或“必须有报告”。

## 10. 状态、元数据与生命周期事件

### 10.1 顶部元数据

议题顶部只保留高频信息：

- 当前状态。
- 模式。
- 创建时间。
- 调用量。

议题 ID、完整 ISO 时间、限制参数等放入“详情”弹层或折叠区域。

`CALLS 20 / 100` 建议改为更直观的“调用 20 / 100”，并配合细进度条，不需要扩大字号。

### 10.2 生命周期事件

当前生命周期事件直接平铺在长页面底部，查找困难且视觉上接近正文。建议：

- 改名为“流程记录”。
- 默认折叠，仅显示事件数量、首尾时间和是否存在警告/失败。
- 展开后按时间顺序显示。
- 普通事件使用低对比度；警告和失败保留清晰标识。
- 事件同时可出现在右侧时间线的“显示流程”视图中。

## 11. 视觉状态示例

### 11.1 默认阅读状态

- 档案索引可见。
- Agent 正文全部正常展示。
- 时间线仅显示正文、检查点和成果。
- 程序化流程卡和生命周期事件折叠。
- 代码块限制最大高度。

### 11.2 审计状态

用户开启“显示流程”后：

- 时间线补充所有流程锚点。
- 每个 Agent 输出显示完整阶段、调用来源和逻辑调用 ID。
- 生命周期事件展开。
- 不改变正文顺序和原始内容。

这样可以兼顾日常阅读与运行审计，而无需为两类用户维护两套页面。

## 12. 实施优先级

### P0：直接改善当前阅读体验

1. 将正文、标题和表格调整为紧凑字号与间距。
2. 为 Markdown 表格增加专用样式和横向滚动容器。
3. 重构单次 Agent 输出卡片，正文突出，调用与阶段信息弱化。
4. 将生命周期事件改为默认折叠的“流程记录”。
5. 报告为空时不渲染成果占位区。

### P1：解决长内容定位问题

1. 增加右侧时间线锚点导航。
2. 使用当前锚点高亮和 URL hash 定位。
3. 增加“仅正文 / 显示流程”切换。
4. 支持上一条、下一条 Agent 正文跳转。
5. 档案索引支持折叠，为正文释放空间。

### P2：完善多设备与个性化

1. 增加紧凑/舒适密度切换。
2. 移动端增加时间线抽屉。
3. 长代码块展开与复制。
4. 表格首列粘性与长表格滚动提示。
5. Agent 或阶段过滤。

## 13. 与当前前端实现的对应关系

本方案主要涉及 `src/web/index.ts` 中的展示代码，不要求修改模型侧：

| 当前实现 | 建议调整 |
|---|---|
| `STYLES_CSS` | 字号、间距、三栏布局、表格、代码块、正文卡、流程卡和响应式样式 |
| `openArchive()` | 按实际数据决定是否渲染成果区；增加时间线容器和流程记录折叠区 |
| `transcriptHtml(records)` | 使用现有 `id / at / stage / agentId / invocation` 生成锚点与分层卡片 |
| `eventHtml(event)` | 输出紧凑流程条目，并支持时间线关联 |
| `data.reportHtml` | 仅在非空时渲染，不作为所有议题的必备区域 |
| 现有 transcript/event 数据 | 直接生成时间线，无需新增模型输出字段 |

如果 Markdown 表格目前由服务端直接生成 HTML，前端可在渲染后为 `table` 包装滚动容器，或在 Markdown 渲染阶段统一输出 `.table-scroll` 容器；两种方式都不影响模型输出。

## 14. 验收标准

### 14.1 阅读与排版

- 1366px 宽屏下，正文无需横向滚动。
- 2304px 超宽屏下，内容整体视觉居中，右侧空间被时间线合理使用。
- 普通 Agent 正文字号默认为约 15px，不再继承过大的报告字号。
- 表格默认字号约 13px，表头和数据行能够快速横向对照。
- 宽表格只在自身容器内横向滚动。
- 行内代码和代码块满足清晰可读的文字对比度。

### 14.2 时间线

- 每一条 Agent 输出都有可点击锚点。
- 点击时间线后准确定位到对应内容。
- 当前阅读条目能够自动高亮。
- 默认时间线不被调用开始、冻结、提交等事件淹没。
- 开启“显示流程”后可访问完整事件序列。
- 刷新带 hash 的页面后能够恢复到相同锚点。

### 14.3 内容分层

- Agent 正文始终是单次输出卡片中最醒目的内容。
- `stage`、CLI、preset、logicalCallId 和耗时默认弱化或折叠。
- 流程型输出和生命周期事件可审计，但不会与普通发言使用相同视觉权重。
- 代码块保持可辨识，不因“流程弱化”规则误伤普通正文中的技术内容。

### 14.4 模式兼容

- 没有报告的议题不会出现空成果区或错误暗示。
- 自由讨论能够自然呈现为连续对话。
- 结构化审议仍可查看阶段、检查点、审核与报告。
- 所有优化均不依赖修改模型提示或输出格式。
